# 第15讲｜Context Engineering — 从 Prompt 到上下文系统

**时长**：30 分钟
**目标**：让团队理解 2024 Q4 业界正式命名的范式升级——从"写好一条 prompt"到"设计每次模型调用前的上下文窗口"；掌握 7 块组成、3 项核心技术、4 层 memory、4 大反模式；建立 2025 做 AI 应用的"上下文工程"心智模型
**配套演示**：demo.html（Prompt vs Context 对比 + 三位推动者引语 + 4 个交互式 demo：7 块组成可视化、Budget 模拟器、History Compaction、Bad vs Good 上下文对照）

## 开场（2 分钟）

先讲一个真事，发生在上个月。

某客户 Demo 阶段，我们用 GPT-4o 跑"客户访谈需求提取"，效果惊艳——客户当场拍板。生产环境我们升级到 Claude Opus，因为客户要求"上线必须用最强的模型"。

结果上线第二周，客户来了一句让我们整个团队蒙圈的反馈：

> "你们换了模型之后，效果还不如 Demo 版本。"

我们的本能反应是：**不可能**。Opus 各家 benchmark 都比 4o 强。然后开始排查：

- prompt 改了吗？没改。
- 知识库变了吗？没变。
- 数据格式有差异吗？没有。

最后定位到的根因，是个让所有工程师都觉得"我居然没想到这个"的东西——

**我们的 prompt 组装顺序错了**。

具体来说：我们的代码是 `system_prompt + retrieved_docs + user_query`。但在 Opus 上，我们调用 API 的时候不小心把 retrieved_docs **压到了 system 消息的开头**——也就是说，模型把客户的访谈文档当成了"系统指令"来理解。

GPT-4o 对这种顺序不敏感，但 Opus 对 system 段非常敏感——它会把 system 里的内容当成"权威指令"。结果模型一直在"按照客户访谈的口吻去回答"，而不是"分析客户访谈"。

**这就是 Context Engineering 这个新词背后的核心命题**——上下文窗口里塞什么、不塞什么、按什么顺序、压成多少，这些**全都是工程问题**，不是 "prompt 写得好不好"能囊括的。

打开 demo.html，今天我们把这个事系统讲清楚。

## 第一部分：从 Prompt 到 Context 的范式升级（4 分钟）

【看第一节对比表 + 三位推动者的引语】

2023 年我们做 AI 应用，开口闭口"prompt engineering"。Tobias Lütke（Shopify CEO）2024 年 9 月在推特上发了一句话，被广泛认为是"Context Engineering"这个词正式登上业界舞台的节点：

> "I really like the term context engineering over prompt engineering. It describes the core skill better: the art of providing all the context for the task to be plausibly solvable by the LLM."

翻译过来——**"上下文工程"这个词比"提示工程"更准确地描述了核心技能：为任务提供恰当的全部上下文，让 LLM 有可能解决它。**

紧接着 Karpathy（前 Tesla AI 主管、OpenAI 创始团队）转发并补充：

> "+1 for context engineering. People associate prompts with short instructions. Context engineering is the delicate art and science of filling the context window with just the right information for the next step."

——**"+1 给上下文工程。人们一提 prompt 就想到短指令。上下文工程是精细的艺术与科学——为下一步操作，把上下文窗口填上恰好正确的信息。"**

然后 LangChain 的 Lance Martin 在博客《The rise of context engineering》里把它正式系统化了。Anthropic 在 Claude Code 的设计文档里、Cognition（Devin 母公司）在工程笔记里、Cursor 的 Composer 文档里——**几乎所有一线 AI 工程团队都在 2024-2025 年正式采用了这个术语**。

### 为什么需要这个新词

【看对比表】

我列了 6 个维度，逐条看 Prompt Engineering 和 Context Engineering 的差别——

**维度 1：关注对象**

Prompt Engineering 关注的是**一条 prompt** ——它的措辞、结构、范例。
Context Engineering 关注的是**整个上下文窗口** ——system prompt、retrieved knowledge、conversation history、few-shot examples、tool schemas、scratchpad、memory，**这 7 个块的总和**。

**维度 2：工作产物**

Prompt Engineering 的产物是一段**静态的 prompt 字符串**——存在 `prompt.txt` 里。
Context Engineering 的产物是一个**运行时上下文组装器** ——一个 `build_context(task, state) → str` 的函数。每次调用前根据任务和当前状态动态拼装。

**这就是核心区别——Context 是动态的，不是静态的**。

**维度 3：优化手段**

Prompt 工程的优化手段是：改措辞、加 few-shot、调 CoT 触发词。
Context 工程的优化手段是：**retrieval gating**（决定要不要检索）、**history compaction**（压缩历史）、**subagent isolation**（隔离子 agent）、**memory layering**（分层记忆）。

完全不同的工程范畴。

**维度 4：质量瓶颈**

Prompt 工程时代，你最常的反馈是"prompt 写得不够清楚"。
Context 工程时代，你最常的反馈是"上下文塞错了、塞多了、顺序错了、该删的没删"。

我开场那个例子——**Opus 把客户文档当系统指令**——就是典型的"上下文顺序错了"。Prompt 本身没问题，是组装上下文这件事错了。

**维度 5 & 6**：典型符号和代表实践就不重复念了，重点是这一行——

**2025 的代表实践都不是"一条好 prompt"，是"一个好的 context 系统"**：Claude Code 的 `/compact` 命令、Cursor Composer 的 file-level gating、Devin 的 episodic memory、Anthropic 的 multi-agent research——全是工程系统级的设计。

### 一句话总结

【看下面绿色块】

**2023 靠 prompt，2024 靠 RAG + 工具，2025 决定上限的是 Context Engineering**。

谁能把每次模型调用前的上下文窗口设计成一个工程系统——决定塞什么、不塞什么、什么时候压缩、什么时候开子 agent——谁就能把 AI 用稳。

## 第二部分：Context Window 的 7 大组成块（5 分钟）

【滚动到第二节 · 演示一：可视化】

要做 context engineering，第一步是知道**上下文窗口里到底有什么**。它不是一个袋子，是一个**有结构的 7 层堆栈**。

逐块看——

### Block 1 · 系统提示（System Prompt）

【点击第一个绿色块】

定义角色、任务、输出格式、规则。是上下文的"宪法"——其他所有块都受它约束。

典型大小：500 - 5K tokens。

**注意点**：修改 system prompt 等价于改一个未声明的全局变量——一行措辞调整可能让所有下游回答风格全变。所以**system prompt 必须进 git，必须有 review**，不能想改就改。

### Block 2 · 检索注入（Retrieved Knowledge）

【点击蓝色块】

RAG 召回的外部文档、知识库片段、行业先例。让模型能用"私有 / 实时"信息回答。

典型大小：0 - 100K tokens。变化范围最大。

**注意点**——这一点 2025 业界的认知有重大转变——**不是越多越好**。召回 8 段相关性 0.4 的文档，远不如召回 2 段相关性 0.8 的。**注意力被稀释比信息不足更糟**。

2023 我们崇尚"检索召回率"，2025 业界开始反思——**Lost in the Middle 论文表明，模型对上下文中段几乎视而不见**。你召回到了关键事实，但塞在第 5 段，模型可能看不见。

### Block 3 · 少样本范例（Few-shot Examples）

【点击橙色块】

给模型展示几对"输入-输出"范例。比文字说明更有效——模型从范例里学结构和边界。

典型大小：500 - 5K tokens。

**注意点**：**范例选择本身是个工程问题**。bad few-shot 会把模型带偏，比没有 few-shot 还糟。

我们项目里有过一次教训——为了演示"需求提取"的输出结构，我们给了 3 个范例，其中 2 个都是 P0 优先级。结果模型学会了"大部分需求都是 P0"，所有客户访谈的输出里 P0 占比 80%——这显然不对。**few-shot 的分布会影响模型输出的分布**。

### Block 4 · 工具定义（Tool Schemas）

【点击紫色块】

function calling 的工具列表 + 每个工具的参数 schema + 描述。让模型知道"我能做什么"。

典型大小：500 - 30K tokens。

**注意点**——**Tool Soup 反模式**。给 agent 挂 30 个工具，每个 schema 800 token，**光工具定义就吃掉 24K 上下文**。模型还没开始干活，注意力就被工具菜单耗尽了。

2025 业界普遍做法是 **tool retrieval**——先根据任务从工具池里挑 3-5 个真正相关的工具，再塞给模型。Cursor、Cline 都在这么做。

### Block 5 · 工作记事本（Scratchpad / CoT）

【点击青蓝块】

让模型在 `<thinking>` 标签里推理、列子目标、记笔记。提高复杂任务的成功率。

典型大小：500 - 10K tokens。

**注意点**：scratchpad 的内容**大部分应该在最终回答前被舍弃**——它是"工作记忆"，不是"知识"。如果你把上一轮的 scratchpad 完整塞进下一轮的 history，会有严重的**Context Rot**——模型会被自己上次的思考过程缠住。

### Block 6 · 对话历史（Conversation History）

【点击粉色块】

前面 N 轮的 user / assistant 消息。让模型记得我们刚才在聊什么。

典型大小：1K - 100K tokens。

**注意点**——**Lost in the Middle 和 Context Rot 主要发生在这一块**。长会话必须接入 compaction。Claude Code 的 `/compact` 命令就是干这个的——把 30 轮历史压缩成 4K 摘要，保留决策和约束，丢掉寒暄和中间状态。

### Block 7 · 持久记忆（Memory）

【点击灰色块】

跨 session 沉淀的偏好、知识、技能。从对话外部注入——通常从向量库或规则文件读取。

典型大小：200 - 5K tokens。

**注意点**：语义记忆容易过时——用户改了偏好但旧 memory 还在，会让模型回到旧规则。需要 **expire 机制**。

### 演示：拖动复杂度滑条

【拖动顶部的"任务复杂度"滑条，从 0 到 3】

看占比怎么变——

- **简单单轮 QA**：system prompt 主导（30%），其他块几乎为 0
- **中等任务**：retrieved 主导（30%），history 上升到 25%
- **复杂 RAG 问答**：retrieved 暴涨到 55%，因为要塞大量文档
- **长 Agent 任务**：history 反超到 38%，tools 也涨到 15%

**这就是 Context Engineering 的第一个心智模型——不同任务的最优分配完全不同，没有"通用模板"**。

## 第三部分：Context Budget 管控（5 分钟）

【滚动到第三节 · 演示二：Budget 模拟器】

很多人对 Context Engineering 有个误解：**Claude / Gemini 都有 200K 窗口了，我塞满不就行了？**

错。200K 不是免费午餐——你每多 1k token 都要付三笔账：

**第一笔：cost（成本）**

Claude Opus 4 输入价是 $15 per 1M tokens。你塞 100K 上下文，**单次调用 input 成本 $1.5**。一天调用 10 万次就是 **$15 万 cost**。这不是 demo 数字，是我们某个客户真实账单。

**第二笔：latency（延迟）**

输入 token 越多，TTFT（time to first token）越长。200K 输入的 TTFT 普遍在 2-3 秒。**用户对话场景这就是不可接受**。

**第三笔：attention（注意力衰减）**

最致命的一笔。Stanford 2023 论文 *Lost in the Middle* 已经证明：**模型对上下文开头和末尾的注意力强，中段几乎被忽略**。你塞 200K 进去，中间 100K 等于扔进黑洞。

### 用模拟器玩一遍

【看演示二】

我做了个简单的 Budget 模拟器。你可以：

1. 选总预算——20K、50K、200K
2. 拖动 7 个块的滑条，分配每块占多少 token
3. 实时看 4 个指标：总占用、质量分、估算成本、估算延迟

我演示几个典型情况——

**情况 1：点击 "载入反面教材"**

【点击 "载入反面教材"】

看这个分配：retrieved 35K、history 30K、tools 12K，**总和 77K**。但是：
- system prompt 只有 200——**模型缺少身份和规则锚点**
- few-shot 是 0——**抽取任务质量难保证**
- scratchpad 只有 200——**推理链路容易断**
- 历史占 60%——**Context Rot 高危**
- tools 12K——**Tool Soup**

质量分：30 多分。这是个**典型的"塞太多但都塞错地方"的反例**。

**情况 2：点击 "载入参考分配"**

【点击 "载入参考分配"】

看这个分配：system 1.5K、retrieved 12K、few-shot 1.8K、tools 2.5K、scratchpad 2K、history 6K、memory 1.2K，**总和 27K**。

- 比反面教材少了一半
- 但每个块都在合理区间
- 没有任何反模式触发

质量分：90+。**这才是 production 级的预算结构**。

**核心洞察**：

- **总量不是越大越好** —— 反面教材 77K，参考方案 27K，质量分差 60+
- **每个块都有"健康区间"** —— 不在区间内会触发反模式
- **预算分配本身是工程决策** —— 不是 "我开个大上下文就完事了"

### 我们项目的实际预算

讲个我们项目的真实数字——

KB-System 的"项目洞察"功能，文档全文喂进 Opus，决策上限是 200K 窗口。我们的实际分配（你们可以去 `backend/services/agentic/executor.py` 里看 `_build_sources_index`）：

- system prompt：约 2K
- retrieved（核心 D 类文档，每份最多 30K，最多 4-7 份）：50-80K
- few-shot：0（因为是开放生成任务）
- tools：约 1K
- scratchpad：约 5K（让模型先思考）
- history：通常 0（每次都是独立调用）
- memory：项目元数据约 2K

总计 60-90K，预留 100K 给输出。**这是我们和用户在 2026 年 4 月明确确认过的决策——文档喂全文，不走切片召回**。

这就是 Context Budget 的实战——**每一块都是有意识的工程决策，不是默认值**。

## 第四部分：History Compaction · 长对话的救命稻草（4 分钟）

【滚动到第四节 · 演示三：Compaction】

讲完 Budget，讲三项核心技术里最常用的——**History Compaction（历史压缩）**。

长 agent 会话最大的敌人是**对话历史的指数级膨胀**。一个 Cursor 编辑器开 4 小时、Claude Code 跑 30 轮、Devin 跑一个长任务——历史能涨到 50K+。

但其中 **80% 是垃圾**——

- 寒暄（"好的"、"明白了"、"继续"）
- 重复确认（"是这样吗？""对"）
- 已完成的中间步骤（"我先 cd 到目录"、"我读了文件 A"）
- 模型的内部思考链（CoT 残留）

### 演示：30 轮对话压成 4K 摘要

【看演示三】

这是我们项目"CRM 需求提取"场景的真实 30 轮对话。

【点击 "Compact Now" 按钮】

左边：30 轮原始对话，约 **26.4K tokens**。看到很多对话被划掉了（红线删除），这些是"丢弃"的内容——寒暄、重复确认、已完成步骤。

右边：**/compact 输出**，约 **3.9K tokens**。压缩率 85%。

看 /compact 输出的结构——

```
## DECISIONS · 已确认的决策
- 抽取字段: { 描述, 类别, 优先级, 是否影响留存, 证据 }
- 类别枚举: 性能/集成/业务功能/权限/体验
- 优先级: P0 / P1 / P2
- 第三方集成 (如钉钉对接) 默认 P1
- 分销商权限类场景 = P0

## CONSTRAINTS · 全局约束
- 输出纯 JSON，无 emoji
- 每条结论必须含证据字段

## TODO · 未完成
- 第 6 份访谈待开始
- 后续按相同格式继续

## DROPPED · 已舍弃
- 所有中间确认对话、寒暄、临时疑问
- 已完成且无后续依赖的 JSON 输出
```

**这就是 compaction 的核心算法**——

**保留**：决策、约束、未完成项、关键变量。
**丢弃**：寒暄、重复确认、已完成且无后续依赖的中间步骤、内部 CoT。

### 各家的 compaction 实现

- **Claude Code 的 `/compact`**：用户手动触发，或在窗口快满时自动触发。底层是 LLM-driven summarizer——用 LLM 自己来做语义压缩。
- **Cursor 的 chat history**：自动按 token 阈值截断，并保留"重要消息"标记。
- **ChatGPT memory**：跨 session 的 semantic memory，自动从对话中提炼用户偏好。
- **Devin 的 plan tree**：把对话压成"计划树"，每个 step 都是压缩后的状态——这是更激进的做法。

### 实战建议

对我们项目的实战建议——

1. **任何长会话功能都必须设计 compaction**——不是上线之后再补，是设计时就考虑
2. **不要简单截断**（保留最近 N 轮）——会丢失早期的关键决策
3. **/compact 用 LLM 做** ——不要用规则。规则压缩会丢语义。
4. **每 20 轮强制触发一次**——不要等到窗口爆了再压

## 第五部分：三项核心技术 · 完整版（3 分钟）

【滚动到第五节】

刚才讲了 History Compaction，现在快速过另外两项核心技术——

### 技术 1：Retrieval Gating · 不再无脑前置检索

2023 年的 RAG 范式是"每次 query 都先检索"。2025 的认知升级是——**检索本身也是上下文污染**。

召回 8 段相关性 0.4 的文档，远比"不检索"更糟糕——
- 模型注意力被稀释
- 容易被无关材料带偏
- 增加 latency 和 cost

新做法 —— **Retrieval Gating**：

1. 先让模型自问"这个问题需要外部知识吗？"
2. 如果需要，再问"需要哪一类知识？"
3. 只在必要时检索，并且检索粒度、数量、是否拼回上下文都**动态决定**

**典型实现**：
- Claude Code 的 **Agentic Search**——agent 决定是否搜索
- Cursor 的 file-level retrieval——只在需要时读 file
- 我们项目里 D 类文档喂全文，C 类文档走 RAG——**也是一种 gating**

### 技术 2：Subagent Isolation · 父 Agent 的上下文保护墙

Anthropic 的 multi-agent research 系统给出了一个反直觉的结论——

**让 subagent 跑独立 context window，比让父 agent 直接读全部资料效果更好**。

原因：subagent 处理掉 50K 原始材料后只返回 2K 摘要给父 agent。**父 agent 的窗口被"保护"了**——它永远看到的是"摘要后的世界"。

**典型实现**：
- Claude Code 的 **Task 工具**——默认每个子任务跑独立 agent
- Devin 的 planner-executor 分窗口
- Cline 的 task 树结构

**对我们项目的启示**：未来"项目洞察"功能可以拆 subagent——每份 D 类文档由一个 subagent 处理，返回摘要给主 agent。**主 agent 永远不直接读 80K 全文，只读摘要**。这是个明确的优化方向。

## 第六部分：Memory 分层架构（2 分钟）

【滚动到第五节下半部分 · Memory 四层】

讲完三项技术，讲 memory 分层——这是把"context"升级为"持久状态"的关键。

**4 层架构**——

### Layer 1 · 工作记忆（Working Memory）

当前几轮对话直接放在 context window 里。**最快、最贵**，容量上限 = 模型窗口。

- scope: 当前 session
- size: ~10-50K
- ttl: until /compact

Claude Code 的 conversation buffer、Cursor 的当前 chat 都在这一层。

### Layer 2 · 情景记忆（Episodic Memory）

过去 session 的**任务 trace 摘要**——做过什么、用了什么工具、得出什么结论。

- scope: 跨 session, 同用户/同项目
- size: ~每条 200-2k tokens
- storage: vector DB + 元数据

Devin 的 episodic memory、Cursor 的 chat history 是典型。

### Layer 3 · 语义记忆（Semantic Memory）

沉淀下来的**知识、概念、用户偏好**。

- scope: 长期, 跨项目
- size: ~50-500 条事实
- example: "用户偏好 P0/P1/P2 三级优先级"

ChatGPT memory、Claude user preferences、我们项目的"用户画像"都在这一层。

### Layer 4 · 程序记忆（Procedural Memory）

"**怎么做**"的固化技能。

- scope: 全局, 跨用户
- size: 每条 100-3k tokens
- example: CLAUDE.md / .cursorrules

**这是 2025 最重要的一层**——Claude Code 的 Skills、Cursor 的 rules、AGENTS.md。本质是把"经过验证的好上下文"打包复用。

**对团队的实战提示**：我们项目根目录的 `CLAUDE.md` 就是 Procedural Memory。**它就是 2025 的代码**——决定了每个 AI 调用的基础行为。它必须像代码一样严肃维护。

## 第七部分：Bad vs Good Context · 同问题两个上下文（3 分钟）

【滚动到第六节 · 演示四：Bad vs Good】

现在做个总实验。同一道题——**让模型分析一段客户访谈记录**。两种上下文工程，看输出差距。

### 左 tab · BAD CONTEXT

【点击 "BAD CONTEXT" tab】

**上下文构成**：只有一块——用户输入。访谈记录 + "请分析"。完。

**模型输出**：

> 根据您提供的访谈记录，客户提到了几个问题：1. CRM 不好用；2. 商机列表加载慢；3. 移动端崩溃...建议您优化系统性能、简化审批流程，并考虑对接外部系统。

**指标**：质量 3/10，输入 1.2K token，延迟 1.8 秒，可用性：**NO**。

**为什么差**？模型不知道：
- 你是销售、产品、还是顾问？
- 你要的输出是 JSON 还是 markdown？
- 优先级怎么定？
- 边界 case 怎么处理？

它只能给一份**罗列式、无优先级、无可执行性的复述**。

### 右 tab · GOOD CONTEXT

【点击 "GOOD CONTEXT" tab】

**上下文构成** —— 5 块：

1. **System prompt（800 token）**：定义角色（SaaS CRM 实施顾问）、任务（提取需求池）、输出格式（JSON 数组）、字段定义、优先级规则
2. **Few-shot（1.4K token）**：3 个 in-domain 范例，演示边界判断
3. **Retrieved（2.2K token）**：从我们项目库召回的"行业先例"——类似客户的流失案例、高频诉求统计
4. **Scratchpad 引导**：让模型先在 `<thinking>` 里扫描"痛点信号词"再产出
5. **User input（1.8K token）**：访谈原文，**放在最后**

**模型输出**：

```json
[
  { "描述": "商机列表加载 8 秒", "类别": "性能", "优先级": "P0", "影响留存": true,
    "证据": "性能问题历史上导致过类似客户流失 ¥380w ARR" },
  { "描述": "跟进提醒缺失", "类别": "业务功能", "优先级": "P0", "影响留存": true,
    "证据": "客户 Lily 明确说 客户 A 流失 因为漏跟进" },
  ...
]
```

**指标**：质量 9/10，输入 6.2K token，延迟 3.4 秒，可用性：**YES**。

### 对比

【看下面的总结】

- **Token 涨了 5 倍**（1.2K → 6.2K）
- **延迟涨了 1.9 倍**（1.8s → 3.4s）
- **质量从 3/10 涨到 9/10**——**不能用 → 直接进需求池**

**这不是 +20% 的差距，是 0→1 的差距**。

而所有这些"好"的来源**都不是改 prompt 措辞**，是改"上下文里塞什么"——加 system prompt、加 few-shot、加 retrieved、加 scratchpad。

**这就是 Context Engineering 的实战价值**——它把"AI 能不能用"从一个玄学问题变成一个工程问题。

## 第八部分：四大反模式 + 2025 工具对比（2 分钟）

【滚动到第七、八节】

最后讲两个内容——**反模式**和**工具对比**。

### 四大反模式

**Anti-pattern 1 · Lost in the Middle（中段遗忘）**

模型对上下文开头和结尾注意力强，中段几乎被忽略。
**修**：关键事实放系统提示末尾或用户输入最后一段；retrieved 段数 ≤5。

**Anti-pattern 2 · Context Rot（上下文腐烂）**

长 agent 跑久了，历史里堆满了已经被推翻的决策、过期状态。
**修**：每 N 轮 `/compact`；显式标注"已推翻 / 已完成"。

**Anti-pattern 3 · Tool Soup（工具汤）**

挂 30 个工具，schema 吃掉 24K，注意力被耗尽。
**修**：分组挂工具；schema 压缩到 ≤150 tokens；用 tool retrieval。

**Anti-pattern 4 · Prompt Drift（提示漂移）**

system prompt 被 N 个人改了 N 次，里面有互相矛盾的指令。
**修**：prompt 进 git；建立 prompt regression 测试集；定期对账。

### 2025 工具对比

【看工具对比表】

5 个一线产品的 Context 策略——

- **Claude Code**：CLAUDE.md + Agentic Search + Sub-Task isolation + `/compact`
- **Cursor Composer**：.cursorrules + indexed codebase + file-level gating
- **Devin**：长 agent loop + episodic memory + planner-executor 分窗口
- **Aider**：repo-map（压缩版项目结构）+ /add /drop 显式管理
- **Cline**：task 树 + .clinerules + token usage 可视化

**共性观察**：**所有头部产品都不再"塞满 200K"，全都在做"窗口节流"**。它们的差异只是节流方式不同——但"上下文是稀缺资源、必须做工程化管理"这件事，**2025 是行业共识**。

## 收尾（1 分钟）

【滚动到总结】

把今天的核心收束成一句话：

> **Prompt Engineering 是关心如何写好一句话；Context Engineering 是关心如何设计模型每次调用前的整个上下文系统。**

**今天讲清楚的 5 件事**：

1. **7 块组成**——system / retrieved / few-shot / tools / scratchpad / history / memory
2. **Budget 管控**——200K 不是免费午餐，每多 1k 都是 cost + latency + attention 三重账
3. **三项核心技术**——retrieval gating、history compaction、subagent isolation
4. **Memory 四层**——working / episodic / semantic / procedural
5. **四大反模式**——Lost in the Middle、Context Rot、Tool Soup、Prompt Drift

**对我们项目最实用的 4 条建议**：

1. 把"组装 prompt 的代码"**独立成 `build_context()` 函数**，纳入 review——不要散落在各处字符串拼接里
2. 长会话场景**强制接入 /compact 或自建 summarizer**，每 20 轮压一次
3. Agent 任务**能拆 subagent 就拆**，父 agent 永远只读摘要不读全量——这对项目洞察功能尤其重要
4. 把 CLAUDE.md / .cursorrules 当成**"程序记忆"严肃维护**——它就是 2025 的代码

**Prompt Engineering 关注单点输入，Context Engineering 关注系统设计**。谁掌握后者，谁就能把 AI 用稳。

下一讲：**Harness Engineering — 让 Agent 在长循环里持续工作的外壳**。Context Engineering 解决"每次调用前塞什么"，Harness Engineering 解决"调用之间的循环怎么持续运行不崩溃、不失忆、不发散"——这是 2025 agent 工程的另一半。

## 备注：常见疑问

**Q：Context Engineering 和 Prompt Engineering 是替代关系吗？**
A：不是。Prompt 仍然是 Context 的核心一块（system prompt + user prompt）。但 Context Engineering **包含但不限于** Prompt Engineering——它把视野扩大到整个上下文窗口，并引入了 retrieval、history、memory、tools 等新维度的工程问题。简单说：**Prompt 是 Context 的子集**。

**Q：模型窗口越来越大（200K → 1M → 10M），Context Engineering 是不是就不重要了？**
A：恰好相反——**窗口越大，Context Engineering 越重要**。原因：① cost 和 latency 是线性增长的，10M 窗口的 cost 不可承受；② attention 衰减问题在长窗口上反而更严重（Lost in the Middle 在 1M 窗口下中段几乎完全失效）；③ 大窗口让人有"塞满"的诱惑，反而需要更强的工程约束。

**Q：Retrieval Gating 怎么决定要不要检索？**
A：典型实现是用一个轻量级 classifier（甚至是 LLM 本身的一个小提示）先判断"这个 query 是否需要外部知识"。我们项目里的简单做法是：根据 query 类型（事实问询、操作请求、创作任务）分类，事实问询走检索，创作任务跳过。

**Q：subagent 隔离会不会引入新的协调问题？**
A：会。这是 Harness Engineering 要解决的问题——父 agent 怎么和 subagent 通信、怎么处理 subagent 的失败、怎么合并 subagent 的结果。**所以下一讲专门讲 Harness Engineering**。

**Q：CLAUDE.md 应该写多长？**
A：Anthropic 的建议是 **2-5K tokens**。超过 5K 容易导致 system prompt 占比过大，反而损害质量。我们项目的 CLAUDE.md 现在约 1.5K，覆盖了核心约定。复杂规则应该拆到 Skills 文件，按需加载，不要全塞进 CLAUDE.md。

**Q：业务上要怎么向产品 / 老板解释 Context Engineering 的价值？**
A：**用 Bad vs Good 对比演示**。同一道题，同样的模型，BAD context 输出 3/10 不可用，GOOD context 输出 9/10 直接进生产。Token 涨 5 倍但单次调用成本仍可控（几分钱），**业务价值是 0→1 的差距**。这是最直接的"为什么要做 Context Engineering"的论证。
