LESSON 15 / 23

CONTEXT
ENGINEERING

从 Prompt 到上下文系统
上一讲我们说"没有评估和观测，生产环境就是黑盒"。但即使观测全装上、评估全跑过， 真正决定模型每次输出质量的，是上下文窗口里塞了什么、不塞什么、怎么排序、怎么压缩。 2024 Q4 业界给这件事正式命名——Context Engineering。 今天讲清楚 Prompt Engineering 到 Context Engineering 的范式升级。
CHAPTER · 01
FROM PROMPT TO CONTEXT
范式升级：从"写好一条话"到"设计上下文系统"
Prompt Engineering 关注的是一条 prompt 怎么写。 Context Engineering 关注的是每一次模型调用前，整个上下文窗口里塞什么、不塞什么、按什么顺序、压成多少——它把"写 prompt"从一门手艺变成了一门系统工程。

CHAPTER · 03
CONTEXT BUDGET
演示二：Budget 模拟器 · 200K 不是免费午餐
200K token 不等于"想塞多少塞多少"。每多 1k token 你都要付出三笔账： cost（更贵）、latency（更慢）、attention（注意力衰减、Lost in the Middle）。 自己拖滑条试试，看你的预算分配能拿到多少质量分。
CHAPTER · 04
HISTORY COMPACTION
演示三：长对话压缩 · 30 轮 → 4k 摘要
长 agent 会话最大的敌人是对话历史的指数级膨胀。Claude Code 的 /compact、 Cursor 的 rules、ChatGPT 的 memory 都在解决同一件事—— 把不重要的丢掉，把决策、约束、未完成项保留并升级为摘要。下面是我们项目"需求提取"场景的真实 30 轮对话。
/compact · KB-SYSTEM · CRM 需求提取
压缩前 · 30 轮原始对话
压缩后 · /compact 输出

什么必须保留：决策（"用 P0/P1/P2 三级"）、约束（"不要 emoji"）、未完成项（"还差 2 个边界 case"）、关键变量（user_id、project_id）。 什么可以丢：寒暄、重复确认、已经完成且无后续依赖的中间步骤、模型的内部 chain-of-thought。 Claude Code 的 /compact 会把这些规则编码成一个 LLM-driven summarizer，定期对历史窗口做"语义重排"。

CHAPTER · 05
THREE CORE TECHNIQUES
三项核心技术：Retrieval Gating · Subagent 隔离 · Memory 分层
Context Engineering 不是单一技巧，是三个核心技术构成的系统设计模式—— 什么时候检索、什么时候独立开 subagent、什么时候沉淀进 memory， 这三个决定决定了上下文窗口的健康度。

① Retrieval Gating · 不再是无脑前置检索

早期 RAG 的设计是每次 query 都先检索。Context Engineering 时代我们承认：检索本身也是上下文污染—— 召回 8 段不太相关的文档，远比"不检索"更糟糕（注意力被稀释、模型被无关材料带偏）。

2025 的做法是 retrieval gating——先让模型自问"这个问题需要外部知识吗？需要哪一类？"， 只在需要时才检索，并且检索的粒度、数量、是否拼回上下文都是动态决定的。 Cursor 的 Composer、Claude Code 的 Agentic Search 都是这个范式。

② Subagent 隔离 · 父 agent 的"上下文保护墙"

Anthropic 的 multi-agent research 系统给出了一个反直觉的结论：让 subagent 跑独立 context window，比让父 agent 直接读全部资料效果更好。 原因是 subagent 处理掉 50k 原始材料后只返回 2k 摘要给父 agent，父 agent 的窗口被"保护"了。

这就是为什么 Claude Code 默认每个 Task / Agent 子任务都跑独立 context——父 agent 永远看到的是"摘要后的世界"， 不是"全量原始材料"。Devin、Cline 也是同款架构。

③ Memory 分层 · 把上下文升级为持久状态

LAYER · 01
工作记忆
WORKING MEMORY

当前几轮对话直接放在 context window 里。最快、最贵，容量上限 = 模型窗口。Claude Code 的 conversation buffer 就在这一层。

scope: 当前 session
size: ~10-50K tokens
ttl: until /compact
LAYER · 02
情景记忆
EPISODIC MEMORY

过去 session 的任务 trace 摘要——做过什么、用了什么工具、得出什么结论。Devin 的 episodic memory、Cursor 的 chat history 都属此类。

scope: 跨 session, 同用户/同项目
size: ~每条 200-2k tokens
storage: vector DB + 元数据
LAYER · 03
语义记忆
SEMANTIC MEMORY

沉淀下来的知识、概念、用户偏好。ChatGPT 的 memory、Claude 的 user preferences、我们项目的"用户画像"都在这一层。

scope: 长期, 跨项目
size: ~50-500 条事实
example: "用户偏好 P0/P1/P2 三级优先级"
LAYER · 04
程序记忆
PROCEDURAL MEMORY

"怎么做"的固化技能——Claude Code 的 Skills、Cursor rules、AGENTS.md。本质是把"经过验证的好上下文"打包复用。

scope: 全局, 跨用户
size: 每条 100-3k tokens
example: CLAUDE.md / .cursorrules

CHAPTER · 07
FOUR ANTI-PATTERNS
四大反模式：你大概率正在踩坑
Context Engineering 80% 的工作是识别并拆除已经在系统里的反模式。 下面 4 个是 2024-2025 业界最常见的"上下文设计错误"，每条都来自真实生产事故。
ANTI · 01
Lost in the Middle
中段遗忘

Stanford 2023 论文：模型对上下文开头和结尾的注意力强，中间段几乎被忽略。当你把关键事实塞在第 3-8 段 retrieved 的中间，模型经常看不见。

典型症状：检索召回是对的、模型输出却没用上。

修：把最关键事实放系统提示末尾或用户输入最后一段；retrieved 段数控制在 ≤5；重要信息 dedup 后放头部。
ANTI · 02
Context Rot
上下文腐烂

长 agent 跑久了，历史里堆满了已经被推翻的决策、过期的中间状态、用户改过 3 次的需求。模型每次回答都被这些"垃圾上下文"误导。

典型症状：agent 在第 20 轮坚持要做用户 5 轮前已经否定的方案。

修：每 N 轮强制 /compact；显式标注"已推翻 / 已完成"；scratchpad 写入持久 memory 后从 working memory 清掉。
ANTI · 03
Tool Soup
工具汤

给 agent 挂 30 个工具，每个工具的 schema 都 800 tokens——光工具定义就吃掉 24K 上下文，还没开始干活就把模型注意力耗尽了。

典型症状：agent 在简单任务上反复纠结选哪个工具，或选错工具。

修：分组挂载工具（按场景动态拼）；schema 压缩到 ≤150 tokens；用 tool retrieval——根据任务先选 5 个工具再传给模型。
ANTI · 04
Prompt Drift
提示漂移

系统提示被 N 个人改了 N 次，里面有互相矛盾的指令："简洁" + "尽可能详细"、"用中文" + "always answer in English"。模型在矛盾里乱跳。

典型症状：相同问题不同时段给出风格完全不同的回答。

修：system prompt 进 git、每次改加 review；建立prompt regression 测试集；定期"对账"——把当前 prompt 和原始设计意图对比。

RECAP

核心收束 · 上下文是 LLM 应用的真正"代码"

Prompt Engineering 时代我们把模型当成"会答题的考生"——你给一个题目，它给一个答案。 Context Engineering 时代我们把模型当成"装在上下文里的 CPU"——它的能力上限取决于你给它的运行时环境。

"Prompt Engineering 是关心如何写好一句话；
Context Engineering 是关心如何设计模型每次调用前的整个上下文系统。"

今天讲清楚的 5 件事： ① 上下文窗口的 7 块组成——system / retrieved / few-shot / tools / scratchpad / history / memory； ② Budget 管控——200K 不是免费午餐，每多 1k 都是 cost + latency + attention 三重负担； ③ 三项核心技术——retrieval gating、history compaction、subagent isolation； ④ Memory 四层——working / episodic / semantic / procedural； ⑤ 四大反模式——Lost in the Middle、Context Rot、Tool Soup、Prompt Drift。

对我们项目最实用的 4 条建议： ① 把"组装 prompt 的代码"独立成 build_context() 函数，纳入 review； ② 长会话场景强制接入 /compact 或自建 summarizer，每 20 轮压一次； ③ Agent 任务能拆 subagent 就拆，父 agent 永远只读摘要不读全量； ④ 把 CLAUDE.md / .cursorrules / AGENTS.md 当成"程序记忆"严肃维护——它就是 2025 的代码。

第 14 讲我们装好了眼睛（评估和观测）。今天这一讲我们建立了大脑的"工作台"（context engineering）。 下一讲我们要谈 agent 的"骨骼"——Harness Engineering， 把长循环里的 agent 包进一层外壳，让它能持续运行而不崩溃、不失忆、不发散。
