LESSON 16 / 23

HARNESS
ENGINEERING

Agent 长程任务的外壳设计
上一讲讲 Context Engineering ——怎么把"对的信息"塞进模型脑子里。 今天讲它的兄弟问题——Agent 是策略，Harness 是骨架。 没有 harness，Agent 是 demo 玩具；有了 harness，Agent 才能跑 8 小时不崩。 这是 2025 年所有严肃 Agent 系统都在解决的工程问题—— 把模型放进一个能稳定运行的"外壳"。
CHAPTER · 01
AGENT vs HARNESS
策略与骨架：清晰分界
Anthropic 的 Boris Cherny 在 Claude Code 的工程笔记里反复强调一个区分： Agent 决定"做什么"，Harness 决定"怎么持续地做下去"。 做过 demo 没做过生产环境的同学，往往把两者搅在一起，结果在生产里全靠运气。
维度
Agent（策略）
Harness（骨架）
职责定义
Decide what to do——每一步决定调哪个工具、生成什么内容、推理路径走哪边
Make sure it keeps running——保证循环不崩、中断能恢复、context 不爆、错误可降级
实现载体
Prompt + 模型 + 工具集。靠 Claude / GPT-4o 的推理能力
主循环代码 + 状态机 + 持久化。靠工程师写 Python / TypeScript
变化频率
每次模型升级、Prompt 调优都在变——"软"层
一次写好、半年不动也能跑——"硬"层
谁来负责
产品 + AI 工程师——设计 prompt、调 tool、验质量
平台 / 基建工程师——保证可靠性、可观测、可恢复
失败模式
幻觉、走错路、tool 选错——"质量"问题
死循环、超时、context 炸、中断丢失——"可用性"问题
类比
司机的"判断"——往左还是往右，加速还是刹车
汽车的"底盘 + 安全带 + 油表 + 故障灯"——保证司机能开

2025 年的核心认知转变：以前大家盯着"换更好的模型"， 以为模型够强 agent 就能稳。但 Claude Sonnet 4.5 / Opus 4.7 已经强到很难再换的程度了—— 瓶颈不在策略，而在骨架。 Anthropic Boris Cherny / Cognition Walden Yan 的一系列工程笔记说的就是这件事—— "模型能力决定 Agent 上限的 80%，Harness 决定能不能稳定到这个上限"。

CHAPTER · 06
FAILURE MODES
Harness 4 大常见失败模式
团队自己造 harness 时最容易踩的 4 个坑。这些坑都是 demo 时看不出来、 生产跑 1 小时就崩的那种——"看着能跑，关键时刻全废"。
FAILURE · 01
Tool Soup
注册 50 个 tool，模型迷失

想着"功能丰富一点"，把 50+ 个工具一股脑全注册。 模型在每一步都要从这 50 个里挑一个，"选择困难症"被放大 10 倍。 典型表现：模型反复在 read_file / list_dir / grep 之间打转，3 步能干完的事干了 12 步。

修复路径：按"任务场景"做 tool gating（见 DEMO 2）。 一次最多暴露 8-12 个 tool。tool 描述写清楚"什么时候用 / 什么时候不用"。
FAILURE · 02
Context Bomb
让 agent 自己读完整代码库

给 agent 一个 "search_code(query)"工具，结果模型为了"调研一下"， 一上来就 read_file 二十几个大文件。200k context 五分钟内就被填满， 后面什么活都干不了，prompt-cache 全失效，账单飙升。

修复路径：read_file 默认只返回前 N 行（如 500 行）+ ToC； 按需 expand。在 system prompt 里明确"不要批量读文件，先用 grep / glob 定位"。
FAILURE · 03
死循环 / 错误放大
Infinite Loop & Error Amplification

tool 返回了一个错误，agent 不会反思，原样重试 30 次。 或者更糟——agent 看到错误后觉得"我应该再补一个文件"，于是越改越糟， 一晚上烧了 8000 块 token 钱。这是 LangChain 早期被骂得最惨的问题。

修复路径：① 同一个 tool 同一个 input 重复 ≥3 次强制中断； ② 设 hard max_iterations（如 200 轮）；③ 成本超阈值告警 +暂停。
FAILURE · 04
中断无 Resume 能力
No Resume = 1 hour wasted

agent 跑了 1 小时，做了 30 步，调用了 18 个 tool，写了 6 个文件。 用户 Ctrl+C 一下——所有状态全没了， 下次只能从头再来。这是 demo 阶段看不出来、生产阶段灾难性的失败模式。

修复路径：① 每完成一个 tool call 写一次 checkpoint； ② 中断时把当前 context + tool history + 工作目录 diff 全部序列化； ③ resume 时回放 + 询问用户"从哪里继续"。
CHAPTER · 07
MINIMUM VIABLE HARNESS
从 30 行 Python 开始造自己的 harness
Harness 听起来工程量大,其实核心骨架 30 行就能跑。 下面是一个 MVP 主循环——保留 6 大组件最核心步骤,删去 import / setup / error handling 装饰代码。 看清楚之后,你就能在自己的项目里搭起来。
# minimal_harness.py · MVP harness 核心骨架 def run(task, max_iter=100, gate=None): state = load_ckpt() or {"messages":[{"role":"user","content":task}], "step":0} interrupted = [False] signal.signal(signal.SIGINT, lambda *a: interrupted.__setitem__(0, True)) while state["step"] < max_iter: # ① Interruption: Ctrl+C 时存盘退出 if interrupted[0]: save_ckpt(state); return # ② Tool Gating: 按场景过滤 subset exposed = [s for n,(s,_) in TOOLS.items() if not gate or gate(n,state)] # ③ Auto-Compact: ctx 接近爆时 summarize if estimate_tokens(state["messages"]) > 150_000: state["messages"] = compact(state["messages"]) # ④ Error Recovery: retry with backoff resp = call_with_retry(exposed, state["messages"]) state["messages"].append({"role":"assistant","content":resp.content}) if resp.stop_reason == "end_turn": clear_ckpt(); return resp.content # ⑤ Tool 调用 + ⑥ Checkpoint(每步都存) results = [run_tool(b) for b in resp.content if b.type == "tool_use"] state["messages"].append({"role":"user","content":results}) state["step"] += 1 save_ckpt(state)

这 30 行已经覆盖了 6 大组件的核心骨架—— Loop Control(while 主循环 + max_iter)、Tool Gating(gate 过滤)、 Interruption(SIGINT flag)、Checkpointing(pickle 每步存)、 Error Recovery(retry backoff)、Auto-Compact(阈值触发)。 这就是 Claude Code 早期原型的样子—— 然后才一点点加 subagent、hooks、permissions、IDE 集成等周边能力。

CHAPTER · 08
DESIGN TRADE-OFFS
4 个核心设计权衡
Harness 设计没有"标准答案"，只有"权衡"。 不同的 harness 在以下 4 个维度上做了截然不同的选择。
TRADE-OFF · 01
自主 vs 可控
AUTONOMY vs STEERABILITY

光谱两端：Devin 全自主跑完才给看；Cline / Aider 每步要 approval。 中间是 Claude Code / Cursor——默认信任,关键 action 才停。

自主→快但易跑偏；可控→慢但安心
TRADE-OFF · 02
多 Agent vs 单 Agent + Subagent
MULTI vs SINGLE+SUBAGENTS

Cognition "Don't Build Multi-Agents":多 agent 共享 context 难、信息衰减。 Anthropic 走另一条路——single agent 主导 + 临时 subagent 隔离上下文。

多 agent 适合并行可分解;单 + subagent 适合复杂连续
TRADE-OFF · 03
动态选 tool vs 固定 tool set
IN-CONTEXT vs PRE-DEFINED

固定:可预测可缓存,但场景切换 tool 选错率高。 Claude Code 走动态(skills),Aider 走固定(diff/edit/shell 三件套)。

动态 gating 需要场景识别——多一层判断逻辑
TRADE-OFF · 04
同步 vs 异步执行
SYNC vs ASYNC

同步:用户坐着等,体验直接。异步:提交后离开半小时回来看—— Devin / Codex Web 走异步,适合 long-running > 30 分钟。

同步简单;异步是分布式系统问题
RECAP

核心收束 · Harness 决定 Agent 能否走出 demo

上一讲 Context Engineering 讲的是"怎么把对的信息塞进模型脑子"。 今天 Harness Engineering 讲的是"怎么让模型脑子持续运转 8 小时不崩"。 两者合起来——Context 是"输入"，Harness 是"运行环境"—— 构成了 2025 年所有严肃 Agent 系统的工程底座。

"模型能力决定 Agent 上限的 80%，
Harness 决定能不能稳定到这个上限。"

6 大核心组件——Loop Control、Tool Gating、Interruption、 Checkpointing、Error Recovery、Auto-Compact——缺一不可。 4 大失败模式——Tool Soup、Context Bomb、死循环、无 Resume—— 都是 demo 阶段看不到、生产阶段灾难性的坑。

对我们项目最实用的建议： ① 不要从零造大而全的 harness——从 80 行 MVP 起步； ② 先把 checkpoint 和 interruption 做对——这是 long-running 的生死线； ③ Tool gating 比"加更多 tool"更重要——少即是多； ④ 想清楚同步 vs 异步——决定整个 harness 形态； ⑤ 多 agent 不是默认答案——能用 single+subagent 就别上多 agent。
