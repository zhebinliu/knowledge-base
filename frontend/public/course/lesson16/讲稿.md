## 开场(2分钟)

进入第四部分了。第四部分一共4讲——**企业级落地**。前三部分讲的是"AI能做什么"、"怎么调通"、"怎么工程化"。第四部分讲的是"怎么在企业里真正活下来"。

今天第16讲,我只聊一件事——**知识库**。

为什么单独拎出来一讲讲知识库?因为我们做了三年项目,有一个非常清晰的观察:**所有企业级AI项目的成败,几乎都卡在知识库这一层**。Prompt写得再漂亮、Agent设计得再精巧、模型用得再贵——如果底层知识库工程化不到位,上面全是空中楼阁。

我先抛一个真实案例。

去年底我们交付了一个客户——华东某制造业头部企业,1万员工,我们给他们做了一套基于RAG的"销售知识助手"。

**Demo阶段满意度80%**。客户领导亲自试,问"我们去年Q3那个新华三的SOW交付节奏是怎么样的",AI对答如流,带溯源、引页码、列附件——客户当场拍板,要扩到全集团200个销售用。

我们也很高兴,觉得这事成了,准备进入Q3扩展期。

**两个月后,扩到 200 人,满意度掉到 35%**。客户投诉雪花一样飘过来——"我问自己的合同它给我看别人的合同——这是泄密"、"上周改的报价表 AI 还在引用上上周的旧版本"、"同一问题三次答案都不一样"、"查得越来越慢平均 8 秒"、"这个月光 AI 接口费就花了 18 万"。

我们派人去现场排查,**一查发现三个大雷**:

1. **权限边界**——Demo阶段只有一份测试数据,所有人看一份;扩到200人后,客户A的销售能搜到客户B的合同,这是合规事故。
2. **文档增量同步**——飞书云文档里的合同每天都在改版,但我们的向量库是Demo时一次性导入的,**根本没有增量更新机制**,所以"上周改的报价表"永远不会进库。
3. **查询成本失控**——200人一天5000次查询,每次平均8000 token,光是Claude API一个月就15万——比当初Demo阶段烧的100倍。

我们花了6周才把这三个雷彻底扫掉。但回头看,**这三个雷在Demo阶段就该提前预防——不是事后救火**。

**这就是今天这一讲的全部主题**:从一个能跑通的RAG Demo,到100个团队同时用、可治理可演进的企业级知识库,中间隔着11道坎。今天讲清楚每一道坎,以及2026年最新的工程化解法。

打开 demo.html,我们一起把这件事系统讲清楚。

## 第一部分:Demo vs 生产的9大差距(7分钟)

【看第一节差距表】

我列了9个维度,你可以挨个点开看。今天我们重点讲其中5个——也就是我们项目踩过最深坑的5个。

### 差距1 · 数据接入

【点击"数据接入"那一行】

**Demo**:一份PDF,手工拖进上传框,跑通RAG。

**生产**:N个系统持续吐文档。光是我们项目里就有——飞书云文档、企业微信群文件、SharePoint(很多客户用)、邮件附件、CRM里的合同、工单系统的解决方案、Confluence的内部Wiki……每天新增几百份,有些还要每小时一次。

这种规模手工是没法做的。你需要的是**CDC——Change Data Capture**——监听源系统的变更事件,增量同步、失败重试、删除回传。

2026 年主流工具:**Airbyte AI Connectors**(一键接入 100+ 源)/ **Fivetran HVR**(数据库 CDC 老牌)/ **Unstructured.io Platform**(文档源专用)/ **LlamaHub**(LlamaIndex 集市,500+ connector)/ **飞书/钉钉/企微 webhook**(必须自写,难度不大)。

**实战建议**:从第一天就用 CDC,不要用"先跑通再说"的导入脚本。我们最早就是用脚本,后来扩展时改造极痛。

### 差距2 · 文档解析

【点击"文档解析"那一行】

**Demo**:PyPDF或pdfplumber抽文本,中文乱码就硬吃。

**生产**:扫描件OCR、表格结构化(行列关系)、图片caption、公式LaTeX化、版面恢复(标题/段落/页眉页脚分离)。

我们项目踩过最经典的坑——一份 SOW 的"附件A 报价表"是 3 列 8 行。PyPDF 抽出来变成散乱数字串"模块工期验收标准 CRM线索池 2周...",**行列对应关系完全丢失**。销售一问"该模块工期是多少天",AI 就开始瞎编。

**2026 年主流工具**:**Unstructured.io**(通用 40+ 格式,生产首选)/ **LlamaParse**(GPT-4 视觉级,贵但准)/ **DeepDoc**(RAGFlow,中文合同/年报碾压)/ **阿里 mPLUG-DocOwl 2**(国产多模态)/ **Marker**(开源,PDF→Markdown 带表格)。

**实战建议**:不同文档类型用不同解析器。合同/财报用 LlamaParse / DeepDoc;一般 Word / PDF 用 Unstructured。一刀切方案会有 30% 文档翻车。

### 差距3 · 切片策略

【点击"切片策略"那一行】

**Demo**:固定512 token窗口,前后重叠50,完事。

**生产**:这是2026年RAG领域进步最大的方向。三个新概念你必须知道:

**① Layout-aware chunking**——按文档的标题层级切,不要把"3.1节"和"3.2节"切到同一个chunk里。一个chunk内部语义统一,召回精度立刻涨5-10%。

**② Late chunking**(Jina 2024年提出)——先把整文档过embedding,**然后**再切片。这样每个chunk的向量自带"整文档的上下文"。对长文档效果惊人。

**③ ColPali / ColQwen**——多模态切片。不再把图变成caption再embed,而是**直接对图块embed**(用类似ColBERT的late interaction)。一份产品手册的截图,可以直接被"线索池界面长什么样"召回。

**实战数字**:一份 300 页年报,固定 512 切 2000 块,layout-aware 只切 600 块。我们项目实测从固定切换成 layout-aware + late chunking,**召回准确率 68% → 83%**。

### 差距4 · Embedding更新

【点击"Embedding更新"那一行】

**Demo**:OpenAI text-embedding-3-small跑一次,向量入库,结束。

**生产**:新文档要实时增量embed、Embedding模型升级时**全量回填**、旧版本向量保留一段时间做A/B、不同语言/领域用不同模型分库。

最痛的是**模型升级**。Embedding 模型一年大概升一次大版本,每次升级你要决定——是全量回填(贵但一致),还是新文档用新模型旧文档保留(便宜但混排)。我们项目去年从 ada-002 升级到 BGE-M3,**1200 万向量全量回填,本地算力 48 小时跑完,业务无感**。前提是元数据要带 `embedding_model_version`,否则升级时根本不知道哪些要回填。

**2026 年主流 embedding 模型**:**BGE-M3**(智源,中英双语 + 多向量,中国项目首选)、**Cohere Embed v4**(多语言多模态商业 API)、**Voyage-3-large**(Anthropic 推荐)、**OpenAI text-embedding-3-large**、**阿里 GTE-Qwen2**(中文强)。

**性能数字**:BGE-M3 本地推理 A10 GPU,单条 8ms,batch 32 单条 1.2ms。Cohere v4 API 平均 80ms 一次。

### 差距5 · 权限边界(这是企业级最关键的)

【点击"权限边界"那一行】

**Demo**:所有人看所有文档。

**生产**:三种方案,每种都有自己的安全/性能权衡:

**① per-tenant index(多租户物理隔离)**——每个客户/团队一个独立的 collection。最安全,但贵——100个团队就要100个 collection。运维成本高。
**② per-row ACL(文档级权限)**——所有文档放一个 collection,每条记录带 acl 字段。性能好,但权限模型复杂。
**③ 检索后过滤**——召回 top_k 不带过滤,然后在结果里筛权限内的。**这是最危险的方案**——召回100个全是别人的,过滤完只剩0条,用户以为"系统没数据"。

我们项目用的是**方案②的变种**:Qdrant 的 payload filter,在向量检索的**同时**做权限过滤——`filter: {tenant_id: 'xxx', acl: {any_of: user_roles}}`。**性能数字**:1000 万向量加权限 filter,P99 召回 ~80ms;若用方案③(后过滤),同规模 P99 涨到 400ms,且 top_k=5 有 30% 概率被过滤光。

【其他四个差距点开看,这里不一一展开:可观测、灰度发布、成本控制、版本治理、文档解析——加起来9条,每条都是Demo做不到的工程量。】

## 第二部分:文档处理流水线(7分钟)

【滚动到第二节流水线】

光说差距太抽象,看一份真实文档怎么走完5个阶段——**采集 → 解析 → 切片 → 嵌入 → 召回**。

我准备了3种典型文档,每一种都不一样。先看SOW合同。

【点击 SOW合同 按钮,然后逐个点 STAGE 卡片】

### Stage 1 · 采集(INGEST)

【点击 STAGE · 01】

看 body 部分。这是我们项目飞书 webhook 抓到一份 SOW 更新事件后的真实入队数据。

注意几个关键字段:
- `change_event: "飞书 webhook · doc.updated"`——这是CDC增量同步,不是手工上传
- `version: "v3 (之前v2为2026-04-18)"`——版本追溯,可以回查
- `tenant_id`, `project_id`——后面权限过滤的基础

Demo 阶段你不会有这些字段。生产阶段缺一个都会出事。

### Stage 2 · 解析(PARSE)

【点击 STAGE · 02】

我们用 Unstructured.io 做主解析器,LlamaParse 做 fallback。一份 12 页 SOW 解析出 147 个 elements——段落、标题、表格、列表。

**最关键的看表格那一段**:
```
{ type: "Table",
  html: "<table>...</table>",
  md: "| 模块 | 工期 | 验收标准 |\n| --- | --- | --- |\n| CRM线索池 | 2周 | ... |",
  rows: 8, cols: 3 }
```

**表格同时保留 HTML 和 Markdown 双形态**——切片用 Markdown(LLM 友好),溯源用 HTML(前端渲染)。这是生产细节,Demo 阶段没人care。

### Stage 3 · 切片(CHUNK)

【点击 STAGE · 03】

策略写得很明确——**layout_aware(按 Title level 边界)+ 表格不切**。一份 SOW 切出 23 个 chunk,平均 487 token。

**注意 metadata**:
```
metadata: {
  doc_id: "sow_xinhua",
  version: "v3",
  page: 4,
  tenant_id: "t_xinhua_001",
  acl: ["sales:xinhua", "delivery:xinhua"]
}
```

`tenant_id` 和 `acl` 是切片阶段就要塞进去的——这是后面"权限检索时过滤"的基础。**很多团队是在召回结果上再过权限,那已经晚了**。

### Stage 4 · 嵌入(EMBED)

【点击 STAGE · 04】

dense + sparse **双向量**——dense 用 BGE-M3,sparse 用 BM25。这两个都要写进 Qdrant,后面查询走 hybrid search。

**老向量(v2)保留 30 天**——这是给"出问题可回滚"留的窗口。Demo 阶段一删就没了,生产必须留。

成本:本地 BGE-M3 推理,几乎无 API 成本,只有算力。

### Stage 5 · 召回(RETRIEVE)

【点击 STAGE · 05】

这是用户真正发问的时刻。三步:

**① Query 改写**——用户问的口语化("新华三这单的CRM线索池模块工期多久,验收标准是什么?"),LLM 重写成检索 query("新华三 SOW CRM线索池 工期 验收标准")。
**② 检索 + 权限 filter 同时进行**——这是关键。filter 不是后过滤,是 Qdrant 在向量搜索的同时做的。
**③ Rerank**——召回 top_k=10,bge-reranker-v2-m3 重排,留 top_n=3。

**最终延迟 P99 = 187ms**。这是生产可接受的水平。Demo 通常 1-3 秒。

---

【切换到访谈记录】访谈类文档完全不一样——Stage 1 多了 `needs_redact: true`,走"先脱敏再索引"流水线(我们项目最近 Block 2 做的事)。Stage 3 用 semantic chunking 而非 layout-aware——访谈没有清晰标题结构,要按语义边界切。

【切换到产品手册】80 页改了 7 页,**只重新 embed 这 7 页对应的 chunk**,其余 149 个复用 v3.1 的向量,省 90% 成本。Stage 4 出现了 **ColPali 多模态嵌入**——123 张截图直接对图块 embed,用户问"线索分配规则在哪个菜单"可以直接召回出截图。**这就是 2026 年 RAG 的新能力**——图片不再是 caption 的附属品,而是一等公民。

## 第三部分:向量数据库选型(5分钟)

【滚动到第三节决策树】

向量库选型是 RAG 项目最容易出错的决策之一。**选错一次,迁移成本 = 半年工程量**。我自己亲身经历过——从 Pinecone 迁到 Qdrant,1500 万向量,光是数据迁移加索引重建就 3 周,业务影响 1 个月。

【看决策树,6 个问题】

我把 2026 年的主流向量库选型逻辑做成了决策树。回答 6 个问题——数据量、QPS、是否需要hybrid、是否需要图能力、预算、自管 vs 托管——给你推荐方案。

【按默认选项,展示推荐结果:Qdrant】

默认配置——100 万到 1 亿向量、10-500 QPS、要 hybrid、不要图、月预算 3-30k、自管——给出的推荐是 **Qdrant**。这是我们项目用的方案,也是 2025-2026 中国项目的事实标准。

### 主流向量库 2026 选型口诀

**Qdrant**(Rust 性能稳,中国项目首选,我们用的)/ **Milvus**(分布式 + GPU,10 亿级以上)/ **Weaviate**(GraphQL 友好,功能近 Qdrant)/ **pgvector**(已用 PG 时无敌,业务 join 杀手锏,百万级 50ms)/ **TurboPuffer**(serverless,冷数据极便宜)/ **LanceDB**(本地优先,嵌入式)。

**选型口诀**:不知道选啥 → Qdrant;已用 PG → pgvector;数据超 5 亿 → Milvus;要 GraphRAG → Neo4j + 向量插件;不想运维 → TurboPuffer / Pinecone Serverless。

### GraphRAG 和 Hybrid Search 简单说一下

**GraphRAG**(微软 2024 提出,2025 出 v2 + LazyGraphRAG)——把文档构造成"实体-关系图",检索时不只检索文本,还检索图上的"社区摘要"。**适合实体关系密集的领域**——医疗、法律、学术。CRM 场景里如果实体关系简单(客户-销售-合同),GraphRAG 价值不大,反而增加复杂度。

**Hybrid Search**——BM25(稀疏向量,关键词匹配)+ Vector(稠密向量,语义匹配)用 RRF(Reciprocal Rank Fusion)融合。**中文场景必须用**——中文的语义匹配经常会"丢词",比如用户问"线索池配置",纯向量召回可能漏掉只有"线索池"三字的核心文档。BM25 补这个短板。

**ColBERT / Late Interaction**——更高级的方案,每个 token 都有独立向量,召回时做 max-sim 计算。比 dense vector 召回精度高 10%+,但存储和计算贵 10倍。2026 年开始用于"高价值低频"场景(法律、医疗)。

## 第四部分:1→100 团队扩展模拟器(5分钟)

【滚动到第四节扩展模拟器】

最后这个演示最直观——拖动三个滑块感受规模的非线性变化。

### 起点:1 个团队,5 万文档,20 万次查询/月

【调整滑块到 teams=1, docs=5, queries=20】

看到了——
- **月成本 ¥6,400 左右**
- **单查询 ¥0.032**
- **P99 延迟 ~580ms**
- **推荐架构:原型级**——Qdrant 单节点 + BGE-M3 本地 + Redis,1 台 4 核 8G

风险点不多——文档版本治理、评估集兜底。一个工程师兼职就能维护。

### 扩展到:20 个团队,50 万文档,200 万次查询

【拖动到 teams=20, docs=50, queries=200】

- **月成本飙到 ¥4-5 万**
- **单查询 ¥0.02 左右**(缓存命中率上来了)
- **P99 ~850ms**
- **推荐架构变成 SMB 级**——Qdrant 升 16 核 32G,异步 embed worker、Langfuse trace 必装

风险点新增——**权限边界**、**Embedding 升级回填**、**评估集要更新**。**至少 0.3 个 FTE 专人维护**。

### 扩展到:100 个团队,500 万文档,2000 万次查询

【拖动到 teams=100, docs=500, queries=2000】

- **月成本 ¥30 万+**(警告变红)
- **单查询 ¥0.015** —— 缓存命中率到 70%
- **P99 1500ms+**(警告)
- **推荐架构:大型企业级**—— Milvus / Zilliz 集群 + 多区域 + per-tenant 隔离 + 完整 MLOps

风险点全亮:**多租户物理隔离**、**LLM 供应商单点**、**冷热分层**、**热点查询限流**、**24/7 oncall**。**3-5 人专职平台团队 + 1 个 SRE**。

### 这个模拟器告诉我们什么?

**关键洞察 1**——成本不是线性的,**有阶梯**。从 1 团队到 10 团队,成本可能涨 5 倍。从 10 到 50 涨 6 倍。从 50 到 100 涨 4 倍(因为缓存命中率提升、单位查询成本下降)。但运维人力**完全是阶梯式**——从 0.1 FTE 到 0.3 FTE 到 1-2 人到 3-5 人。

**关键洞察 2**——架构会**质变**。Qdrant 单节点能撑到大约 1 亿向量、1000 QPS。再上就必须换 Milvus / 分布式方案,或者切到云托管。**不要等到崩才换,要提前 3-6 个月规划**。

**关键洞察 3**——风险点会**叠加**。100 团队规模下,任何一个雷爆了都是 P0 事故——权限泄露、向量库挂掉、LLM 限流。所以你看 demo 里 100 团队规模时,风险点最多——这是真实的。

**实战建议**:每个项目立项时,先用这个模拟器**算清未来 12 个月的目标规模**——成本、架构、人力都要算。**不要先做 Demo 再想扩展**——架构债是利滚利的。

## 收尾(2分钟)

【滚动到总结】

把今天的核心收束成一句话:

> **能做 Demo 的人多,能把 Demo 撑到企业级的少。**

知识库工程化是 AI 落地的**护城河**。模型是公共能力,Prompt 容易抄,Agent 框架开源遍地——只有**为你的企业沉淀知识、加工知识、治理知识的那一套基础设施**,是别人抄不走的。

**对我们项目最实用的 4 条建议**:

**① 知识库从第一天就规划权限模型**。tenant_id / project_id / acl 三件套从 schema 设计开始就埋进去。后补成本极高——我们项目改一次权限模型,花了 4 周,影响业务 2 周。

**② CDC 增量同步替代手工上传**。即使 Demo 阶段就用 Airbyte 或 LlamaHub 接入飞书 webhook,不要图省事写个 `upload_pdf.py`。手工脚本最后都得推倒重来。

**③ Embedding 模型明确升级回填策略**。元数据里加 `embedding_model_version` 字段,模型升级时全量回填,旧向量保留 30 天做 A/B。我们项目 BGE-M3 → 升级时 1200 万向量回填,48 小时业务无感——靠的就是这套策略。

**④ 查询级 Trace 必装**。Langfuse 开源 + 自建 PostgreSQL 看板。**召回质量(recall@k, MRR)和答案质量(LLM-as-judge / 用户点赞)要分离监控**——不然出问题永远定位不到是检索的锅还是生成的锅。

### 回到开场那个案例

华东那家制造业客户的项目,我们用 **6 周时间** 救回来:第 1-2 周补 per-row ACL,第 3 周上飞书 webhook 增量同步,第 4 周接入 Langfuse trace,第 5 周上 Redis 语义缓存 + LLMLingua prompt 压缩,第 6 周灰度发布。

**结果**:用户满意度 35% → 78%、单查询成本 ¥0.18 → ¥0.038、P99 延迟 8 秒 → 1.2 秒、月度 AI 成本 18 万 → 6.5 万。**这就是知识库工程化的价值**——Demo 阶段看起来"差不多",生产阶段差的是 50 倍的工程量,以及差的是"能不能继续做下去"。

## 备注:常见疑问

**Q: 我们刚起步,要不要直接上 GraphRAG?**
A: 不要。GraphRAG 复杂度高 3-5 倍。**先把基础 RAG 跑顺,有可观测、可评估,再考虑 GraphRAG**。我们项目至今没上——基础 RAG + Hybrid + Rerank 已经覆盖 90% 场景。

**Q: 向量库一定要选 Qdrant 吗?**
A: 不一定。"不知道选啥"就 Qdrant(Rust 性能稳、原生 hybrid、payload filter 强、社区版够用、私有化简单)。已用 PG 就 pgvector;数据超 5 亿就 Milvus / Zilliz。

**Q: Embedding 模型选 BGE-M3 还是商业 API?**
A: 国产项目 90% 用 BGE-M3(开源、本地、零 API 费),英文密集场景用 Voyage / Cohere v4。

**Q: 1000 万向量规模,月预算 5 万够吗?**
A: 紧但够。Qdrant 3 节点 ¥1.2 万 + A10 GPU embedding ¥0.8 万 + LLM(月查询 100 万,缓存 50%)用 DeepSeek ¥2 万 / 用 Claude Opus ¥6 万 + 应用层 ¥0.5 万。**模型选型直接决定单月成本翻倍**。

**Q: 评估集多久跑一次?**
A: Prompt / 模型 / 切片策略变更立即全量跑。日常每天跑一次回归。**每周从真实流量采样 50-100 个 case 进评估集——评估集是活的**。
