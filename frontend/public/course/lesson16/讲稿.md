## 开场（2 分钟）

PART 3 最后一讲。

我先抛个真实场景。

去年我们一个合作团队，拿 GPT-4 + LangChain 拼了一个"自动写代码 agent"。Demo 阶段效果惊艳——产品给个需求，agent 跑 10 分钟，自动写出 3 个文件，跑过测试。客户当场买单。

上线两个月后，团队来找我们救火。问题清单：

- 跑长任务（≥ 1 小时）<strong>必崩</strong>，原因不明
- 用户中途 Ctrl+C，所有状态丢失，下次<strong>从头跑</strong>
- 同样的 tool error，agent <strong>原样重试 30 次</strong>，一晚上烧了 8000 块
- 偶尔遇到 context 满了直接抛异常，<strong>没有任何恢复机制</strong>
- 同一个 prompt，今天跑 OK，明天跑出来内容完全不对，无法复现

团队的工程师一直在调 prompt、换模型、加 retry——<strong>都没用</strong>。

我看了一眼代码，问题不在 agent，<strong>在 harness</strong>——他们根本没有 harness，只有一个 `while True: model.invoke(prompt)`。

这就是今天要讲的主题。

上一讲（L15 Context Engineering）讲怎么把"对的信息塞进模型脑子"。今天讲它的兄弟问题——<strong>怎么让模型脑子持续运转，跑 8 小时不崩</strong>。

## 第一部分：Agent vs Harness 的分界（4 分钟）

【看第一节对比表】

2025 年随 Claude Code、Agent SDK、Cursor Composer、Devin 这波 agent 系统大量浮现，一个关键工程概念被反复强调——<strong>Agent 是策略，Harness 是骨架</strong>。

这不是我编的，这是 Anthropic 的 Boris Cherny（Claude Code 团队负责人）在工程笔记里反复说的。Cognition 的 Walden Yan 在那篇有名的 "Don't Build Multi-Agents" 里也是这个区分。Karpathy 在 2024 年的 talk 里把这个比喻得更直观——<strong>Agent 是大脑，Harness 是骨骼 + 肌肉 + 神经系统</strong>。

我们逐条看分界：

### 维度 1：职责定义

<strong>Agent</strong> —— Decide what to do。每一步决定调哪个工具、生成什么内容、推理路径走哪边。这是模型的"判断力"。

<strong>Harness</strong> —— Make sure it keeps running。保证循环不崩、中断能恢复、context 不爆、错误可降级。这是工程师写的"基建"。

### 维度 2：实现载体

<strong>Agent</strong> —— Prompt + 模型 + 工具集。靠 Claude / GPT-4o / Gemini 的推理能力。

<strong>Harness</strong> —— 主循环代码 + 状态机 + 持久化。靠工程师写的 Python / TypeScript。

### 维度 3：变化频率

<strong>Agent</strong> —— 每次模型升级、prompt 调优都在变。是"软"层。

<strong>Harness</strong> —— 一次写好、半年不动也能跑。是"硬"层。

这就解释了为什么 Claude Code 团队不动 harness 也能跑通 Sonnet 3.5 → Sonnet 4 → Sonnet 4.5 → Opus 4.7 的多次模型升级。Harness 不变，agent 自动变强。

### 维度 4：谁来负责

<strong>Agent</strong> —— 产品 + AI 工程师。设计 prompt、调 tool、验质量。

<strong>Harness</strong> —— 平台 / 基建工程师。保证可靠性、可观测、可恢复。

我们项目里这个分工也必须想清楚——<strong>不要让 AI 工程师同时背 harness 的锅</strong>。两件事都做做不好。

### 维度 5：失败模式

<strong>Agent</strong> —— 幻觉、走错路、tool 选错——"质量"问题。

<strong>Harness</strong> —— 死循环、超时、context 炸、中断丢失——"可用性"问题。

这两类问题<strong>需要完全不同的修法</strong>。质量问题改 prompt / 加 few-shot / 升模型；可用性问题改代码 / 加 checkpoint / 改主循环。混在一起改就是越改越糟。

### 类比：司机与汽车

司机的"判断"——往左还是往右，加速还是刹车——是 Agent。

汽车的"底盘 + 安全带 + 油表 + 故障灯"——保证司机能开——是 Harness。

司机再厉害，开一辆没刹车没油表的破车，<strong>开不了多远</strong>。

【看下面绿色块】

### 2025 年的核心认知转变

以前大家盯着"换更好的模型"，以为模型够强 agent 就能稳。

但 Claude Sonnet 4.5 / Opus 4.7 已经强到很难再换的程度了。再用同样的 prompt 在 Opus 4.7 上跑，质量提升边际递减。

<strong>瓶颈不在策略，而在骨架</strong>。

这就是为什么 2025 年所有严肃 agent 团队都在做 harness——而不是只调 prompt。一句话总结：

> 模型能力决定 Agent 上限的 80%，<br>
> Harness 决定能不能稳定到这个上限。

## 第二部分：Harness 6 大核心组件（7 分钟）

【滚动到 DEMO 1 · 6 组件全景】

把市面上所有严肃 agent 系统（Claude Code、Cursor、Devin、Aider、Cline、Codex CLI）的 harness 拆开看，最终都会沉淀出这 6 大组件。<strong>缺一个，agent 就会在某种场景下崩</strong>。

【点击 COMPONENT 01】

### 组件 1：Loop Control · 主循环控制

所有 harness 的心脏——一个 while 循环，决定"什么时候继续 / 什么时候停"。

<strong>典型实现</strong>：
```python
while step < max_iter and not done:
    response = model.invoke(...)
    if response.stop_reason == "end_turn": break
    execute_tools(...)
    step += 1
```

<strong>停止条件</strong>三层：
1. 自然停——模型说 end_turn
2. 硬上限——max_iterations（Claude Code 默认 200，Cursor Agent 默认 25）
3. 软上限——成本预算 / 时间预算

<strong>典型坑</strong>：不设上限。一晚上烧 8000 块的故事每个团队都有。我们项目里也踩过——一个 Agent 在 retry 一个永远会失败的 tool，跑了 6 小时，烧了 1200 美金。

【点击 COMPONENT 02】

### 组件 2：Tool Registration & Gating · 工具注册与门控

把 tool 注册进 harness，再决定每一步给模型暴露哪些。<strong>Gating 是 2025 最被低估的优化</strong>。

<strong>典型实现</strong>：
```python
TOOLS = {}  # 注册表: name -> (schema, fn)
def gate(scenario, all_tools):
    return [t for t in all_tools if t.cat in scenario_categories[scenario]]
```

<strong>Claude Code 怎么做</strong>：Skills 系统——每个 skill 是一个 tool subset，根据当前任务 description 动态加载。比如"调研型任务"自动加载 grep / read / glob skills；"部署任务"自动加载 docker / k8s skills。

<strong>Cursor 怎么做</strong>：Agent Rules 文件 + 文件类型自动决定暴露 lint / format / test 哪些工具。

<strong>常见坑</strong>：
- tool 描述写太短——模型不知道什么时候用 / 什么时候不用
- tool 之间职责重叠互相竞争（如 search_code 和 grep）
- 一次暴露 50 个 tool——"tool soup"，模型迷失

<strong>实战建议</strong>：一次最多暴露 8-12 个 tool；tool 描述必须包含 "when to use / when NOT to use"。

【点击 COMPONENT 03】

### 组件 3：Interruption Handling · 中断处理

用户 Ctrl+C 时，agent 不应"丢盔弃甲"——所有进行中的工作要安全保存。

<strong>典型实现</strong>：
```python
interrupted = [False]
signal.signal(signal.SIGINT, lambda *a: interrupted.__setitem__(0, True))

while running:
    if interrupted[0]:
        save_checkpoint(state)
        print("⏸ saved. resume with run()")
        return
```

<strong>Claude Code 怎么做</strong>：`ESC` 暂停当前 tool 调用并保留 context；`Ctrl+C 二次`退出但保留 session。`/resume` 列出最近 session 让你选。

<strong>Cursor 怎么做</strong>："Stop" 按钮停止当前 generation，但保留对话历史。Composer 的 multi-file edit 支持 partial accept / reject。

<strong>常见坑</strong>：中断时 tool 还在跑——保存的 state 是"半完成"的，resume 时不知道这步到底有没有做完。

<strong>修复</strong>：把 tool 调用做成幂等；中断时记录 "tool call 已发出 / 已返回 / 已应用" 三态。

【点击 COMPONENT 04】

### 组件 4：Checkpointing · 检查点

每一步都保存状态——崩了能续，跨会话能恢复。<strong>这是 long-running 任务的生死线</strong>。

<strong>典型实现</strong>：
```python
def save_ckpt(state):
    pickle.dump(state, open(CKPT_FILE, "wb"))

# 主循环里每完成一个 tool call 后:
save_ckpt(state)
```

<strong>Claude Code 怎么做</strong>：`~/.claude/sessions/<id>.jsonl` 全量 trace，`/resume` 列出最近 session 让你选。每次 tool call 都 append 一行 JSON——既是 checkpoint 也是 trace。

<strong>Cursor 怎么做</strong>：Conversation history 存 SQLite，可随时回到任意历史 checkpoint 重新分支。

<strong>常见坑</strong>：
- checkpoint 太大——messages 几百 K，pickle 失败
- 太频繁——每秒 IO 把磁盘打爆
- 漏存关键状态——只存 messages 不存 working directory diff，resume 时文件状态对不上

<strong>实战建议</strong>：checkpoint 粒度 = 每个 tool call 边界；存量超过 N 个时 rotate 旧的；只存增量更高效。

【点击 COMPONENT 05】

### 组件 5：Error Recovery · 错误恢复

tool 失败、API 429、模型超时——不应直接终止，应分级降级。

<strong>分级策略</strong>：
- L1：瞬时错误（429 / 5xx / timeout）→ exponential backoff 重试 3 次
- L2：tool 错误（fn raise exception）→ 把 ERROR 字符串返回给模型，让模型自决
- L3：模型主错误（refuse / loop）→ fallback model（Opus → Sonnet）
- L4：连续失败 ≥ 3 次 → escalate 给人工

<strong>Claude Code 怎么做</strong>：内置 5xx / 429 自动 retry；tool 异常返回字符串 ERROR 让模型自己决定下一步；不会"假装成功"。

<strong>常见坑</strong>：<strong>"原样重试"导致错误放大</strong>——同一个 input 错了 30 次还在试，token 烧光。

<strong>修复</strong>：同 input 连续失败 ≥3 次必须中断；不同错误码用不同策略。

【点击 COMPONENT 06】

### 组件 6：Auto-Compact · Context 自动压缩

Context 接近 200k 时，harness 主动 summarize 前半部分，腾出空间继续跑。<strong>这是 long-running 任务能突破单轮 context 限制的关键</strong>。

<strong>典型实现</strong>：
```python
if estimate_tokens(messages) > 150_000:
    # 保留 system prompt + 用户原始任务 + 最近 K 轮原文
    # 中间用另一个 LLM call 压成 summary
    messages = compact(messages)
```

<strong>Claude Code 怎么做</strong>：`/compact` 显式触发，或自动触发——保留最近文件操作 + summary + system prompt。compact 后会明确告诉模型"前文已压缩"，避免模型误以为 context 完整。

<strong>常见坑</strong>：
- compact 时丢关键信息——比如用户的硬约束 ("不要改 auth 模块") 被压掉了，agent 接着干就闯祸
- prompt-cache 失效——compact 后 cache 全清空，下一轮成本翻倍

<strong>修复</strong>：system prompt + 用户原始任务 + 最近 K 轮永远保留；中间靠 summary。

### 6 组件如何协同

这 6 个组件不是孤立的——它们紧密协同：

- Loop Control 是骨架，<strong>所有其他组件都挂在它上面</strong>
- 每轮 loop 里：Tool Gating 决定暴露 → 模型推理 → Tool 执行 → Error Recovery 兜底
- 每个 tool call 后：Checkpoint 保存
- Context 接近上限时：Auto-Compact 触发
- 用户 Ctrl+C：Interruption Handler 触发 Checkpoint

<strong>缺一个，整个外壳就漏了</strong>。

## 第三部分：Tool Gating 实战演示（4 分钟）

【滚动到 DEMO 2 · Tool Gating】

我把 Tool Gating 拎出来单独讲，因为这是<strong>最容易被低估、最容易做错</strong>的 harness 决策。

【看演示】

总共 28 个 tool。横跨文件 / 搜索 / shell / git / 测试 / 部署 / 调试 7 个类别。

我做了 5 种场景。我们逐个看：

### 场景 1：代码编辑【点击】

暴露 6 个 tool：read_file / edit_file / write_file / list_dir / grep / glob。

<strong>关键决策</strong>：
- edit_file 优先于 write_file——diff 比全量更安全
- <strong>delete_file 不暴露</strong>——删除文件 risk 太高，需要用户单独确认
- 没有 shell / git / deploy 工具——编辑阶段不该碰这些

这是 Cursor Composer 默认 tool set 的思路。

### 场景 2：文件搜索【点击】

暴露 6 个 tool：read_file / list_dir / grep / glob / codebase_search / web_search。

<strong>关键决策</strong>：<strong>没有任何写权限</strong>。这是最安全的 subset，所有"调研类"任务先走这层，明确目标后再切换到编辑场景。

Claude Code 的 `plan` 模式正是这个思路——先调研、出 plan、用户确认、再进编辑。

### 场景 3：测试运行【点击】

暴露 test 工具 + 读类工具。<strong>不暴露 write_file</strong>——agent 不应该在跑测试时偷偷改代码。

这是 Aider 的工作流：编辑 → commit → 测试 → 失败回到编辑。每个阶段 tool set 是清晰隔离的。

### 场景 4：部署【点击】

暴露部署专用 tool。<strong>这是最危险的场景</strong>——每个 deploy tool 都标记为 danger=true，harness 应在 tool 调用前强制弹出确认对话框。

Devin 的部署流程默认要求人工 approve。

### 场景 5：调试【点击】

暴露 7 个 read/inspect 工具，<strong>无写权限</strong>。

调试阶段需要先理解问题，不能边调边改。一旦 root cause 确认，切换到编辑场景做 fix。

### Tool Gating 的设计哲学

【看下面绿色块】

<strong>少即是多</strong>。

模型不是越多 tool 越能干。给它 50 个 tool，它会迷失；给它 8 个相关 tool，它能干得漂亮。

<strong>实战建议</strong>：
1. 按"任务场景"分组，一次暴露 8-12 个 tool
2. 写权限默认 gate（特别是 delete / deploy）
3. Tool 描述写 "when to use / when NOT to use"
4. 危险 tool 标记 danger，调用前强制确认

## 第四部分：8 小时长任务时间线（5 分钟）

【滚动到 DEMO 3 · Long-running Timeline】

光讲组件抽象。来看一个真实的 long-running 任务。

任务："重构 nest.js 模块到 fastify"。预计 8 小时。实际跑了 7 小时 28 分钟，消耗 $8.74。

时间线上有 11 个事件点，对应 4 种节点：

- ◆ <strong>Checkpoint</strong>（绿）——可回滚 / 可 resume 的安全点
- ⏵ <strong>User Intervene</strong>（蓝）——用户介入
- ✕ <strong>Error Recovery</strong>（红）——错误处理
- ⤓ <strong>Auto-Compact</strong>（橙）——context 自动压缩

【点击 CKPT-01】

### 00:08 · 初始 checkpoint

任务启动 8 分钟。Agent 完成 codebase 调研，识别出 12 个 controller、8 个 service。<strong>checkpoint 写入</strong>：messages + 文件清单 + 重构计划草案。

如果这一步用户中断，下次能直接拿到计划继续，不用重读代码。<strong>这就是 checkpoint 的价值</strong>。

【点击 CKPT-02】

### 00:58 · 第一批文件迁移完成

Agent 完成 4 个 controller 迁移（auth / user / company / dept），每个都过了对应单测。

<strong>checkpoint 包含</strong>：4 个新文件 diff、原文件 backup、测试结果。这是第一个"可以放心回滚到这里"的时点。

【点击 ERR】

### 01:46 · Tool 失败 · 重试 2 次后降级

`run_test` 工具超时——nest.js 旧测试和新 fastify 框架不兼容。

Harness 执行：
1. 第一次 retry（2s 后）失败
2. 第二次 retry（4s）失败
3. 不再 retry，把 ERROR 返回给模型
4. <strong>模型自决</strong>：跳过该测试，标记为 "tech-debt"，继续推进

<strong>这就是正确的错误处理</strong>——不放大、不假装、让模型理解失败上下文自己决策。

【点击 COMPACT】

### 02:28 · Context 自动压缩

Context 接近 200k，触发 auto-compact。

- 压缩前：152k tokens
- 压缩后：78k tokens

前 25 步压成 summary（包含已完成迁移清单、关键决策、tech-debt 项），保留最近 5 步原文 + system prompt + 用户原始任务。

<strong>注意</strong>：prompt-cache 在这里失效，下一轮成本会临时上升。这是 compact 的"代价"，不可避免。

【点击 USER】

### 03:02 · 用户介入 · 修正路线

用户回来 review 进度，发现 agent 把 `auth/jwt.ts` 改坏了——用了不兼容的 fastify-jwt 写法。

用户操作：
1. 暂停 agent
2. 给一条 instruction "auth 模块全部保留原 nest 实现，先不迁"
3. 点击 "resume from CKPT-02 + new instruction"

Agent 从 CKPT-02 重新出发，跳过 auth。

<strong>这就是 harness 给"人在环里"的价值</strong>——agent 跑歪了，用户能精确回滚 + 修正，不用从头来。

【点击 CKPT-03】

### 03:55 · 第二批迁移完成

Skip auth 后，agent 完成 product / order / payment 三个核心模块迁移。涉及数据库连接池、事务管理——最复杂的部分。

【点击 ERR-2】

### 04:38 · API 429 · 自动 backoff

Anthropic API 返回 429 rate limit。Harness 自动 backoff（2s / 4s / 8s 指数退避），<strong>无需任何代码改动 agent 继续</strong>。

<strong>关键设计原则</strong>：这种瞬时错误 harness 应该完全吸收，不要让模型"看见"——否则模型会困惑，可能开始反思"我是不是哪里做错了"。

【点击 COMPACT-2 / CKPT-04 / USER-2 / CKPT-FINAL】

后面几个节点的故事类似，最终 7h28m 任务完成，$8.74。

### 这个时间线告诉我们什么

<strong>没有 harness</strong>，这 8 小时会变成：

- 第 1 次 tool 超时——直接崩
- 第 1 次 context 满——直接抛异常
- 用户介入——只能从头跑
- API 429——直接报错给用户
- <strong>实际能跑通的概率：接近 0</strong>

<strong>有 harness</strong>，每个异常都被吸收 / 降级 / checkpoint，整个任务的"成功概率"从 < 5% 提升到 > 90%。

这就是 harness 在 long-running 任务上的<strong>本质价值</strong>。

## 第五部分：2025 主流 Harness 对比（4 分钟）

【滚动到 DEMO 4 · Harness Matrix】

2025 年 6 大主流 agent harness。每个在 8 个维度上做了截然不同的选择。

【看矩阵】

8 个维度：自主程度 / 用户审核 / 多文件编辑 / Git 集成 / Context 管理 / Subagent / Hooks / 开源。

【点击 Claude Code 行】

### Claude Code

<strong>当前最完整的开发型 agent harness</strong>。在 2025 年成为团队 daily driver 的首选。

差异化能力：
1. subagent + skills 系统——让 single agent 也能处理超大任务
2. hooks 系统（PreToolUse / PostToolUse / Stop / SessionStart 等 8 类）——让团队能注入自己的工程规范
3. 终端 + IDE 双形态——CLI 跑长任务、IDE 改文件

典型场景：8 小时重构、跨服务迁移、复杂 bug fix。

【点击 Cursor Composer】

### Cursor Composer

<strong>IDE 原生 agent 的代表</strong>。强项在"多文件协调编辑"——一个 prompt 改 8 个文件，每个文件给你 diff 让你 apply。

差异化能力：
1. Apply 模型（专门小模型）做精确 diff 应用
2. Agent Rules 文件给项目级约束
3. Tab 模型做 inline 补全

<strong>vs Claude Code</strong>：
- Cursor 更适合"我知道要改什么，让 AI 帮我写"
- Claude Code 更适合"我不太知道怎么改，让 AI 自己探索"

两者互补，团队里经常同时用。

【点击 Devin】

### Devin

<strong>最激进的全自主 harness</strong>。你给一个 Linear ticket，几小时后 Devin 给你一个 PR。

差异化能力：
1. 异步执行——不需要你盯着
2. GUI 可视化 reasoning 步骤（hover 看到它在想什么）
3. PR-native——产出是审核就绪的 PR

Cognition 团队的核心观点（来自 "Don't Build Multi-Agents"）：他们用 <strong>single agent + 强 context engineering</strong>，而不是多 agent。

缺陷：贵（$500/月起）、不可控（跑偏了你只能等它跑完）、不开源。

【点击 Aider】

### Aider

<strong>最受工程师社区喜爱的 OSS harness</strong>。设计哲学：保守 + git-first + 简单可读。

差异化能力：
1. Repo map——用 ctags 给模型 codebase 的 ToC，避免 context bomb
2. 每次 edit 自动 git commit——回滚就是 git revert
3. diff-based edit（不是全量重写）
4. 支持几乎所有主流 LLM 提供商

vs Claude Code：Aider 更"小巧锋利"，没有 hooks / subagent 这些花架子，但稳定可靠。工程师团队的 backup 选择。

【点击 Cline / RooCode】

### Cline / RooCode

<strong>open-source 阵营里最像 Claude Code 的 harness</strong>。RooCode 是 Cline 的 fork，差异在 UI / modes 系统。

差异化能力：
1. 每个 tool call 都弹 approve 对话框（最保守的可控性）
2. 支持多模型切换（包括本地 Ollama）
3. modes 系统让你切换 Architect / Code / Ask 等角色

典型场景：
- 不信任商业服务的团队
- 用本地模型或自有 API
- 想要"每步看清楚"的新手用户

【点击 Codex CLI】

### Codex CLI

<strong>OpenAI 2025 推出的官方 CLI harness</strong>。是对 Claude Code 的直接回应——基于 GPT-5 / Codex 模型。

差异化能力：
1. Web 版（Codex Web）异步运行，类似 Devin 但开放
2. Sandboxed 执行——每次 tool call 在隔离 container 里
3. OpenAI 模型生态深度集成

vs Claude Code：模型层是 GPT-5 vs Claude Opus（各有所长），harness 层 Claude Code 在 hooks / subagent / skills 上更成熟，Codex CLI 在 sandbox 安全上更激进。

<strong>趋势观察</strong>：2025 年下半年两者快速趋同，"agent harness"的形态在收敛。

### 怎么给团队选 harness

简单决策树：

- 团队 IDE 优先 → Cursor Composer
- 长任务 + 终端友好 → Claude Code
- 不信任 SaaS / 数据合规 → Cline + 本地模型，或 Aider
- 要求全自主 + PR 产出 → Devin（贵）
- OpenAI 模型为主 → Codex CLI

我们项目里目前 Claude Code 主力 + Cursor 辅助——这是 2025 主流团队的搭配。

## 第六部分：4 大失败模式（3 分钟）

【滚动到 Failure Modes】

团队自己造 harness 时<strong>最容易踩的 4 个坑</strong>。这些都是 demo 时看不出来、生产跑 1 小时就崩的那种。

### 失败 1：Tool Soup（注册 50 个 tool）

想着"功能丰富一点"，把 50+ 个工具一股脑全注册。

模型在每一步都要从这 50 个里挑一个，<strong>"选择困难症"被放大 10 倍</strong>。典型表现：模型反复在 read_file / list_dir / grep 之间打转，3 步能干完的事干了 12 步。

<strong>修复</strong>：按"任务场景"做 tool gating；一次最多暴露 8-12 个 tool；tool 描述写清楚 "when to use / when NOT to use"。

### 失败 2：Context Bomb（让 agent 自己读完整代码库）

给 agent 一个 "search_code(query)" 工具，结果模型为了"调研一下"，一上来就 read_file 二十几个大文件。

<strong>200k context 五分钟内就被填满</strong>，后面什么活都干不了，prompt-cache 全失效，账单飙升。

<strong>修复</strong>：read_file 默认只返回前 N 行（如 500 行）+ ToC，按需 expand；system prompt 明确"不要批量读文件，先用 grep / glob 定位"。

### 失败 3：死循环 / 错误放大

tool 返回了一个错误，agent 不会反思，原样重试 30 次。

或者更糟——agent 看到错误后觉得"我应该再补一个文件"，于是越改越糟，<strong>一晚上烧了 8000 块 token 钱</strong>。这是 LangChain 早期被骂得最惨的问题。

<strong>修复</strong>：
1. 同一个 tool 同一个 input 重复 ≥3 次强制中断
2. 设 hard max_iterations（如 200 轮）
3. 成本超阈值告警 + 暂停

### 失败 4：中断无 Resume 能力

agent 跑了 1 小时，做了 30 步，调用了 18 个 tool，写了 6 个文件。

用户 Ctrl+C 一下——<strong>所有状态全没了</strong>，下次只能从头再来。

这是 demo 阶段看不出来、生产阶段灾难性的失败模式。

<strong>修复</strong>：
1. 每完成一个 tool call 写一次 checkpoint
2. 中断时把当前 context + tool history + 工作目录 diff 全部序列化
3. resume 时回放 + 询问用户"从哪里继续"

## 第七部分：从 80 行 Python 起步（2 分钟）

【滚动到 Minimum Viable Harness】

讲了这么多，可能有同学觉得 harness 工程量大、做不动。

我给大家算笔账——<strong>最简 harness 80 行 Python 就能跑</strong>。

【看代码块】

这 80 行覆盖了 6 大组件的 5 个：

1. <strong>Loop Control</strong>——while 主循环 + max_iter
2. <strong>Tool Registration & Gating</strong>——TOOLS 字典 + gate 函数
3. <strong>Interruption</strong>——SIGINT handler + save_ckpt
4. <strong>Checkpointing</strong>——pickle 每步存
5. <strong>Error Recovery</strong>——retry with backoff
6. <strong>Auto-Compact</strong>——context 阈值触发 summarize

<strong>这就是 Claude Code 早期原型的样子</strong>——然后才一点点加 subagent、hooks、permissions、IDE 集成等周边能力。

我建议团队里每个做 agent 的同学，<strong>自己亲手写一遍这 80 行</strong>。写完之后，对 harness 的认知会从"听起来很复杂"变成"原来就这点东西"。

实战路径：
1. 第一周：抄这 80 行，跑通一个简单 agent task（如"读 10 个文件、生成 README"）
2. 第二周：加 3 个真实业务 tool（如读 CRM 客户、查订单、发邮件）
3. 第三周：测试 long-running——故意 Ctrl+C，看能不能 resume
4. 第四周：根据踩到的坑，加上失败模式 4 的所有防护

四周后你就有了一个 production-ready 的 harness。

## 第八部分：4 个设计权衡（2 分钟）

【滚动到 Design Trade-offs】

最后讲 harness 设计的 4 个关键权衡。<strong>没有标准答案，只有匹配场景的选择</strong>。

### 权衡 1：自主 vs 可控

光谱两端：
- Devin / Cognition——全自主，跑完才给你看
- Cline / Aider——每步都要用户 approval
- Claude Code / Cursor——中间路线，默认信任，关键 action 才停下来

权衡：自主→速度快但容易跑偏；可控→慢但用户安心。

### 权衡 2：多 Agent vs 单 Agent + Subagent

Cognition 的 "Don't Build Multi-Agents" 主张：多 agent 间共享 context 太难，信息会衰减。

Anthropic 的 Claude Code Subagents 走的是另一条路——single agent 主导 + 临时 subagent 隔离上下文。

经验法则：
- 多 agent 适合"并行可分解任务"（research、benchmark、批量翻译）
- 单 agent + subagent 适合"复杂连续任务"（重构、调试、长程开发）

### 权衡 3：动态选 tool vs 固定 tool set

- 固定 tool set——可预测、可缓存，但场景切换时 tool 选错率高
- 动态 gating——每个场景只暴露相关 tool，模型选择负担小，但需要"场景识别"逻辑

Claude Code 走动态路线（skills 机制），Aider 走固定路线（diff / edit / shell 三件套）。

### 权衡 4：同步 vs 异步执行

- 同步——用户对面坐着等，体验直接（Claude Code、Cursor、Aider、Cline）
- 异步——用户提交后离开，半小时回来看结果（Devin、Codex Web）

异步适合 long-running > 30 分钟的任务，需要更强的 checkpoint + 通知机制。同步 harness 简单；异步 harness 是分布式系统问题。

## 备注：常见疑问

<strong>Q: 我们已经在用 Claude Code / Cursor 了，还需要自己造 harness 吗？</strong>
A: 工程师"用"是一回事，"生产环境的 agent"是另一回事。Claude Code / Cursor 是给开发者用的工具，<strong>你的产品里给最终用户的 agent，依然要自己造 harness</strong>。但你可以借鉴 Claude Code 的设计思路——6 大组件、tool gating、checkpoint。

<strong>Q: 80 行 harness 看起来太简单，生产真能用吗？</strong>
A: 能用，但要加监控、告警、成本控制、并发管理。基础架构在那 80 行里。生产级 harness 大约 500-1500 行，主要扩展是：① 多用户隔离 ② 持久化到数据库（不是 pickle） ③ Trace 上报到 Langfuse / 自建 ④ 成本预算系统 ⑤ 失败告警。

<strong>Q: 多 agent 真的没用吗？</strong>
A: Cognition 的观点不是"多 agent 没用"，而是"<strong>多 agent 不是默认答案</strong>"。能用 single + subagent 就别上多 agent。真正适合多 agent 的场景：① 任务可并行分解（如 benchmark） ② Agent 间通信简单（如 producer / consumer） ③ 不需要共享中间状态。复杂连续任务用多 agent 通常更糟。

<strong>Q: Context auto-compact 会丢信息，那 long-running 任务怎么办？</strong>
A: ① 关键信息（system prompt、用户原始任务、约束、tech-debt）永远保留 ② compact 后明确告诉模型"前文已压缩" ③ 重要决策点同时记到结构化 state（不只在 messages 里） ④ 接受"长任务 = 信息会损失"这个事实，设计任务时尽量让每个 checkpoint 是"自洽"的。

<strong>Q: Harness vs Agent Framework（LangGraph / AutoGen）什么关系？</strong>
A: Framework 是别人的 harness。LangGraph 把循环、状态、tool 封装成 DSL。<strong>但 framework 的抽象不一定匹配你的业务</strong>——比如它的 checkpoint 粒度、interruption 处理可能不符合你的需求。建议：原型阶段用 framework 快速迭代；规模化阶段评估是否要自己造（通常是需要更精细控制时）。

<strong>Q: Tool gating 需要"场景识别"，这一步怎么做？</strong>
A: 三种思路：① 静态规则——按文件路径、命令前缀判断（如 *.test.ts → 测试场景） ② LLM 路由——让一个轻量模型先判断场景再分发 tool set ③ 用户显式切换——Claude Code 的 plan/edit 模式就是用户主动切。我们项目里推荐前两种结合。
