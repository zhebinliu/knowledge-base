LESSON 14 / 23

EVALS &
OBSERVABILITY

AI应用的评估与可观测性
上一讲我们说"多Agent调试爆炸"。但单Agent也有同样的问题—— AI系统在生产里像个黑盒。它好不好你说不清，它出问题你查不到。 今天讲清楚AI工程化最关键、最被忽视的基础设施。
CHAPTER · 01
SOFTWARE vs AI
为什么传统测试方法对AI完全失效
做过软件工程的同学都熟悉单元测试、集成测试、CI/CD。 但这些方法对AI应用基本失效——AI系统的不确定性、主观性、动态性是全新的工程挑战。
维度
传统软件
AI 应用
输出确定性
同输入 → 同输出。可以做严格断言：assertEqual(result, expected)
同输入 → 每次输出都不一样。assertEqual完全无效，需要语义评估
正确性判断
客观、二元——对/错明确
主观、连续——"好/中/差"。同一个回答不同的人会有不同评价
回归测试
改一行代码，跑一遍测试集，红了就修
改一句Prompt，输出全变了——很难判断"变好还是变差"
线上故障定位
看异常堆栈、日志、APM——路径明确
没有异常，但输出"不太对"——不知道哪步出了问题
性能波动
代码不变，性能基本稳定
同一段代码，不同时段质量飘忽（模型升级/降级、负载变化）
"修复"成本
改代码、加单测、合并PR——过程标准
改Prompt治标，改不动模型本身——很多问题"无解"，只能workaround

核心结论：传统软件的"测试驱动开发（TDD）"思路对AI系统不工作了。 AI系统需要新的工程范式——评估驱动开发（EDD, Eval-Driven Development）。 没有评估体系的AI系统，等于在黑暗中开车。

CHAPTER · 02
THE EVAL PYRAMID
评估的4个层次：从单元到生产
就像传统软件有"单元-集成-端到端"的测试金字塔，AI应用也有自己的评估金字塔。 4层从基础到高级，覆盖范围越来越大，频率越来越低。点击查看详情。
L1 · 单元评估
UNIT EVAL
单个组件的输出质量。如：一个Prompt的回答是否符合预期、一个工具调用是否正确
L2 · 链路评估
CHAIN EVAL
完整任务流程的输出质量。如：RAG的端到端答案、Agent完成任务的最终结果
L3 · 用户评估
USER EVAL
真实用户的反馈和满意度。如：点赞/踩、对话评分、用户行为指标（任务完成率、回访率）
L4 · 业务评估
BUSINESS EVAL
AI对业务指标的影响。如：客单价提升、转化率变化、人均处理量、成本节省

CHAPTER · 04
PRODUCTION TRACING
生产环境的全链路追踪
评估集解决"上线前"，Trace解决"上线后"。每一次AI调用都要可追溯—— 用户问了什么、检索了哪些文档、调用了哪些工具、模型怎么"想"的、最后回答了什么。
TRACE_ID: 2025-04-19-T14:32:08-a3f5b921
USER: sales_lily STATUS: DEGRADED TOTAL: 4847ms COST: $0.043
[14:32:08.012] ▸ USER_INPUT: "客户A本月的销售情况怎么样，需要我跟进吗？"
└─ [14:32:08.156] AGENT.think: "需要查询客户A本月业绩"144ms
└─ [14:32:08.892] TOOL.search_customer({name:"客户A"})736ms
└─ [14:32:08.890] RESULT: {customer_id: "CUST-2024-001", ...}
└─ [14:32:09.456] TOOL.get_order_history({...})564ms
└─ [14:32:09.450] RESULT: {orders: [...], total: 975000}
└─ [14:32:10.234] TOOL.get_customer_health({...})778ms
└─ [14:32:13.012] ERROR: TimeoutError - health_score_service degraded2778ms
└─ [14:32:13.234] AGENT.fallback: "继续基于已有数据回答"222ms
└─ [14:32:14.859] AGENT.final_answer: "客户A本月签了3个订单，共97.5万..."1625ms
▼ AUTOMATED ANALYSIS

问题识别：health_score_service超时（2.8秒），导致整体响应时间从预期1.5秒延长到4.8秒。 Agent正确处理了超时——使用fallback回答，没让用户看到错误。 但 "是否需要跟进"的判断少了关键的健康度信息，回答质量打折扣。

修复建议：① 检查health_score_service性能 ② 给该工具设置200ms超时阈值 ③ 在Agent的Prompt中明确"健康度数据缺失时如何降级回答"

这个Trace告诉我们三件事：① 哪一步慢了——health_score_service ② 哪一步出错了——超时降级 ③ 用户的体验如何——回答完成但不完整。 没有这种Trace，所有这些都是黑盒。你只会收到客户的反馈——"AI助手怎么有点慢，回答也不太完整"，然后两眼一抹黑。

CHAPTER · 05
FOUR METRICS
必须监控的4个核心指标
从无数指标里挑出最关键的4个。这4个看清楚了，AI系统的健康度基本能掌握。 涵盖质量、性能、成本、用户体验四个维度。
METRIC · 01 · 质量
通过率
EVAL PASS RATE

每次代码或Prompt变更后，评估集的通过率。这是AI版本的"绿灯/红灯"—— 通过率掉了就说明这次改坏了。

通过率 = 通过用例数 / 总用例数
按维度细分：分类 / 抽取 / 推理 / 边界
▸ 健康线：>90%；告警线：<85%
METRIC · 02 · 性能
P95延迟
LATENCY P95

95%的请求在多长时间内完成。AI应用比传统应用慢（Token逐个生成）， 但用户对延迟敏感。监控P95比平均值更能反映真实体验。

P95 = 95%请求的响应时间分位数
按链路细分：检索 / 生成 / 工具调用
▸ 对话场景：<3秒；批处理：<30秒
METRIC · 03 · 成本
单次请求成本
COST PER REQUEST

每次AI调用消耗的Token成本。这是AI应用最容易失控的部分—— 一个prompt写得不好、一个Agent循环了十几次，成本可能翻10倍。

成本 = (input_tokens × $/1k) + (output_tokens × $/1k)
监控分位：P50 / P95 / P99
▸ 设预算：超过P99 2倍触发告警
METRIC · 04 · 用户
满意率
USER SATISFACTION

用户对回答的正/负反馈率。最直接的真实质量信号—— 所有自动化评估都比不上真实用户用脚投票。

点赞率 = 👍 / (👍+👎)
细分：按用户角色、按问题类型、按时段
▸ 健康线：>80%；持续观察<70%要重点优化
CHAPTER · 06
THE TOOLCHAIN
2025年AI可观测性工具栈
工具是基础，不是花架子。下面是2025年生产环境最常用的9个工具，按职能分3类。
CATEGORY · 01 · TRACING
LangSmith
by LangChain

最成熟的LLM Trace平台。每次调用全链路追溯，支持评估集管理、版本对比、Prompt调试。 推荐作为第一选择。

CATEGORY · 01 · TRACING
Langfuse
OPEN SOURCE

开源版本的LangSmith。支持私有化部署，对企业内部AI应用非常友好。 国内项目用得多。

CATEGORY · 01 · TRACING
Phoenix / Arize
OBSERVABILITY

专注AI模型监控和漂移检测的平台。对生产级AI系统更专业， 能识别质量漂移、数据分布变化。

CATEGORY · 02 · EVAL
Promptfoo
CLI / OPEN SOURCE

命令行的Prompt评估工具。YAML定义测试用例，CI集成方便。 适合工程师团队快速搭建评估pipeline。

CATEGORY · 02 · EVAL
DeepEval
PYTHON / OPEN SOURCE

像pytest的AI评估框架。提供14+评估指标（hallucination、relevance、faithfulness等）， 可以直接接入CI。

CATEGORY · 02 · EVAL
RAGAS
RAG专用

专门评估RAG系统的工具集。包含上下文召回率、答案忠实度等RAG特有指标。 做RAG必备。

CATEGORY · 03 · USER FB
Helicone
LLM PROXY

作为LLM API代理，透明记录所有调用。无侵入式接入， 适合后接入观测能力的项目。

CATEGORY · 03 · USER FB
PostHog
PRODUCT ANALYTICS

产品分析工具。记录用户与AI的交互行为——点赞、跳出、重新提问等， 形成真实用户体验信号。

CATEGORY · 03 · USER FB
自建数据库
PostgreSQL + 看板

中国项目里最实用的方案。把上面所有信号导入自己的数据库， 用Metabase / Superset做看板，完全可控。

实战配置建议：原型阶段用 Langfuse（开源、私有化）+ DeepEval（CI集成）；生产阶段加 PostHog（用户行为）+ 自建PostgreSQL看板。 不要等到出问题才装观测——它必须从第一天就在。

RECAP

核心收束 · 工程化AI的"基础设施"

第三部分（工程落地）讲到这里——Prompt、RAG、Function Calling、Agent、多Agent—— 都是"功能"。今天讲的是支撑这些功能可靠运行的"基础设施"。

"没有评估，AI开发就是在凭感觉调；
没有可观测性，生产环境就是在凭运气跑。"

评估金字塔——单元、链路、用户、业务四层，覆盖从开发到生产的全流程。 4个核心指标——通过率、P95延迟、单次成本、满意率——简单但有效。 9个工具覆盖Tracing、Eval、用户反馈三大类需求。

对我们项目最实用的建议： ① 每个上线的AI功能必须有评估集（最少30个用例）； ② 接入Langfuse或自建Trace，每次调用可追溯； ③ 在产品里加👍👎按钮，把真实用户反馈沉淀下来； ④ 周会review四个核心指标的趋势，不出问题不代表健康。
