RAG：让模型用上你的私有知识
大模型博学，但它不知道你公司的事。它不知道你们去年和某客户签的合同条款，不知道你们产品的最新功能列表，不知道你们的SOP。 RAG解决的就是这个问题——让模型在回答前，先检索你的知识库。 这是企业AI落地最重要的一项技术，没有之一。
CHAPTER · 01
THE PROBLEM
大模型最大的"短板"
你训练时再聪明的模型，也不知道你公司私有的、动态的、最新的信息。 这不是模型不够强，是它根本没见过这些数据。直接问会发生什么？
USER QUESTION 我们公司的SaaS项目延期违约金条款是怎么规定的？
▼ 直接问大模型（无RAG）
根据一般行业惯例，SaaS项目延期违约金通常按合同金额的0.5%-1%每日计算，上限一般是合同金额的20%-30%。具体条款建议查阅贵司具体合同...
× 这是泛泛而谈，不是你公司的真实规定
× 数字看起来精确，但完全是编的
× 无法追溯到具体合同/版本/章节
× 员工拿这个回答给客户，会出大事
▼ 接入RAG之后
根据《SaaS服务合同模板v3.2》第8.3条，项目延期违约金规定为：合同总金额的0.3%/日，上限不超过合同总额的15%。同时第8.4条规定，因不可抗力或客户原因造成的延期不计入此条款...
✓ 引用真实文档（合同模板v3.2）
✓ 具体到章节（第8.3条、第8.4条）
✓ 含有免责条款等关键细节
✓ 可以直接用，可以追溯、可以验证

关键认知：大模型的核心能力是"理解+生成"，不是"记忆事实"。 让它记住公司私有信息，要么靠训练（贵且慢），要么靠在调用时把相关信息送进它的上下文。 RAG就是后者的工程化方案—— 用"检索"把私有知识找出来，"生成"时让模型基于这些知识回答。

CHAPTER · 02
THE ARCHITECTURE
RAG的两条流水线
RAG系统有两条独立的流水线——离线索引（Indexing）把你的文档变成可检索的向量库； 在线检索（Retrieval）在用户提问时实时找出相关内容。点击任一步骤查看细节。
▼ INDEXING · 离线索引（一次性）
01
文档加载
PDF / Word / Excel / 网页 / 数据库 → 统一文本
02
文档切分（Chunking）
大文档切成小块（Chunks），每块500-1000字
03
向量化（Embedding）
每个Chunk → 高维向量（如1024维）
04
存入向量数据库
Milvus / Qdrant / PgVector / Chroma
▼ RETRIEVAL · 在线检索（每次提问）
05
问题向量化
用户问题 → 同一个Embedding模型 → 向量
06
向量检索（Top-K）
在向量库里找最相似的K个Chunks
07
重排序（Rerank）
用更精细的模型再筛选一遍 (可选但推荐)
08
拼接Prompt + 大模型生成
把检索结果作为上下文喂给LLM，生成回答
STEP · 01 · INDEXING

文档加载（Document Loading）

把企业里各种格式的文档统一成纯文本。这一步看似简单，但是整个系统质量的源头。

常见挑战：PDF里的表格、图片、公式怎么处理？Word里的修订记录、批注要不要保留？Excel里行列结构怎么转线性文本？扫描件需要先OCR。

实践经验：不同类型文档需要不同的Loader。我们项目里通常用 unstructured 库 + 针对性后处理。一个普遍的坑——表格被切成无意义的文本碎片，必须用专门的Table Extractor。

关键产物：结构化的纯文本 + 元数据（来源文件、页码、章节、修改时间）。元数据后面在引用追溯时至关重要。

CHAPTER · 03
SEE IT IN ACTION
看一次完整检索：从问题到答案
切换不同问题，看RAG是怎么从知识库里捞出相关Chunks，再生成最终答案的。
▼ 选择一个问题（模拟用户输入）
▼ 检索到的Top-5 Chunks（按相似度排序）
▼ LLM基于Chunks生成的最终答案

观察要点： ① 模型只用了检索到的Chunks内容，不会瞎编 ； ② 答案中明确标注来源，可追溯； ③ 不相关的Chunks（左边低分的）被自动过滤； ④ 每次提问的Chunks都不一样，检索是动态的。

CHAPTER · 04
CHUNKING STRATEGIES
切分策略：决定RAG质量上限
切分（Chunking）是RAG里最容易被低估、但对效果影响最大的环节。 切错了，再好的Embedding和LLM都救不回来。下面是三种主流策略的对比。
STRATEGY · 01

固定长度切分

按字数切（每500字一段，重叠50字）。最简单的策略， 很多入门教程用这种。

优点：实现简单、速度快、对文档格式无要求
缺点：语义会被生硬切断（比如一个表格被切成两半），导致检索准确率低
STRATEGY · 02

结构化切分

按文档结构切（按段落、章节、标题）。大部分生产系统的首选， 适合有明确层级的文档。

优点：保留语义完整性、引用粒度清晰（"第8章第3节"）
缺点：需要文档有良好结构，对扫描件/无格式文本不友好
STRATEGY · 03

语义切分

用LLM或Embedding模型识别"语义边界"，在语义自然结束的地方切。 最先进的策略。

优点：语义完整性最好、对表格/列表友好
缺点：计算成本高（每次切分要调LLM）、慢

实战建议： ① CRM场景下，结构化切分 + 元数据增强是性价比最高的方案； ② 合同/法律类文档必须按"条/款"切，每个条款一个Chunk； ③ 表格单独抽出来，不要和正文混在一起； ④ 每个Chunk都要带来源元数据（文件名、页码、章节、版本号），这是引用追溯的基础。

CHAPTER · 05
PRODUCTION PITFALLS
生产环境的6个坑
RAG从demo到生产，差距是巨大的。下面这些坑我们项目里都踩过，每一个都会让效果断崖式下降。
坑名 典型症状 修正方法
表格被切碎 用户问"产品功能对比表"，检索回来一堆零散的单元格碎片，LLM拼不回原表 用 Markdown表格 或 HTML 完整保存表格作为单个Chunk，不参与文本切分
问题表达和文档不一致 用户问"延期罚款"，文档里写"违约金"，向量相似度低，检索失败 引入 查询改写（Query Rewriting） + 同义词扩展，必要时用LLM先改写问题
多文档版本冲突 v2.0合同模板和v3.0模板都被检索到，LLM不知道用哪个，给出错误回答 元数据加 版本号 + 时间戳，检索时按时间倒序加权，老版本降权或过滤
答非所问 用户问"金额是多少"，模型回了一段相关但不直接的内容 加 Reranker（重排模型）。bge-reranker-v2-m3等模型能把Top-K进一步精筛

CHAPTER · 06
RAG vs FINE-TUNING
什么时候用RAG，什么时候微调？
这是企业AI落地最常被问的决策题。简短答案：99%的场景应该选RAG。 下面是更细的判断框架。

用 RAG · 适合大多数场景

知识动态变化：合同条款、产品文档、政策法规——这些会更新。

需要可追溯：客服、法务、医疗——必须能告诉用户"这个答案来自哪里"。

规模较大：知识库超过10万字。

成本敏感：上线快、改起来快、不依赖训练资源。
典型案例： 客服问答 / 合同助手 / 知识库搜索 / 销售支持工具

用 微调 · 仅在特定场景

需要风格/格式定制：让模型固定用某种话术回答。

数据高度专业：医疗影像术语、法律条文风格——通用模型理解有偏差。

有大量高质量训练数据：至少几千条精心标注的范例。

知识相对固定：训完几个月不会过期。
典型案例： 行业领域模型 / 写作风格定制 / 专业术语转换

实战经验：RAG和微调不是非此即彼。 高级方案是组合——微调一个理解你领域语言的小模型作为Embedding， 配合RAG用大模型做生成。这样既有领域适应性，又有动态知识更新能力。 但90%的项目，先把RAG做扎实再考虑微调。

RECAP

核心收束

RAG的本质很简单：把知识库的相关内容塞进Prompt再让模型回答。 但要做好极难——切分、嵌入、检索、重排、Prompt拼接，每一环都有讲究。

做RAG的关键是认识到——它不是一个模型问题，是一个工程系统问题。 最好的Embedding也救不了切错的文档。最强的LLM也消化不了不相关的检索结果。 系统的瓶颈往往在最弱的环节。

对我们做CRM实施的特别意义——知识库工程化是2025年的主战场。 每一个企业客户都需要把内部知识AI化。谁能把RAG做得稳、准、可追溯、可治理， 谁就能拿下这块市场。

TAKEAWAY · 01
RAG是企业AI的标配。99%的场景比微调更合适，且能解决"私有知识"和"可追溯"两大刚需。
TAKEAWAY · 02
切分质量决定上限。文档处理和Chunking是最容易被低估、但回报最高的优化点。
TAKEAWAY · 03
Reranker能显著提质。在向量检索之后加一层精排，是从demo到生产的关键一跃。
