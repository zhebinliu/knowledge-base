LESSON 17 / 23

CLAUDE
CODE

底层原理架构拆解 · 工程级 Agent 系统怎么搭
上一讲 Harness Engineering 把"AI 外壳"这个概念讲透了。本讲拆开 2025 年最值得抄的工程 AI 产品—— Claude Code。它把 Context Engineering 和 Harness Engineering 变成了一个真实跑通 long-horizon coding 的开箱即用产品。 不教用法，教架构。
CHAPTER · 01
PRODUCT POSITIONING
Claude Code 究竟是什么 · 跟 Cursor / Devin / Aider 的差异
Claude Code 不是 chat、不是 IDE 插件、不是替代 Cursor 的产品。它是一个 agent harness—— 目标是 long-horizon coding tasks（>1 小时）：多文件改、跑测试、生成 commit、提 PR。 Boris Cherny 在 2025 ai.engineer 上反复强调一句话："Claude Code 是一个 CLI，因为 CLI 是所有工程师工作的最小公约数。"
FORM · CLI / TUI
Claude Code
TERMINAL-NATIVE

定位：long-horizon agent harness。

形态：终端为主、IDE 插件辅。git-native、可远程、可容器、可 CI。

主场：自动重构、迁移、补测试、提 PR。一次跑 30 分钟–几小时不奇怪。

FORM · IDE
Cursor
VS CODE FORK

定位：AI-first IDE，强项是"边写边补全"。

形态：编辑器为中心，Composer 是后来加的 agent 模式。

更适合 in-IDE 短任务、对话式重构、跟开发者并坐。

FORM · CLOUD
Devin
SAAS AGENT

定位：远程"AI 同事"，开会给任务、等结果。

形态：云端 sandbox，不在你电脑上跑。

黑盒程度高、过程不可干预；适合委派型任务，不适合协作式开发。

FORM · CLI / OSS
Aider
OPEN SOURCE

定位：开源 pair-programmer CLI，先驱式产品。

形态：单文件 diff 为主，需要更多人工 review。

Claude Code 在它的轨道上做了"工程级强化"——sandbox、worktree、hooks、skills。

一个关键认知：把 Claude Code 当 Cursor 的"替代品"看，会错过它真正的价值。Cursor 在编辑器里加 AI，Claude Code 是围绕 AI 重新搭一套工程外壳——CLI、文件系统、git、shell、hooks、subagent，全都是这个外壳的组件。

CHAPTER · 03
THREE DESIGN CALLS
3 个让 Claude Code "长得不一样"的设计取舍
Anthropic 内部反复打磨的几个反直觉决策。Boris Cherny、Cat Wu、Erik Schluntz 在 ai.engineer 2025、Latent Space podcast 上都讲过—— 正是这些取舍把 Claude Code 跟 Cursor / Devin 区分开。
决策 01 · 为什么是 CLI 而不是 IDE-native？
CLI-FIRST vs IDE-FIRST
IDE-FIRST · CURSOR 路线
编辑器中心。AI 是"编辑器里的助手"，强项是 inline 补全和文件级 diff。
代价：绑死单一 IDE、远程难用、CI / 服务器场景半残。
VS
CLI-FIRST · CLAUDE CODE 路线
终端中心。CLI 是工程师的最小公约数——本地 / SSH / Docker / CI / GitHub Action 都能跑同一个 binary。IDE 插件只是"包了一层 UI 的 CLI"。
决策 02 · 为什么是 single-agent + subagents 而不是 multi-agent？
SINGLE AGENT + SUBAGENTS vs MULTI-AGENT
MULTI-AGENT · AUTOGEN / CREWAI 路线
多个长期存在的 agent 角色（Planner / Coder / Tester）互相对话。
代价：context 漂移、消息爆炸、谁拍板说不清；Anthropic blog《Building effective agents》明确指出 "don't do this unless you must"。
VS
SINGLE + SUBAGENTS · CLAUDE CODE 路线
主 agent 一条命令链，需要隔离上下文的子任务时 spawn 一个 subagent，跑完把结论返回就关。Subagent 不污染主线 context，主线掌握全局。
决策 03 · 为什么是 Hooks 而不是 Plugins？
SHELL HOOKS vs PLUGIN SDK
PLUGIN · VS CODE 路线
插件是语言绑定 SDK（TypeScript / Python），有生命周期、有 API、有上架审核。
代价：扩展门槛高、生态分裂、跟 shell 工具链脱节。
VS
HOOK · CLAUDE CODE 路线
扩展点 = 一行 shell 命令。PreToolUse / PostToolUse / Stop / Notification 四个事件点，挂任何能在终端跑的东西——bash、jq、pytest、curl、Slack webhook。组合即扩展，无 SDK。

CHAPTER · 06
THE SKILLS SYSTEM
按需触发 vs 全量注册 · 2025 年的 Context 节流大招
Plugin 模式的痼疾——你装了 50 个插件，system prompt 里全得列一遍，没用上的也在烧 token。 Skills（2025 后半推出）反过来：按 trigger 语义匹配再加载。Skill 是一个文件夹 + frontmatter + 文档，用到时才进 context。
STEP · 01
用户消息
"帮我把这个 .docx 改一下标题字体"
▶
STEP · 02 · MATCH
Skill 路由器
扫描已注册 skills 的 description frontmatter，匹配"docx / word document"。
▶
STEP · 03 · LOAD
加载 docx skill
只在这一回合把 skill 的 SKILL.md + 示例 + 工具脚本注入 context。其他 49 个 skill 一动不动。
▸ PLUGIN MODE · 全量注册

50 个插件 × 平均 800 token = system prompt 多 40k token。

每次对话都付费、都干扰注意力。

新加一个插件，老的 prompt 都要重测。

▸ SKILL MODE · 按需加载

50 个 skill × 60 token (description) = system prompt 多 3k token。

命中的 skill 才注入完整 SKILL.md。

用到 docx 才加载 docx，不影响其他场景。

Skill 的本质是"延迟加载 + 语义路由"。这背后是 Anthropic 内部的一个核心信念—— context 是稀缺资源。1M context 也得这么用，否则 cost、延迟、attention drift 一样会出来咬你。这套思路完全可以搬到自家 RAG / Agent 产品里：能力清单按 description 提供，实际 prompt 用到时再 inject 全文。

CHAPTER · 07
TOOL DESIGN DETAILS
Read / Edit / Write 为何要拆 · Bash 为何要隔离 · 输出为何要截断
工具系统是 Claude Code 反复打磨最多的一层。看似冗余的拆分，背后都是踩过的坑。
DESIGN · 01
Read / Edit / Write 三件套为什么要拆开？

Read 强制带行号——为了让 Edit 能精确定位。

Edit 拒绝在没 Read 过的文件上动手——防止"凭印象改"。

Write 是覆盖式的，专用于新建——避免误用 Edit 把整个文件干掉。

三个工具互相设了前置条件，把"一边读一边猜一边改"的常见错误模式从入口就堵掉。

DESIGN · 02
Bash 工具的子进程 / 超时 / background 三件套

每次 Bash() 调用都开新子进程——不继承 shell 状态、不残留 env 变量。

默认 2 分钟超时，最长 10 分钟。避免 LLM 启动一个 server 然后死等。

run_in_background=true 让长任务（dev server / test runner）后台跑，不占当前 tool turn。

DESIGN · 03
为什么 Glob / Grep 要做成独立工具？

Bash 里写 find . -name '*.ts' | xargs grep ... LLM 一半会漏 quote、漏 escape、误 alternation 顺序。

Glob / Grep 是有 schema 的 API——pattern、path、include、exclude、line_count 全是参数。LLM 不用记 find / grep 的 flag 语法。

Anthropic 内部 eval 显示：同样的 codebase 检索任务，专用工具成功率比 Bash 通配高一个数量级。

DESIGN · 04
Tool 输出为什么要截断？

读一个 30k 行的 log、grep 一个 5MB 的源码 dump——一次工具调用就能填满 context。

Claude Code 在 tool 层做智能截断：长输出留头尾 + 提示"中间省略 N 行"，让 LLM 主动决定要不要二次定位。

这是把 context 预算从"事后压缩"提前到"事前裁剪"——auto-compact 是兜底，工具层截断是第一道闸。

CHAPTER · 08 · DEMO 04
HOOKS AS PROGRAMMABLE SHELL
演示四 · 可编程的工程外壳
四个事件点（PreToolUse / PostToolUse / Stop / Notification）各挂一行 shell，看一次任务的时间线—— hook 怎么拦截 rm -rf、怎么自动跑 lint、怎么任务完成时发通知。
▸ PRE TOOL USE
每次工具被调用前执行。返回非零退出码可拦截整个工具调用——这是 Claude Code 防御危险命令的第一道闸。

        

        

          
▸ POST TOOL USE

          
每次工具调用后执行。不可拦截但能触发副作用——auto format、自动跑测试、刷新 cache 都靠它。

          
          

        

        

          
▸ STOP

          
Agent 决定结束当前回合时执行。常用于收尾检查——lint 必须过、未提交变更必须提示、覆盖率必须达标。

          
          

        

        

          
▸ NOTIFICATION

          
需要用户注意时触发（任务完成、等待输入）——可以推 Slack、调系统 TTS、亮屏唤醒。

CHAPTER · 09

    
PORTING THE LESSONS

    
从 Claude Code 学到什么能搬到自家产品

    

      不是让大家把 Claude Code 抄一遍——是把它背后的工程思路抠出来，搬到自家 AI 产品里。
      我们项目的知识库 Agent、需求提取、调研问卷生成，每一处都能找到对照。
    


    

      

        
TAKEAWAY · 01

        
Subagent 隔离的工程价值

        
主线 agent 跑一个长任务，需要做"批量打分 100 条评论"这种污染 context 的子任务时，spawn subagent，跑完只回结论。我们项目的「文档转写脱敏」就是典型 subagent 场景——主线 Agent 不该看到全文，只该拿到脱敏后的版本。

      

      

        
TAKEAWAY · 02

        
Hook 等价的"扩展点设计"

        
不要给自家 Agent 设计 SDK / plugin 体系。每个关键节点暴露一个 webhook 或 shell 钩子就够——客户能自己挂审计、加敏感词、连 OA。这是 SaaS 厂商最被低估的扩展机制。

      

      

        
TAKEAWAY · 03

        
Skill 按需加载

        
不要在 system prompt 里塞 50 条"如果用户问 X 就 Y"。把每个能力做成 SKILL.md，路由器只看 description。我们项目的「调研问卷生成」、「合同条款抽取」都该是 skill，而不是塞进主 prompt。

      

      

        
TAKEAWAY · 04

        
终端 first 但 UI 辅助的混合范式

        
SaaS 产品大多 UI-only。但给 power user 留一个 CLI / API 入口，让他们能批处理、能脚本化、能进 CI。Claude Code 教我们：UI 是包装，CLI 是骨架。我们项目应该提供一个 kbctl，让客户能脚本化跑 KB 同步、批量脱敏、状态查询。

      

      

        
TAKEAWAY · 05

        
工具层就开始裁剪 context

        
不要等 auto-compact 才管 context——在工具输出层就做截断。我们项目的 RAG 检索如果一次返回 30 段，要在工具层只回前 5 段 + 提示"还有 25 段，需要看请指定"。

      

      

        
TAKEAWAY · 06

        
单一长循环 + 持久 transcript

        
不要写"对话历史用 8 条窗口、超了就丢"的逻辑。持久化整个 JSONL transcript，让 agent 可以 resume、可以回放、可以审计。我们项目的 Agent 调用都应该有可恢复的 session ID，断网重启不丢。
