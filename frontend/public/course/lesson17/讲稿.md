## 开场（2分钟）

PART 3 工程落地的最后一讲，我先抛一个真实案例。

去年我接触过一个团队，做的事情很典型——他们想给自家代码库做一件事：**自动重构一个老模块、跑通测试、生成 commit、提 PR**。听起来是 long-horizon 任务，但每一步都很明确。

他们用什么做？Cursor。买的 Pro 账号，所有工程师都用。

结果呢？**折腾了 3 个月，做不出来**。

不是 Cursor 不好用——日常 inline 补全、单文件重构、写个函数，Cursor 体验拉满。但他们这个任务长。20 分钟内的事 Cursor 撑得住，超过 30 分钟之后开始飘——context 越来越满、tool 调用越来越乱、改了 A 文件忘记 B 文件、跑了 test 不会处理失败、要 commit 又不知道写什么。

他们换了几种思路：

- 把任务拆小让 Cursor 一段一段做——人盯着串起来，**效率反而比手写还低**。
- 上 Devin——黑盒一个工程，跑出来的结果跟需求漂得厉害，**没法干预过程**。
- 试 Aider——可以，但每一步都要确认，**人变成了 Aider 的助手**。

最后切到 Claude Code。**第一周就跑通了完整流程**——重构、测试、commit、PR 一气呵成。中间还能 Ctrl+C 干预、能 /resume 继续、能挂 hook 自动跑 lint。

**这事让我反思了很久**。模型都一样——他们之前也用的是 Claude。区别在哪？区别在 **harness**。

Claude Code 把 Context Engineering 和 Harness Engineering 这两个我们前两讲讲过的概念——**变成了一个真实可用、跑通了 long-horizon coding 的开箱即用产品**。

今天这一讲不教大家怎么用 Claude Code（用法网上一大堆），讲的是它**底层架构是怎么搭的**。8 层、3 个关键决策、4 件套（Skills / Tools / Auto-compact / Hooks），把这套思路看明白，你就有了搭工程级 Agent 系统的骨架。

打开 demo.html，对着拆。

## 第一部分：Claude Code 是什么 · 跟同类产品的差异（4分钟）

【看第一节四方对比】

我先把 Claude Code 这个产品"定位"讲清楚——它不是 Cursor 的替代品，也不是 chat 的延伸。**它是一个 agent harness**，目标是 long-horizon coding tasks（一次跑超过 1 小时不是奇怪事）。

Boris Cherny 是 Claude Code 的 PM Lead。他在 2025 ai.engineer 大会上反复强调一句话——**"CLI 是工程师工作的最小公约数"**。本地开发是 shell、远程 SSH 是 shell、Docker 容器是 shell、CI / CD 是 shell、GitHub Action 是 shell。**只有 CLI 形态能在所有这些场景里跑同一个 binary**。

所以 Claude Code 选了 CLI 作为一等公民——VS Code / JetBrains 插件只是"包了一层 UI 的 CLI"，不是反过来。

我们对照 3 个同类产品：

### Cursor · IDE-FIRST 路线

Cursor 是一个 VS Code fork，把 AI 当成"编辑器里的助手"。强项是 inline 补全和 Composer 模式下的文件级 diff。

**适合什么**：边写代码边补全、对话式重构、跟开发者并坐写代码的场景。

**不适合什么**：脱离 IDE 的环境——SSH 上去改远程代码？不行。在 CI 里跑一段 AI 修复脚本？也不行。

### Devin · CLOUD AGENT 路线

Devin 是远程 SaaS——你给它一个任务，它在云端 sandbox 里跑，最后给你结果。

**适合什么**：完全委派、不需要中间干预的长任务。

**不适合什么**：要观察过程、要手动干预、要在自己的代码库里跑——黑盒程度太高了。

### Aider · OPEN SOURCE CLI 路线

Aider 是 Claude Code 之前的开源 pair-programmer CLI 标杆。轻量、git-friendly、能跑。

**不适合什么**：复杂工程任务——它没有 worktree 自动隔离、没有 hook、没有 skill、没有 subagent，**长任务容易飘**。

Claude Code 可以理解成 **"Aider 的工程化加强版"**——保留 CLI 一等公民、git-native 这两个核心基因，把企业场景缺的那一截全部补齐：sandbox、worktree、hooks、skills、subagents、auto-compact。

### 一个关键认知

【看下面绿色块】

**把 Claude Code 当 Cursor 的替代品看，会错过它真正的价值**。

Cursor 在编辑器里加 AI。
Claude Code 是**围绕 AI 重新搭一套工程外壳**——CLI、文件系统、git、shell、hooks、subagent，全都是这个外壳的组件。

这个分水岭很重要——你判断要不要 fork 这套思路到自家产品，先得看自己的产品形态是"在 UI 里加 AI"还是"围绕 AI 搭一套工作流"。两个路径不冲突，但工程取舍完全不同。

## 第二部分：8 层架构全景图（7分钟）

【看演示一 · 8 层架构】

接下来是本讲最核心的内容——**Claude Code 从外到内 8 层架构**。我点几个关键层讲，剩下的大家自己点开看。

### L8 · 用户接口层（User Interface）

最外层，接收人类指令、显示进展、转发到底层。

**关键模块**：CLI（cc 命令本体）、TUI（终端富交互界面）、VS Code / JetBrains 插件、slash command（/init /clear /compact /resume）、@file 引用。

**为什么重要**：这一层决定了 Claude Code 能在哪些场景里跑。CLI 形态意味着——本地、SSH、Docker、CI 都能跑同一个 binary。这是 Cursor 永远做不到的。

### L7 · 会话与权限层（Session & Permissions）

【点击 L7】

这一层管"agent 在本次会话里允许做什么"。

**关键模块**：`settings.json`（project + user 两级）、permission modes（default / acceptEdits / bypass / plan）、permission rules、4 个 hook 事件点。

**关键决策**：**权限和 hook 在同一层**——这个设计非常聪明。权限是"能不能做"，hook 是"做之前 / 之后做点什么"。两者都是**声明式的 shell 钩子**——可被 audit、可被 grep、可进 git。一个企业团队完全可以把 settings.json 提交进仓库，让所有人共享相同的权限和 hook 配置。

### L6 · Harness 主循环层（Agent Loop）

【点击 L6】

**整个 agent 的心脏**。

逻辑非常简单——把"用户消息 + 上下文 + tool 列表"喂给模型 → 拿到 response → 解析 tool_use → 执行工具 → 把结果塞回去 → 再调一次 → 直到模型决定 stop。

**关键决策**：单循环 + 并行 tool，**不做复杂状态机**。

Boris Cherny 在 Latent Space podcast 里说过一句让我印象很深的话——**"agent 越简单越好。复杂度要藏在 prompt 和 tool 里，不要藏在 loop 里。"**

什么意思？很多团队搭 agent 的时候喜欢搞 state machine——什么 planning state、execution state、verification state、reflection state。Claude Code 全没有。它就是一个 while loop——模型说继续就继续，模型说停就停。

**复杂度藏哪里？**——藏在 prompt（system prompt + CLAUDE.md + skill）和 tool（Read / Edit / Bash / Glob / Grep）里。模型自己会决定下一步做什么。这是个**信任模型**的设计。

### L5 · 上下文管理层（Context Engineering）

【点击 L5】

这一层就是上一讲（Lesson 15 / 16）讲的 Context Engineering 落地。

**关键模块**：
- **system prompt** 模板（约 2-3k token，描述 agent 的角色和工具）
- **CLAUDE.md** 项目长期记忆（每次自动注入）
- **Skills** 按需加载的能力包
- **Subagent** 上下文隔离的子任务
- **Auto-compact** 在 85% 阈值自动触发
- **@file** 显式文件引用

这 6 件套构成一条完整的**上下文流水线**——从"什么必须进 context"到"什么按需进"到"满了怎么办"。

我特别想强调 **subagent 这个概念**。Anthropic 在 2025 年的 blog《Building effective agents》里专门讲过——"single agent + subagent 优于 multi-agent"。

**Subagent 是什么**：主 agent 跑一个长任务，突然要做一个"会污染 context"的子任务（比如批量 review 100 个文件、批量打分 100 条评论），这时候 spawn 一个 subagent，跑完只返回**结论**，子任务的中间过程不进主线 context。

这个机制我们后面 Takeaway 那一节还会深入讲——它是 Claude Code 能跑长任务的**关键之一**。

### L4 · 工具系统（Tool System）

【点击 L4】

提供 agent 跟外部世界互动的所有能力。

**关键模块**：Read / Write / Edit / Bash / Glob / Grep / WebFetch / NotebookEdit 等内置工具、background mode、MCP server 接入、tool gating。

**关键决策**：**专用工具 > 通用 Bash**。这一点 demo 第二个演示会专门拆，先按下。

### L3 · 模型层（Model Layer）

【点击 L3】

调用 Anthropic API。

**关键模块**：Opus 4.6+ 作为主力 / Sonnet 4.5+ 作为快速副手、ANTHROPIC_MODEL 切换、1M context（Opus 4.5 起 GA）、cache_control 自动加在 system prompt、extended thinking。

**关键决策**：**Claude Code 是 prompt cache 的极致用户**——同一个 system prompt 不会重复付费，工具定义、CLAUDE.md 都进 cache。这让长对话的成本**不随 turn 数线性涨**。

我们项目要不要做 prompt cache？要。Anthropic API 默认提供 cache_control header，你只要在 system prompt 和大段 reference content 上加这个 header，**重复使用就走 cache 价（只要原价的 10%）**。我们的需求提取 prompt 现在 2k token，每次都付——加 cache 之后就只在第一次付，后续 10 倍便宜。

### L2 · 持久化层（Persistence）

【点击 L2】

把每次会话的完整 transcript 落盘。

**关键模块**：`~/.claude/sessions/` 下的 JSONL 文件、每个 tool call 单独一条 entry、checkpoint 在关键节点自动建、resume 从最后一条恢复。

**关键决策**：**JSONL 而不是数据库**——可以 grep、可以 jq、可以 diff、可以 commit。整个 session 是 **append-only event log**。

Erik Schluntz 在 Latent Space 上反复说过一句话——**"agent 要可回放"**。Claude Code 把整个 transcript 落成 JSONL 就是为了可回放：出问题了？把 JSONL 抓过来，jq 一下定位到出错那个 turn，看看模型当时看到的 context 是什么、调了哪个工具、返回了什么——**所有 debug 信息都在那个文件里**。

### L1 · 可观测 / 安全层（Observability & Safety）

【点击 L1】

agent 的**地基**。

**关键模块**：audit log、git worktree 自动隔离、Bash sandbox（macOS 用 sandbox-exec，Linux 用 seccomp）、PreToolUse hook 阻断危险命令。

**关键决策**：**worktree 是核心安全边界**。

什么意思？默认情况下，Claude Code 跑在 **git worktree 的副本**里，而不是直接在 main 上动手。Agent 改坏了？切回 main，世界依然完好。这是企业场景能落地的**底线条件**——你不可能让一个 agent 直接动生产分支。

### 端到端追踪：一个请求是怎么穿过这 8 层的

【点上面的"简单改文件"按钮】

看路径：**L8 UI 收命令 → L7 权限层放行 → L6 主循环喂模型 → L4 工具执行 Edit → L2 JSONL 落盘**。中间不需要的层不调。

【点"触发 rm 拦截"】

看路径：**L8 UI 收命令 → L7 权限层 + PreToolUse hook 协同拦截 → L1 安全层记录 → 不进主循环**。Agent 还没看到这个工具的执行，就已经被拦下了——这是 **defense in depth** 的体现。

### 这 8 层给我们的启示

**每一层都有清晰的职责边界**。这是 Claude Code 跟很多"用 chain 拼起来的 agent"最大的区别——它**不是把 LLM 当中间件**，它是**围绕 LLM 搭一个完整的工程栈**。

我们做自家 Agent 产品的时候，这 8 层是个非常好的 checklist——你的系统在哪一层有缺口？

## 第三部分：3 个让 Claude Code "长得不一样"的设计取舍（4分钟）

【滚动到第三节 · 3 个决策】

我挑 3 个 Anthropic 内部反复打磨的反直觉决策——**正是这些取舍把 Claude Code 跟 Cursor / Devin 区分开**。

### 决策 1 · CLI-FIRST vs IDE-FIRST

为什么选 CLI 不选 IDE-native？

Cursor 路线：**IDE 中心**。AI 是"编辑器里的助手"。代价是绑死单一 IDE、远程 / SSH / Docker / CI 场景半残。

Claude Code 路线：**CLI 中心**。CLI 是工程师的最小公约数——本地、SSH、Docker、CI、GitHub Action 都能跑同一个 binary。IDE 插件只是包了 UI 的 CLI。

Boris Cherny 在 ai.engineer 2025 演讲里有句话——**"任何长任务都要能在远程跑、能在 CI 跑、能进自动化流程。IDE 形态把这些都堵死了。"**

### 决策 2 · SINGLE AGENT + SUBAGENT vs MULTI-AGENT

为什么选 single-agent + subagents，不选 multi-agent？

AutoGen / CrewAI 路线：**多个长期存在的 agent 角色**（Planner / Coder / Tester）互相对话。代价是 context 漂移、消息爆炸、谁拍板说不清楚。Anthropic 的《Building effective agents》blog 明确说 **"don't do this unless you must"**。

Claude Code 路线：**主 agent 一条命令链**。需要隔离上下文的子任务时 spawn 一个 subagent，**跑完返回结论就关**。Subagent 不污染主线 context，主线掌握全局。

这个决策我特别认同。我们项目早期试过多 agent 协作（需求提取 agent + 分类 agent + 优先级 agent），三个 agent 互相调结果，调试爆炸——一个 case 出问题，要看三份 trace 才能定位。最后回到单 agent + 工具的路线，调试简单一倍。

### 决策 3 · SHELL HOOKS vs PLUGIN SDK

为什么选 hooks 不选 plugins？

VS Code 路线：**插件是语言绑定 SDK**（TypeScript / Python），有生命周期、有 API、有上架审核。代价是扩展门槛高、生态分裂、跟 shell 工具链脱节。

Claude Code 路线：**扩展点 = 一行 shell 命令**。PreToolUse / PostToolUse / Stop / Notification 四个事件点，挂任何能在终端跑的东西——bash、jq、pytest、curl、Slack webhook。**组合即扩展，无 SDK**。

这个决策的优雅之处——它把 **Unix 哲学**搬进了 AI agent。Unix 之所以强大，是因为它有 shell + pipe + small tools。Claude Code 之所以可扩展，是因为它把"扩展点"做成了 shell hooks——任何能写 shell 的人都能扩展它。

我亲眼见过一个团队，在 PreToolUse 挂了一行 `jq + curl` 把所有工具调用发到他们公司的审计系统。**3 行 shell 解决了"AI 工具调用审计"这个本来要做半年的工程**。这就是 hook 的威力。

## 第四部分：演示二 · 工具集对比（4分钟）

【滚动到演示二 · 工具集对比】

刚才提到 Claude Code 的工具系统："专用工具 > 通用 Bash"。这一点我们用真实任务对比一下。

【点"任务 A · 找 import 路径不一致的 bug"】

**用"只有 Bash"做这个任务**——

工程师可能觉得：grep 不就够了？真正跑下来：

1. `grep -r "import.*foo" .`——漏 quote，没匹配到引号里的内容
2. 改成 `find . -name "*.ts" | xargs grep import`——能跑了
3. 整文件 cat src/utils/foo.ts——70k tokens 进 context
4. cat src/components/bar.tsx——又一个大文件
5. 用 sed 做替换——macOS 上的 BSD sed 语法和 Linux 不一样
6. **FAILED**，部分文件被破坏，要回滚
7. echo $? 检查状态、手动收拾

**结果**：7 次工具调用、92k context 占用、部分失败、需要回滚。

**用"Read / Edit / Glob / Grep 全套"**——

1. `Grep(pattern="from ['\"].*foo['\"]")`——精确匹配，12 处匹配在 6 个文件
2. `Read(src/utils/foo.ts)`——带行号、只读相关片段、4k tokens
3. `Read(src/components/bar.tsx)`
4. `Edit` 批量改 6 文件
5. `Bash(npm run typecheck)`——PASS

**结果**：5 次工具调用、18k context、一次过。

【点"任务 B · fetch 重构成 axios"】

同样的对比模式：Bash-only 跑了 8 步、128k context、勉强完成；Full toolset 跑了 6 步、24k context、一次过。

【点"任务 C · 补单元测试"】

Bash-only 在 heredoc 里栽跟头（模板字符串和反引号转义打架），Full toolset 用 Write 一次写入测试文件。

### 为什么专用工具更准？

【看下面绿色块】

**Bash 是个"通用对话"**——LLM 要先猜命令、再 parse 输出、再决定下一步。每一步都可能出错。

**Read / Edit / Grep 是结构化 API**——参数有 schema，输出有格式，失败模式可预期。LLM 不用记 grep / find / sed 的 flag 语法，直接调 API。

Erik Schluntz 在 Latent Space 上说过：**"给 LLM 一个有 schema 的工具，它的成功率比给它一个 shell 提高一个数量级。"**

这不是 Claude Code 的发现——OpenAI 的 function calling、Anthropic 的 tool use 都是基于这个观察。但 Claude Code 把这个原则**贯彻到了极致**——你看每一个内置工具，都是把 Bash 能做的事拆成一个有 schema 的 API。

### 给我们项目的启示

我们项目里有没有这种"LLM 用 Bash 做本可以做成专用工具的事"？

有。比如调研问卷生成时，我们让模型"基于上传文档生成问题"——背后是一段 Python 代码读文档、提取片段、塞 prompt。**这段 Python 完全可以做成一个 schema 化的工具**：

```
extract_doc_snippets(doc_ids: List[str], topics: List[str]) -> List[Snippet]
```

参数明确、返回明确。比让模型自己拼提取 prompt 准确得多。**所有的"喂数据给模型"的步骤都该是 schema 化的工具，不是 prompt 拼接**。

## 第五部分：演示三 · Auto-compact 是怎么工作的（4分钟）

【滚动到演示三 · Auto-compact】

Claude Opus 现在能跑 **1M context**。问题来了——既然能跑 1M，为什么还需要 compact？

**因为 1M 不是免费午餐**：

1. **成本**：1M token 一轮调用，输入 + 输出 + cache miss，单次调用十几美金不奇怪。
2. **延迟**：1M 上下文模型 prefill 时间显著上升，第一个 token 出来可能要 30 秒。
3. **注意力衰减**：所有大模型都有 lost-in-the-middle 问题——超长上下文里，中段信息容易被忽略。

所以 Claude Code 默认在 **200k 容量、85% 阈值** 触发 auto-compact。

【点击"▶ 推进 5 轮"按钮，反复点直到接近 85%】

看 context 条逐步填满——从 0 到 85%，我们模拟了 30 轮工具调用的过程。每一条上下文条目都标了类别（USER / AGENT / TOOL / DECISION / ERROR / TODO）和大小（K token）。

【点击"⤡ 手动 COMPACT"】

触发 compact 后看——

**保留（绿色 ✓）**：
- 初始 USER 指令——任务到底是什么
- DECISION——用 axios 不用 fetch、用 axios timeout
- 每个 Edit 后留下的"已修改的文件路径"——下次还要改要知道
- ERROR——lint 错误、test timeout 是没解决的
- TODO——清理未使用 import、增加 axios timeout

**丢弃（红色 ✗ 划掉）**：
- 寒暄——"好的"、"明白"、"准备开始"
- 重复 Read——同一文件被读了 3 次，只留最新或不留
- 已完成的中间检查——多次 `npm run lint` 的中间输出
- 已答复的疑问——"也改 billing 吗？"已经答了

### Compact 的设计哲学

【看右侧 COMPACT RULES】

**Keep**：决策、未完成 TODO、文件路径与符号、错误现场。**这些是"未来还要用到的状态"**。

**Drop**：寒暄、重复读、已完成中间产物。**这些是"过去的过程，结论已经在了"**。

这个区分是 Anthropic 内部反复打磨的 heuristic。**核心思想**——agent 的 context 不该是"对话历史"，而该是"任务状态的最新快照"。

### 给我们项目的启示

我们项目的 Agent 对话怎么处理 context？现在做法很粗——保留全部历史，超过窗口就丢最早的。**这是错的**。

正确做法：

1. 设一个 compact 阈值（80%-85%）
2. 触发时让模型自己生成一个"任务状态摘要"——决策、文件、未完成项、错误
3. 用这个摘要替代原始历史

这个机制 LangGraph 提供了原语（`MessagesPlaceholder + summary`），Anthropic SDK 直接支持（system message + 显式压缩调用）。**搭建成本很低，收益很大**——一个长对话的 Agent，开 compact 之后 token 成本能下降 60-80%。

## 第六部分：Skills 系统 · 按需触发 vs 全量注册（3分钟）

【滚动到 Skills 章节】

**Skills 是 2025 年 7 月之后引入的新特性**，是 Claude Code 在"扩展能力"这件事上的一次重大升级。

### 痼疾：Plugin 模式

【看左边对比框】

传统 plugin 模式——你装了 50 个插件，每个插件的功能描述都得在 system prompt 里列一遍。**就算用户根本不会用到，也在烧 token**。

50 个插件 × 平均 800 token = system prompt 多 40k token。**每次对话都付费、都干扰注意力、新加一个插件老 prompt 都要重测**。

### Skills：按需触发

【看右边】

Skill 是一个**文件夹 + frontmatter + 文档**。结构大概是：

```
~/.claude/skills/docx-handler/
├── SKILL.md       # 主文档：怎么处理 docx
├── examples.md    # 示例
└── scripts/       # 工具脚本（可选）
```

`SKILL.md` 的开头是 frontmatter：

```yaml
---
name: docx-handler
description: Use this skill whenever the user wants to create, read, edit, or manipulate Word documents (.docx files). Triggers include mentions of "Word doc", "word document", or ".docx".
---
```

**关键就是这个 description**。

Skill 路由器扫描的是**所有 skills 的 description**——平均 60 token 一个。50 个 skill × 60 token = 3k token，比 plugin 模式少了 90%+。

只有当用户消息匹配某个 skill 的 description 时——比如"帮我把这个 .docx 改一下"——路由器才**把这个 skill 的完整 SKILL.md 注入 context**。其他 49 个 skill 一动不动。

### 给我们项目的启示

【看下面灰色块】

**Skill 的本质是"延迟加载 + 语义路由"**。

我们项目里有多少能力？需求提取、调研问卷生成、合同条款抽取、客户健康度分析、销售话术生成⋯⋯每一个都是一个能力。

现在做法：**全塞进 system prompt 或者分散在不同的 endpoint**。

正确做法：**做成 Skill 风格的能力包**——每个能力一个 `SKILL.md`，写清楚 description（什么时候用这个）、说明、示例。Agent 接到用户消息后，路由器先扫描 description 匹配，命中再加载完整 SKILL.md。

**这个机制完全可以搬到自家产品**。LangChain、LlamaIndex 都有类似的"工具描述路由"原语。我们的 KB Agent 应该用这个模式重构——能力清单按 description 提供，**实际 prompt 用到时再 inject 全文**。

## 第七部分：Tool 设计的精巧细节（3分钟）

【滚动到 Tool Design 章节】

看 4 个看似冗余、实则反复踩坑后的设计。

### 细节 1 · Read / Edit / Write 为什么要拆？

不是"读写文件就行"——三个工具**互相设了前置条件**：

- **Read 强制带行号**——为了让 Edit 能精确定位（用行号或者上下文锚点）。
- **Edit 拒绝在没 Read 过的文件上动手**——防止"凭印象改"。
- **Write 是覆盖式的、专用于新建**——避免误用 Edit 把整个文件干掉。

这三个约束从**入口**就把"一边读一边猜一边改"的常见错误模式堵掉了。

### 细节 2 · Bash 的子进程 / 超时 / background 三件套

- 每次 `Bash()` 调用**都开新子进程**——不继承 shell 状态、不残留 env 变量。**这避免了"上次 cd 了某个目录，下次还在那里"的脏状态**。
- 默认 2 分钟超时，最长 10 分钟。**避免 LLM 启动一个 server 然后死等**——这是个非常常见的失败模式。
- `run_in_background=true` 让长任务（dev server / test runner）后台跑——不占当前 tool turn，agent 可以继续做别的。

### 细节 3 · Glob / Grep 为什么要独立？

Bash 里写 `find . -name '*.ts' | xargs grep ...` LLM 一半会漏 quote、漏 escape、误 alternation 顺序。

Glob / Grep 是**有 schema 的 API**——pattern、path、include、exclude、line_count 全是参数。**LLM 不用记 find / grep 的 flag 语法**。

Anthropic 内部 eval 显示：**同样的 codebase 检索任务，专用工具成功率比 Bash 通配高一个数量级**。

### 细节 4 · Tool 输出为什么要截断？

读一个 30k 行的 log、grep 一个 5MB 的源码 dump——**一次工具调用就能填满 context**。

Claude Code 在 tool 层做**智能截断**——长输出留头尾 + 提示"中间省略 N 行"，让 LLM 主动决定要不要二次定位。

这是**把 context 预算从"事后压缩"提前到"事前裁剪"**。Auto-compact 是兜底，工具层截断是第一道闸。

### 总结

这 4 个细节背后的共同思想——**每一个工具调用都是一次小型的工程决策**。工具应该有 schema、有约束、有 fallback、有截断。**给 LLM 提供的不应该是开放的 shell，而应该是结构化的 API**。

## 第八部分：演示四 · Hook 编程演示（3分钟）

【滚动到演示四 · Hook 编程】

Hook 是 Claude Code 最被低估的特性。我用 demo 演示一下它的威力。

【看四个 hook 槽：PreToolUse / PostToolUse / Stop / Notification】

**4 个事件点**：

1. **PreToolUse**：每次工具调用**前**触发。**返回非零退出码可以拦截工具调用**——这是最强大的能力。
2. **PostToolUse**：工具调用**后**触发。不可拦截但可触发副作用。
3. **Stop**：Agent 决定结束当前回合时触发。常用于收尾检查。
4. **Notification**：需要用户注意时触发（任务完成、等待输入）。

每个槽都可以挂一行 shell。看 demo 提供的几个 preset：

- **PreToolUse · block_rm**：grep 输入是否包含 `rm -rf`，是就 exit 1 拦截。**5 行 shell 解决"AI 不能跑危险命令"**。
- **PostToolUse · auto_format**：每次 Edit 后跑 prettier。**3 行 shell 解决"AI 改完代码自动格式化"**。
- **Stop · run_lint**：Agent 想停的时候强制跑 eslint，有 error 就阻止 stop。**保证不会留下 lint 失败的代码**。
- **Notification · slack**：任务完成发 Slack。**让团队知道 agent 跑完了**。

【点"模拟执行" + 选"场景 · 触发 rm -rf"】

看时间线：
1. 用户："清理一下旧 build 产物"
2. Agent 规划："我要跑 `rm -rf build/ node_modules/.cache/`"
3. **PreToolUse · block_rm**：检测到 rm -rf 模式，exit 1
4. **✗ 工具调用被拦截！rm 未执行**
5. Agent 调整策略：换成 `find build -type f -name "*.tmp" -delete`——安全清理
6. Stop hook 触发、Notification 发 Slack

【场景 · 正常重构】

看一次"无危险" 的任务——hook 在每一步都触发，但不拦截：
1. PreToolUse 记录到 audit
2. Edit 执行
3. PostToolUse 跑 prettier
4. Write 测试
5. PostToolUse 跑 jest
6. Agent 想 stop → Stop hook 跑 eslint → 全 pass
7. Notification 发 Slack

**整个过程没有 SDK、没有插件、没有 API 调用——全是 shell 一行命令**。

### Hook 的工程哲学

**Unix 哲学被搬进了 AI agent**——pipe、filter、composable shell。

我们项目的 Agent 系统应该提供类似机制。具体怎么做？

**给 Agent 的关键节点暴露 webhook**：

- 工具调用前 → webhook 接收 `{tool_name, args}`，可以返回 `{allow: false}` 拦截
- 工具调用后 → webhook 接收 `{tool_name, result}`，用于审计
- 任务完成 → webhook 通知（钉钉、企微、Slack）

**3 个 webhook 端点解决了 80% 的企业扩展需求**——客户能挂自己的审计、加敏感词、连 OA 系统。比写 SDK 简单一个数量级。

## 第九部分：从 Claude Code 学到什么搬到自家产品（2分钟）

【滚动到 Takeaway 章节】

6 条 Takeaway，全部都跟我们项目对得上。

### Takeaway 1 · Subagent 隔离

我们项目的**文档转写脱敏**就是典型 subagent 场景——主线 Agent 不该看到全文，只该拿到脱敏后的版本。把脱敏当 subagent 隔离掉，主线 context 干净。

### Takeaway 2 · Hook 等价的"扩展点设计"

不要给自家 Agent 设计 SDK / plugin 体系。**给每个关键节点暴露一个 webhook 或 shell 钩子**就够——客户能自己挂审计、加敏感词、连 OA。

### Takeaway 3 · Skill 按需加载

不要在 system prompt 里塞 50 条"如果用户问 X 就 Y"。**把每个能力做成 SKILL.md**，路由器只看 description。我们项目的「调研问卷生成」「合同条款抽取」都该是 skill，而不是塞进主 prompt。

### Takeaway 4 · 终端 first 但 UI 辅助

SaaS 产品大多 UI-only。但**给 power user 留一个 CLI / API 入口**，让他们能批处理、能脚本化、能进 CI。我们项目应该提供一个 `kbctl`——脚本化跑 KB 同步、批量脱敏、状态查询。**UI 是包装，CLI 是骨架**。

### Takeaway 5 · 工具层就开始裁剪 context

不要等 auto-compact 才管 context——**在工具输出层就做截断**。我们项目的 RAG 检索如果一次返回 30 段，要在工具层只回前 5 段 + 提示"还有 25 段，需要看请指定"。

### Takeaway 6 · 单一长循环 + 持久 transcript

不要写"对话历史用 8 条窗口、超了就丢"的逻辑。**持久化整个 JSONL transcript**，让 agent 可以 resume、可以回放、可以审计。**断网重启不丢**。

## 收尾（1分钟）

【滚动到总结】

把今天的核心收束成一句话——

> **一个 AI Agent 的上限，80% 决定于模型；但能不能稳定到那 80%，全看 harness。**

我们这一讲从 8 层架构开始，到 3 个关键决策，到 4 件套（Skills / Tools / Auto-compact / Hooks），到 6 条 Takeaway——核心就是要把"harness 是 AI 工程化的工程骨架"这件事**说透**。

模型层（Claude Opus / Sonnet）大家都用得到，区别拉不开。**真正区别在 harness**——你怎么管 context、怎么设计工具、怎么留扩展点、怎么做安全和审计。Claude Code 是这件事的标杆产品。

**对我们项目最实用的 4 条建议**：

1. 给 Agent 设计 **subagent 隔离机制**——长任务不污染主线 context
2. 在系统关键节点暴露 **webhook / hook**——让企业客户自挂审计和合规
3. 把能力做成 **Skill 风格的延迟加载**——system prompt 永远轻
4. 提供一个 **CLI 入口**——让 power user 能脚本化 / 进 CI / 进自动化流程

## 备注：常见疑问

**Q: Claude Code 现在能用吗？怎么访问？**
A: 2025 年 GA，需要 Anthropic Claude API key 或 Pro 订阅。安装：`npm install -g @anthropic-ai/claude-code`。文档在 docs.claude.com/claude-code。

**Q: Claude Code 跟 Cursor Composer 比怎么样？**
A: 两者定位不同。Cursor Composer 适合 in-IDE 的对话式重构（半小时内的任务），Claude Code 适合脱离 IDE 的长任务（重构 + 测试 + 提 PR、CI 跑修复脚本）。**不是替代关系**——很多团队两个都用。

**Q: 我们项目能直接抄 Claude Code 的架构吗？**
A: 不能直接抄（产品形态不同），但**可以抄它的工程思路**。每一条 Takeaway 都是可以独立搬运的设计原则。我建议先抄 3 条：subagent 隔离、webhook 扩展、JSONL 持久化。

**Q: Skills 这个机制实现复杂吗？**
A: 不复杂。核心就是"description 路由器"——扫描所有 skill 的 description，让模型选用哪个，再加载完整 SKILL.md。LangChain `Tool description-based routing`、LlamaIndex `tool_retriever` 都有现成实现。**搭建成本 1-2 天**。

**Q: Auto-compact 在我们项目能直接复用吗？**
A: 思路能复用，实现要重写——因为我们项目的 context 结构跟 Claude Code 不一样。具体做法：在每个 turn 结束时检查 token 用量，超过 80% 就让模型生成"任务状态摘要"（决策 + TODO + 文件 + 错误），用摘要替代原始历史。**Anthropic SDK 直接支持，不需要额外框架**。

**Q: 1M context 我们应该用吗？**
A: 看场景。短对话（&lt;100 turn）默认 200k 够用。长对话或者大文档检索场景，1M 有用。**但 1M 不能滥用**——cost 涨 5 倍、延迟涨 3 倍、注意力衰减问题没解决。建议先用 200k + auto-compact，必要时升 1M。

**Q: 给我们的 CRM SaaS 做 CLI 入口有意义吗？**
A: 对 power user 和企业 ops 团队非常有意义。比如客户的运维想脚本化导入 1000 个文档，没有 CLI 只能写 N 个浏览器自动化或者用我们的 REST API——**CLI 是最低门槛的脚本化入口**。3 天工程量，长期价值大。
