## 第一部分：模型的四个根本局限（5分钟）

【看第一节的局限对照】

为什么一定要给模型装工具？因为纯Prompt + RAG的模型有四个根本性局限。**不是"功能不全"，是"结构性的"——靠优化Prompt永远解决不了**。

### 局限1：不能访问实时数据

模型不知道"现在"。它无法查今天的库存、本月的销售数据、客户的最新工单状态。

它的训练数据停在某个时点，**之后发生的所有事，它一无所知**。RAG解决了"私有知识"问题，但RAG本质上还是"查文档"——查不了**当下的、实时变化的**业务数据。

### 局限2：不能执行任何动作

模型只能"说话"。它不能**真正去做**——发邮件、创建工单、修改记录、触发审批流。

它最多输出"建议你这么做"，但做不了。这是它和"真同事"的本质差距——同事能动手，AI只能动嘴。

### 局限3：不擅长精确计算

大模型本质是"概率预测下一个token"，**不是计算器**。

复杂的数学（百万级数据求和、统计分析、复杂公式）容易出错。我们做项目时遇到过——让模型算"Q3毛利率"，它给的数字看起来很合理，**实际上是它编的**。这种错误最危险——因为它看起来对。

### 局限4：知识有截止日期

训练截止后的事它不知道。RAG解决了"私有知识"，但**没解决"实时事件"**——比如今天的股价、刚发生的会议结果、刚到的工单。

### 解决方案的本质

【看下面绿色的横幅】

**Function Calling = 给模型装上手**。

让它能调用外部工具——查数据库、发邮件、调API、跑计算。模型从"会说"变成"能做"。

这不是花活，是**大模型从"聊天助手"进化到"能干活的同事"的必经之路**。

## 第二部分：5步交互流程（8分钟）

【滚动到第二节的交互演示】

很多人对Function Calling有个误解——以为"模型直接执行函数"。**不是这样**。

Function Calling实际上是模型和你的代码之间的**5步舞蹈**。看一个真实例子。

### 看演示：查询客户业绩

【点击"查询客户业绩"】

注意右上角的"REPLAY"按钮，可以重看动画。

**Step 1**：用户问——"客户A本月签了几个单？金额多少？"

**Step 2**：模型在心里"想"——这个用户想查询客户A本月的销售业绩。我需要：①找到客户A的ID ②查询本月订单。**我应该调用 `search_customer` 工具**。

**Step 3**：模型生成一个**结构化的工具调用请求**：

```json
{
  "name": "search_customer",
  "arguments": { "name": "客户A" }
}
```

**注意——模型只是"生成了"这个JSON，它没有真的去查**。它把JSON输出给我们的代码。

**Step 4**：我们的代码看到这个JSON，**真正去调用CRM接口**，查到了客户A的ID `CUST-2024-001`。把结果作为消息塞回给模型。

**Step 5-6**：模型继续"想"——已经有客户ID了，现在需要查订单。再调用 `get_order_history`，参数包括customer_id和日期范围。

**Step 7**：代码再次执行，这次返回了3条订单数据，总金额97.5万。

**Step 8**：模型基于全部信息，**生成最终的自然语言答复**——"客户A本月共签了3个订单，总金额97.5万。其中最大的订单ORD-001为58万..."

### 关键认知：模型不"执行"，只"决定"

这5步里有个关键认知——**模型从来没有真的执行任何函数**。它只做两件事：

1. **决定该调用哪个工具，传什么参数**（输出JSON）
2. **基于工具返回的结果，生成回复**

**真正的执行是你的代码做的**。模型只是"决策者"，代码是"执行者"。

这个分工非常重要——**安全控制点都在你的代码里**。要不要执行？要不要二次确认？要不要记日志？这些都是你来控制，模型决定不了。

### 多步任务的自主性

【点击"多工具组合"】

这个例子更复杂——"帮我生成上周销售周报，发到老板邮箱"。

模型自己拆解成了3步：①查销售数据 ②生成周报 ③发邮件。**没人告诉它要分几步，它自己规划**。

这就是Function Calling的强大之处——**复杂任务自动拆解、按需调用工具、串联结果**。这已经有点Agent的味道了，但还没有完全走到Agent——下一讲展开。

## 第三部分：工具的定义（5分钟）

【滚动到第三节工具定义】

模型怎么知道你有哪些工具可用？你**用JSON Schema声明每个工具**——名称、用途、参数、返回值。

这听起来简单，**但每个细节都直接影响模型的调用准确率**。看一个真实工具的完整定义。

### 看演示：查询订单工具

【点击"查询订单"】

这是我们项目里实际用的工具定义。我把关键设计点拆解给大家看：

**第1个关键点：description要写清楚"什么时候用"**

```
"description": "查询指定客户在某时间段内的订单记录。当用户询问客户的销售业绩、签单情况、订单历史时使用。"
```

注意后面这一句——"**当用户询问...时使用**"。这一句不是写给开发者看的，**是写给模型看的**。这是模型决定"要不要调用这个工具"的关键依据。

很多人写description像写文档——"这个函数返回订单列表"。**这是给开发者看的，模型不知道什么时候该调**。

正确写法是给"使用场景"——"当用户询问销售业绩、签单情况、订单历史时使用"。

**第2个关键点：参数命名要语义清晰**

```
"customer_id": ...  ← 好
"cid": ...  ← 差（含义不明）
```

模型在生成参数时，会按命名"猜测"该填什么。`customer_id` 一看就知道要填客户ID，`cid` 它可能瞎填。

**第3个关键点：required字段要严格**

```
"required": ["customer_id", "date_from", "date_to"]
```

少一个必填——模型会忽略；多一个必填——模型会乱填。**严格按"业务上必须的"来设置**。

**第4个关键点：能枚举的值都用enum**

```
"status": {
  "enum": ["pending", "signed", "paid", "cancelled"]
}
```

不用enum，模型可能生成 "已签订"、"已签约"、"signed"——三种都对但API只认一种。用enum锁定取值，避免无效参数。

### 看第二个工具：发送邮件

【点击"发送邮件"】

注意里面有个特殊字段——`requires_confirmation`：

```
"requires_confirmation": {
  "description": "是否发送前要求用户二次确认（外部邮件建议true）"
}
```

这是个非常重要的设计——**给敏感操作内置"人在回路"机制**。模型在调用时会自动判断"这是发给外部的"，主动设为true，让代码弹出确认框。

### 看第三个工具：创建工单

【点击"创建工单"】

注意看urgency字段的description：

```
"description": "P0=系统不可用，P1=核心功能受影响，P2=次要问题，P3=咨询"
```

每个枚举值的**业务含义都写清楚**。模型才能根据用户描述准确判断紧急程度。

### 工具定义的核心原则

把这3个工具看完，提炼出工具定义的4个核心原则：

1. **description写"何时调用"，不写"是什么"**
2. **参数命名要业务化、语义化**
3. **enum锁定可枚举值**
4. **复杂概念在description里给出业务含义解释**

掌握这4条，你的工具调用准确率会直接拉高一个档次。

## 第四部分：CRM工具集合的设计（4分钟）

【滚动到第四节工具卡片】

给模型装哪些工具？这是个工程设计问题。下面是我们项目实际接的**9个核心工具**，覆盖了CRM场景90%的"动作型"需求。

### 三类工具

我把它们分成三类。

**Q类（Query · 查询类）**——只读，无副作用，可放心调用：
- `search_customer` 客户查询
- `get_order_history` 订单历史
- `analyze_sales_data` 销售分析

**A类（Action · 动作类）**——会修改数据，可能需要确认：
- `create_task` 创建任务
- `send_email` 发送邮件
- `update_record` 更新记录

**W类（Workflow · 工作流类）**——触发流程、跨系统、影响多人：
- `create_ticket` 创建工单
- `trigger_approval` 触发审批
- `schedule_meeting` 安排会议

### 设计原则：三类工具，三种安全策略

**Q类**：默认开放，无需确认。查询不会破坏数据，让模型放心调用。

**A类**：写操作必须有审计日志。重要操作（修改客户、删除数据）需要二次确认。

**W类**：跨系统、影响多人——**必须经过用户确认**。一个错误的审批触发可能让财务多打几十万。

这个分级在系统层实现——你不用让模型来判断"该不该确认"，你的代码层根据工具类别自动加确认机制。

### 工具集合的"完备性"问题

设计工具集合时常犯一个错误——**工具太多或太少**。

太少：覆盖不了用户需求，AI助手能做的事情有限，价值上不去。

太多：模型选择困难，容易调错。我们见过有项目接了40个工具，结果模型经常调错——**工具描述差不多，模型分不清该用哪个**。

经验值——**先从7-12个核心工具开始**。这个数量模型能稳定记住、合理选择。等业务跑起来后，根据实际需求增加。

## 第五部分：6个生产陷阱（5分钟）

【滚动到第五节陷阱】

Function Calling在demo里特别好使——但**真上线问题一堆**。我们踩过的坑总结成6个，希望大家不用再踩一遍。

### 陷阱1：该调工具时不调

用户说"帮我创建任务"，模型回——"**好的，我已经为您创建了任务**"。**但实际上没调用任何工具，它在演戏**。

为什么会这样？因为模型有"完成任务的倾向"——它想让用户开心，所以编造了"已完成"的回应。

**修正**：在System Prompt明确"必须通过工具完成动作，不要只是描述"。代码层校验——发现模型说"已创建"却没有tool_call时强制重试。

### 陷阱2：不该调时滥调

用户问"什么是LTC流程"——这是个知识问题，应该直接回答。但模型却调用了 `search_customer` 查找一个叫"LTC"的客户。

**这种"幻觉式调用"比不调还危险**——会污染数据，还浪费API费用。

**修正**：工具description明确"仅在用户明确需要查询/操作具体客户时使用"，并加examples说明"什么情况不该调"。

### 陷阱3：参数瞎填

模型生成的参数看起来合理但实际错的——把"本月"翻译成 `"this month"` 但API要的是 `"2025-04"`。

**类型不匹配是最常见的运行时错误**。

**修正**：① JSON Schema严格定义类型 ② 工具内部做参数校验，错了返回明确错误信息让模型自纠 ③ 关键字段加examples。

### 陷阱4：多步操作中途失败

模型决定"先创建客户→再创建订单→最后发邮件"，第二步失败时——**客户已经建了，但订单没建**。数据残缺。

**修正**：① 关键链路加事务包裹 ② 失败时显式告诉模型"前面成功了什么、当前失败了什么"，让它决定补救 ③ 写操作日志，可手工回滚。

### 陷阱5：无限循环

模型查询失败→重试→还失败→再重试，**陷入死循环**。我们见过最长一次循环了37次才被熔断，一次对话烧掉几十块钱token费。

**修正**：① 设置最大tool_call次数（如10次） ② 同一工具同一参数失败超过2次就让模型走另一条路径 ③ 设置token预算超了就退出。

### 陷阱6：权限越界

销售小李用AI助手——"列一下所有大客户的应收账款"——结果AI把财务数据全调出来了。**因为工具本身没接权限校验**。

**修正**：① 每个工具调用必须传入用户身份 ② 工具内部按用户角色校验数据范围 ③ 敏感操作（财务、删除、改权限）必须人类二次确认。

### 6个陷阱的共同教训

把6个陷阱串起来，有个共同的教训——

**Function Calling的安全和可靠性，主要不靠模型，靠你的代码层**。

模型负责"决策"，但所有的安全护栏（防滥用、防越权、防无限循环、防数据残缺）都要在你的工具实现里做。**把模型当成一个聪明但不熟悉公司规矩的新员工——你要给他写规则、限制权限、监督操作**。

## 第六部分：MCP——工具生态的"USB-C时刻"（4分钟）

【滚动到绿色的MCP章节】

这一节讲一个2024年最重要的工程演进——**MCP（Model Context Protocol）**。

### 没有MCP的世界

之前每次接入工具，都是**定制开发**：

- 想让AI助手连接Notion？得在代码里写Notion集成
- 想让它连接Slack？再写一遍
- Salesforce？再写

**问题**：
- 每个工具集成都是定制开发，几周时间
- 不同AI产品之间无法复用
- 工具升级要每个产品都改
- 认证、错误处理、限流——每次重写

我们项目里接Fxiaoke的API就是这样——**手动包装了几十个接口**，花了好几周。

### MCP之后

2024年Anthropic推出了**Model Context Protocol（MCP）**——一个**标准化的工具接入协议**。

变化是这样：

**每个应用提供一个标准的MCP Server**——Notion有自己的MCP Server，Slack有自己的，Salesforce有自己的。

**任何AI客户端只要支持MCP协议，就能用所有这些工具**。

**好处**：
- 工具方一次开发，所有AI产品可用
- AI产品一次接入，立刻获得所有工具
- 升级只需改MCP Server，客户端无感知
- 认证、错误处理在协议层标准化

### 类比理解

打个比方——**没有MCP之前，每个充电器都是私有接口**。Apple用Lightning，Android用Micro-USB，每个厂家自己一套。换设备就要换线。

**MCP是"USB-C接口"**——一个标准接口，所有设备都能用。你买了一个USB-C的充电头，能给手机、笔记本、耳机、平板全部充电。

工具生态正在经历这个"USB-C时刻"。

### 对我们的影响

【看下面的灰色提示框】

**对我们项目的意义**：

过去做CRM AI助手，要把Fxiaoke的所有API手动包装一遍。耗时几周。

如果Fxiaoke官方提供MCP Server（或者我们做一个），**所有AI产品（Claude、GPT、自研Agent）都能直接用**。而且未来Fxiaoke升级API，**我们的AI助手自动跟上**。

我个人判断——**未来1-2年，所有主流SaaS都会提供MCP Server**。这是重大机会，也是重大威胁：
- **机会**：CRM能力可以快速集成到任何AI助手里，扩大触达
- **威胁**：客户可能不再用"CRM专用AI助手"，而是用通用AI助手通过MCP接入CRM

我们要尽早思考——**在MCP生态里，CRM SaaS的位置是什么？**

## 收尾（30秒）

【滚动到总结】

把第三部分前三讲串起来——

> **Prompt** 让模型会说  
> **RAG** 让模型知道  
> **Function Calling** 让模型能做  

这三件事齐了，才有真正可用的**企业级AI助手**。

**实践建议**：从"工具集合"出发设计AI助手，而不是从"对话"出发。

具体怎么做？先问自己：**这个角色的同事每天做哪些动作**？把这些动作变成工具，AI助手自然就能替代90%的日常工作。

**举例**：销售经理每天的动作——查客户、看业绩、安排跟进、发邮件、做决策。把这5类动作设计成工具，AI销售助手就有了。

但工具一多，调度就复杂——**什么时候该用哪个工具？多个工具怎么编排？怎么处理异常分支？**

这就引出了下一讲的主角——**Agent**。能自主规划、调度工具、完成复杂任务的AI实体。

下一讲我们会从今天的"模型+工具"出发，看Agent是怎么从这个基础上长出来的。

## 备注：常见疑问

**Q: Function Calling 和 Tool Use 是同一个东西吗？**
A: 基本是。"Function Calling" 是OpenAI早期叫法，强调"调函数"。"Tool Use" 是Anthropic后来推广的叫法，更宽泛——工具不一定是函数，也可能是数据库查询、API调用、甚至子Agent。现在两个词混用，意思一样。

**Q: 模型为什么不能直接执行函数？**
A: 安全考虑。如果模型能直接执行代码，一个prompt injection就可能让它做任何事——删数据库、转账、发恶意邮件。让模型只生成"调用请求"、由代码层执行，是基本的安全设计。

**Q: 我们要不要等所有SaaS都支持MCP了再做AI助手？**
A: 不能等。先用Function Calling手动包装核心工具（10个左右）。MCP生态成熟后再迁移——你的工具定义和Agent逻辑都是可复用的，迁移成本不高。

**Q: 用哪个模型做Function Calling最好？**
A: GPT-4 / Claude Sonnet / Claude Opus 都很好。开源模型里 Qwen 2.5、DeepSeek V3 也不错。**避开早期的小模型**——它们调用准确率低，会生成无效参数。

**Q: Function Calling会不会让Prompt变得很长？**
A: 工具定义确实占token——10个工具大概1500-2000 token。但这部分可以用prompt cache（OpenAI、Anthropic都支持）缓存起来，几乎不增加成本。

**Q: 一次对话最多能调多少次工具？**
A: 技术上没硬限制，但实际控制在10次以内最好。超过10次说明任务太复杂，应该拆成多个对话或引入Agent架构（下一讲）。
