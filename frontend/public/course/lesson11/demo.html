LESSON 11 / 20

FUNCTION CALLING
让模型 ACT.

Function Calling 与 Tool Use
Prompt 让模型"会说"，RAG 让模型"知道"。但模型还不能"做"—— 它不能查数据库、发邮件、改CRM、调外部API。Function Calling 就是给模型装上手， 让它从"聊天助手"进化成"能干活的同事"。
CHAPTER · 01
THE MISSING CAPABILITY
没有工具的模型 = 缺了手的大脑
纯Prompt + RAG的模型有两个根本性局限。它们不只是"功能缺失"，是结构性的—— 靠优化Prompt永远解决不了。

局限 #1 · 不能访问实时数据

模型不知道"现在"。它无法查今天的库存、本月的销售数据、客户的最新工单状态。 所有"需要查询"的问题它都答不了。

用户："客户A本月签了几个单？"
模型："我无法访问您的CRM数据库……"

局限 #2 · 不能执行任何动作

模型只能"说话"。它不能真正去做——发邮件、创建工单、修改记录、触发审批流。 它只能输出"建议你怎么做"，但做不了。

用户："帮我给客户A发一封跟进邮件"
模型："我建议您发邮件包含以下要点……（无法实际发送）"

局限 #3 · 不擅长精确计算

大模型本质是"概率预测下一个token"，不擅长精确数学。 复杂的计算（百万级数据求和、统计分析）容易出错。

用户："算一下我们Q3的毛利率"
模型："假设营收是X，成本是Y……"
→ 数字可能是编的

局限 #4 · 知识有截止日期

训练截止后的事它不知道。RAG解决了"私有知识"， 但没解决"实时事件"——比如今天的股价、刚发生的会议结果。

用户："今天美股开盘后纳斯达克涨了多少？"
模型："我无法获取实时市场数据……"
Function Calling 解决方案：给模型装上手。让它能调用外部工具——查数据库、发邮件、调API、跑计算。模型从"会说"变成"能做"。

DEFINING TOOLS
怎么"告诉"模型有哪些工具可用
模型怎么知道你有哪些工具？你用JSON Schema声明每个工具—— 包括名称、用途、参数、返回值。下面是真实可用的工具定义示例。

关键设计原则：
① description要写清楚"什么时候用"——这是模型决定调用与否的依据，不是写给开发者看的
② 参数命名要语义清晰——customer_id胜过cid
③ required字段严格——少一个必填模型就会调用失败，多一个让模型混乱
④ enum限制取值——能枚举的值都用enum，避免模型生成无效参数

A REAL CRM TOOLKIT
一个真实的CRM工具集合
给模型装哪些工具？这是个工程设计问题。下面是我们项目里实际接的9个核心工具， 覆盖了CRM场景90%的"动作型"需求。
Q1
search_customer
客户查询
按名称/编号/行业搜索客户，返回详情
Q2
get_order_history
订单历史
查询客户的订单、合同、回款记录
Q3
analyze_sales_data
销售分析
按维度聚合销售数据，输出报表
A1
create_task
创建任务
为销售/客服创建跟进任务，设置时间和负责人
A2
send_email
发送邮件
向客户/同事发送邮件，支持模板和附件
A3
update_record
更新记录
修改客户/订单/合同的字段，含审计日志
W1
create_ticket
创建工单
创建客服工单，自动分类和分派
W2
trigger_approval
触发审批
提交折扣/合同/退款审批流
W3
schedule_meeting
安排会议
查询日历空闲时间，创建会议邀请

分类逻辑：
• Q类（查询）：只读、无副作用、可放心调用
• A类（动作）：会修改数据、可能需要确认、要有撤销机制
• W类（工作流）：触发流程、跨系统、影响多人——必须经过用户确认

CHAPTER · 05
COMMON PITFALLS
生产环境里反复出现的6个陷阱
Function Calling在demo里特别好使，但真上线问题一堆。下面6个是我们踩过的坑。
01
该调工具时不调，只在嘴上说

用户说"帮我创建任务"，模型回"好的，我已经为您创建了任务"—— 但实际上没调用任何工具。它只是在演戏。

修正：在System Prompt明确"必须通过工具完成动作，不要只是描述"，并在代码层校验——发现模型说"已创建"却没有tool_call时，强制重试
02
不该调工具时滥调，造成幻觉式调用

用户问"什么是LTC流程"，模型却调用了search_customer查找一个叫"LTC"的客户。 对工具的滥用比不用还危险——会污染数据。

修正：工具description明确"仅在用户明确需要查询/操作具体客户时使用"，且加examples说明"什么情况不该调"
03
参数瞎填，类型错误，必填漏填

模型生成的参数看起来合理但实际错的——把"本月"翻译成"this month"但API要的是"2025-04"。 类型不匹配是最常见的运行时错误。

修正：① 用JSON Schema严格定义类型 ② 工具内部做参数校验，错了就返回明确错误信息让模型自纠 ③ 关键字段加examples
04
多步操作中途失败，无法回滚

模型决定"先创建客户→再创建订单→最后发邮件"，第二步失败时——客户已经建了，但是订单没建。 数据残缺，没人知道。

修正：① 关键链路加事务包裹 ② 失败时显式告诉模型"前面成功了什么、当前失败了什么"，让它决定补救 ③ 写操作日志，可手工回滚
05
无限循环，不停调工具不停止

模型查询失败→重试→还失败→再重试，陷入死循环。我们见过最长一次循环了37次才被熔断。 一次对话烧掉几十块钱token费。

修正：① 设置最大tool_call次数（如10次），超过强制停止 ② 同一工具同一参数失败超过2次就让模型走另一条路径 ③ 设置token预算，超了就退出
06
权限越界，影响他人数据

销售小李用AI助手，让它"列一下所有大客户的应收账款"——结果AI把财务数据全调出来了。 因为工具本身没接权限校验。

修正：① 每个工具调用必须传入用户身份 ② 工具内部按用户角色校验数据范围 ③ 敏感操作（财务、删除、改权限）必须人类二次确认
CHAPTER · 06
THE MCP REVOLUTION
MCP：工具生态的"USB-C接口"
2024年Anthropic推出的Model Context Protocol（MCP）正在成为新标准—— 把"工具"从应用内嵌入变成可插拔的标准协议。这是过去一年最重要的工程演进之一。
BEFORE · 没有MCP的世界

每个产品自己写一遍工具

想让AI助手连接Notion？得在代码里写Notion集成。 想让它连接Slack？再写一遍。Salesforce？再写。

  • 每个工具集成都是定制开发
  • 不同产品之间无法复用
  • 工具升级要每个产品都改
  • 认证、错误处理、限流——每次重写
AFTER · MCP之后

工具变成标准化的"服务器"

每个应用（Notion、Slack、Salesforce）提供一个标准的MCP Server。 任何AI客户端只要支持MCP协议，就能用所有这些工具。

  • 工具方一次开发，所有AI产品可用
  • AI产品一次接入，立刻获得所有工具
  • 升级只需改MCP Server，客户端无感知
  • 认证、错误处理在协议层标准化

对我们项目的意义： 过去做CRM AI助手要把Fxiaoke的所有API手动包装一遍——耗时几周。 如果Fxiaoke官方提供MCP Server（或者我们做一个），所有AI产品（Claude、GPT、自研Agent）都能直接用， 而且未来Fxiaoke升级API我们的AI助手自动跟上。这是工具生态的"USB-C时刻"。

RECAP

核心收束 · 模型从"会说"到"能做"的桥梁

Function Calling 不是炫技，是大模型从"聊天助手"进化到"能干活的同事"的必经之路。 它给模型装上手——访问数据、执行动作、跨系统集成。

Prompt 让模型会说
RAG 让模型知道
Function Calling 让模型能做
—— 这三件事齐了，才有真正可用的企业级AI助手

实践建议：从"工具集合"出发设计AI助手，而不是从"对话"出发。 先问"这个角色的同事每天做哪些动作？"，把这些动作变成工具， AI助手自然就能替代90%的日常工作。

但工具和工具的组合，开始让事情变得复杂——什么时候该用哪个工具？多个工具怎么编排？ 这就引出了下一讲的主角：Agent——能自主规划、调度工具、完成复杂任务的AI实体。
