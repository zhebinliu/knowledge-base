LESSON 13 / 20

FROM ONE
to MANY

多Agent系统设计
上一讲单Agent能干很多事——但企业级业务流程不是一个Agent能扛的。 专业化分工是软件工程亘古不变的真理。 今天我们看：多个Agent如何协作完成复杂任务，又如何不变成一锅粥。
CHAPTER · 01
WHEN ONE IS NOT ENOUGH
单Agent扛不住的四种场景
Agent很强大——但不是所有任务都该用一个Agent。 下面四种情况，单Agent会出问题，你需要专业分工。

边界 #1 · 任务太复杂，工具集太大

一个Agent接30+个工具，它会"忘记"自己有什么，调错工具。 就像让一个人同时当销售、客服、财务、技术——什么都懂一点，什么都做不精。

实测：超过15个工具时，调用准确率从92%掉到67%

边界 #2 · 涉及多种专业领域

需求分析、合同审核、技术方案、定价策略——每个领域都需要不同的"心智"。 同一个Prompt和工具集，无法同时让Agent在多个领域都做到专家水平。

做销售方案时让Agent兼顾合规审查，会两头都做不好

边界 #3 · 需要多视角和"质疑"

一个Agent给的答案就像"一个人独自做决定"——没有审核、没有质疑、没有不同视角。 重要决策需要"多个角色"互相验证。

客户合同条款不利？销售可能放过，但合规Agent会拦下

边界 #4 · 流程跨多个角色

真实业务流程往往涉及多角色——销售提单→风控审核→财务定价→法务合规→交付。 这种流程天然就是多Agent协作的形态，硬塞到一个Agent里反而别扭。

就像让一个员工独自走完所有部门的审批
CHAPTER · 02
FOUR COLLABORATION PATTERNS
多Agent协作的4种基础模式
不同任务匹配不同协作方式。下面是2024-2025年生产环境最常见的4种模式。 每种都有它适合的场景，不存在"最好的"。
PATTERN · 01
Supervisor
CENTRALIZED · 中心化分发
一个 主管Agent 接收任务，根据任务性质路由给专业子Agent。 子Agent完成后把结果交回主管，主管整合后返回给用户。
适用：客服系统（不同问题路由到不同专家）、产品经理统一调度团队
PATTERN · 02
Pipeline
SEQUENTIAL · 流水线
多个Agent 按固定顺序处理，前一个的输出是后一个的输入。 像工厂流水线——每个工位做一件事，完了交给下一个。
适用：流程固定的业务（合同审批、订单处理）、内容生产管线
PATTERN · 03
Debate
ADVERSARIAL · 对抗辩论
两个Agent持对立立场辩论（Pro vs Con），第三个Agent作为裁判（Judge）评估并给出最终结论。 质疑式协作。
适用：高风险决策（合规审查、合同条款）、需要批判思维的场景
PATTERN · 04
Hierarchical
MULTI-LEVEL · 层级化
多层组织结构——CEO层规划战略，经理层拆分任务，员工层执行具体动作。 像真实公司的组织架构。
适用：超复杂任务（年度规划、大型项目）、模拟组织决策流程

CHAPTER · 04
ORCHESTRATION PATTERNS
3种主流编排框架对比
多Agent协作有不同的编排方式。下面是3种主流框架——不是"哪个最好"，是"哪个适合你"。
FRAMEWORK · 01
LangGraph
STATE MACHINE

基于状态图的编排框架。Agent之间的流转用图的边定义，状态在图里流动。 支持条件分支、循环、人在回路。

特点
控制力强、可视化好
门槛
中等，需要理解状态机
适合
复杂业务流程、需要严格控制路径
FRAMEWORK · 02
CrewAI
ROLE-BASED

以"角色 + 任务"为核心抽象。每个Agent定义角色背景和擅长领域，自动协作完成任务。 上手最快，最像"组团队"。

特点
概念直观、原型快
门槛
低，几行配置就能跑
适合
原型验证、内容生产、调研类任务
FRAMEWORK · 03
AutoGen
CONVERSATIONAL

Microsoft出品。Agent之间通过对话协作，可以提问、回答、补充。 支持人类作为其中一个"Agent"加入。

特点
交互灵活、支持人在回路
门槛
中等，需要设计对话规则
适合
需要人介入的业务、研讨型任务

实战经验：99%的项目不需要从框架入手。 先用最基础的"几个Agent + 一段调度代码"原型验证业务流程是否合理；流程稳定后再引入框架。 过早引入复杂框架是新人最大的坑——框架解决的是"协作复杂度"问题，业务还没跑通时不需要。

CHAPTER · 05
SIX PITFALLS
多Agent系统的6个高频陷阱
多Agent比单Agent复杂10倍。坑也多10倍。这些是2024年我们项目踩过的真实坑。
i
"为了多Agent而多Agent" — 过度设计

一个简单任务被拆成5个Agent协作，不仅没提升效果，还引入额外的失败点。 协作overhead高于一个Agent直接做。

判断标准：能用单Agent + 不同的Prompt切换搞定的，绝不用多Agent。Agent数量 ≤ 真实业务角色数
ii
角色定义模糊 — 谁该做什么不清楚

"需求分析师"和"方案架构师"职责重叠，每个Agent都做了一部分，每个又都没做完整。 最后整合的内容互相矛盾。

修正：用"RACI矩阵"明确每个Agent的Responsible/Accountable/Consulted/Informed职责。每件事必须有且只有一个Agent是R
iii
上下文爆炸 — 信息越传越冗

Agent A的输出全部塞给Agent B，B又把所有东西塞给C…到最后一个Agent，context已经是10万token的"大杂烩"。 性能急剧下降。

修正：① Agent间传递的是"结构化结论"而非原始对话历史 ② 关键信息走"共享Memory"，需要的Agent去查 ③ 给Agent定义明确的"输出schema"，不允许传废话
iv
错误传播 — 一个Agent错了，全队跟着错

需求分析Agent提取错了一个关键字段，后面所有Agent都基于错的需求做事。 到最后输出方案，发现根本不是客户要的。

修正：① 关键节点设"质量门"——必须通过验证才进入下一环 ② 引入Reviewer Agent检查每一步输出 ③ 重要决策让用户确认
v
无限互踢 — Agent间死循环

Agent A需要B的结果，B说"先问C"，C说"先问A"——三个Agent在循环互推。 或者两个Agent就同一个问题反复讨论，永远达不成一致。

修正：① 单次任务设最大轮数（如10轮） ② 设"打破僵局"规则——超时由Supervisor仲裁 ③ Agent间通信图是DAG（有向无环）
vi
调试难度爆炸 — 不知道哪儿出了问题

5个Agent协作输出的结果质量差。查问题时不知道是哪个Agent的锅，每个Agent单测都OK，但合在一起就崩。

修正：① 全链路追踪日志（每个Agent的输入/输出/思考都记录） ② 每个Agent独立可replay ③ 引入LangSmith / Langfuse等观测工具（下一讲展开）
CHAPTER · 06
DECISION FRAMEWORK
单Agent vs 多Agent的决策框架
面对一个新业务场景，要不要用多Agent？按下面5个问题问一遍，答案就清楚了。
Q1
业务流程涉及多少个真实角色？
1-2个 → 单Agent + Prompt切换
3+个 → 多Agent（每个角色一个）
Q2
不同步骤需要的"专业心智"差异有多大？
差异小（同一领域不同视角） → 单Agent
差异大（销售/合规/技术） → 多Agent
Q3
是否需要"质疑/对抗"机制？
不需要 → 单Agent或Pipeline
需要（高风险决策） → Debate模式
Q4
流程是固定的还是动态的？
固定（按部就班） → Pipeline
动态（按情况分发） → Supervisor
Q5
业务流程是否已经稳定（不会频繁变化）？
不稳定 → 先用单Agent验证
稳定 → 再考虑多Agent架构
RECAP

核心收束 · Agent 的组织化

多Agent系统不是"更厉害的AI"，是把"组织"的概念引入AI系统。 每个Agent专业化、有明确职责、通过定义好的协作规则共同完成大型任务。

"单Agent是个全能员工；
多Agent是个有分工的团队。
团队的效率不在每个人多强，
在协作设计有多好。"

4种协作模式——Supervisor（中心化）、Pipeline（流水线）、Debate（对抗）、Hierarchical（层级）—— 对应不同的业务场景。真实项目里往往是这几种的混合。

对我们的意义：CRM实施项目本身就是多角色协作的——售前、产品、技术、合规、交付。 把这种协作模式映射到AI系统里，能真正自动化复杂的业务流程， 不只是回答问题，是替代一个团队的产出。

但也要警惕——多Agent是高级武器，不是入门玩具。 在没有把单Agent做扎实之前，别盲目上多Agent。先单后多，先简后繁，是工程纪律。
