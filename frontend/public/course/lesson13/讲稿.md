## 开场（2分钟）

第三部分的第五讲。上一讲我们讲了Agent——给模型装上自主性。

但有个明显的问题——**单Agent是有极限的**。我先抛个真实场景。

我们项目里有个客户要求："给我做一个AI助手，能从客户访谈里提取需求、设计方案、估算人天、写报价、做合规审查、生成最终PPT"。

听起来是一个完整的"售前自动化"对吧？我们一开始是这么做的——一个Agent接30个工具，覆盖所有环节。

**结果一塌糊涂**。

工具调用准确率从demo时的92%掉到了实际场景的67%。Agent经常调错工具——本来该查产品库的去查了客户库、本来该用合规模块的去用了报价模块。客户拿到的方案文档里，"需求分析"和"实施规划"互相矛盾。

为什么？因为**让一个Agent做30件事，就像让一个员工同时当销售、客服、财务、技术、法务**——什么都懂一点，什么都做不精。

这就是今天要解决的问题——**多Agent协作**。专业化分工是软件工程亘古不变的真理，AI系统也不例外。

打开 demo.html，我们一起来看。

## 第一部分：单Agent的四个边界（5分钟）

【看第一节的限制对照】

什么时候单Agent扛不住？四种情况。

### 边界1：任务太复杂、工具集太大

我刚才说的就是这个——**一个Agent接30+个工具，它会"忘记"自己有什么**。

实测数据：超过15个工具时，调用准确率从92%掉到67%。**这不是模型不够强，是注意力被稀释了**——每个工具的description要进context，模型很难在一长串工具里准确选出最合适的那个。

类比一下：你给同事一个有20个抽屉的工具箱，他用着用着就开始翻错抽屉。

### 边界2：涉及多种专业领域

需求分析、合同审核、技术方案、定价策略——**每个领域都需要不同的"心智"**。

一个Agent的system prompt不可能同时让它在所有领域都做到专家水平。让它做销售方案时严格审视合规问题，它会两头都做不好——**要么方案不够进取，要么合规漏掉细节**。

### 边界3：需要多视角和"质疑"

一个Agent给的答案就像"一个人独自做决定"——**没有审核、没有质疑、没有不同视角**。

但企业级决策很多时候需要"角色对抗"。比如客户合同条款不利？销售视角可能放过（"先签了再说"），合规视角必须拦下（"这条违约金条款风险太大"）。

**单Agent模拟不了这种内部博弈**。

### 边界4：流程跨多个角色

真实业务流程往往涉及多角色——销售提单 → 风控审核 → 财务定价 → 法务合规 → 交付。

**这种流程天然就是多Agent协作的形态**。硬塞到一个Agent里反而别扭——它要一会儿变成销售、一会儿变成风控、一会儿变成财务，角色频繁切换非常累，质量肯定不好。

### 总结：什么时候该上多Agent

把这4个边界串起来，**当你看到"工具多、领域杂、需要质疑、流程跨角色"中的任意两个，就该考虑多Agent了**。

但请记住——**多Agent是高级武器，不是入门玩具**。在没有把单Agent做扎实之前，别盲目上多Agent。这个判断我们最后会展开。

## 第二部分：4种协作模式（8分钟）

【滚动到第二节4种模式】

不同任务匹配不同协作方式。下面是2024-2025年生产环境最常见的4种模式。**每种都有它适合的场景，不存在"最好的"**。

### 模式1：Supervisor（中心化分发）

【看左上角的图】

最直观的模式——**一个主管Agent + 多个专业子Agent**。

主管接收用户任务，根据任务性质判断该路由给哪个子Agent。子Agent完成后把结果交回主管，主管整合后返回给用户。

**就像一个团队的负责人在分活**。

**适用场景**：
- 客服系统——不同问题路由到不同专家（技术问题→技术Agent、财务问题→财务Agent）
- 产品经理统一调度团队
- 任何需要"总体把控+专业执行"的场景

这是**最常见的多Agent模式**，原因简单：好理解、好实现、好调试。一旦出问题，主管Agent的日志能告诉你它把任务分给了谁、为什么。

### 模式2：Pipeline（流水线）

【看右上角】

**多个Agent按固定顺序处理**，前一个的输出是后一个的输入。

像工厂流水线——每个工位做一件事，完了交给下一个。这个模式的特点是**流程是固定的、单向的**。

**适用场景**：
- 流程固定的业务（合同审批、订单处理、内容生产管线）
- 多步骤但每步逻辑独立

举例：销售合同审批流程——销售生成合同 → 法务审核条款 → 财务核算价格 → 总监审批 → 归档。每一步都是固定的，没有跳过、没有回流。

**Pipeline的好处**：每个Agent只关心自己的部分，简单。**坏处**：不灵活，前一步出错后面全乱。

### 模式3：Debate（对抗辩论）

【看左下角】

这是个有意思的模式——**两个Agent持对立立场辩论，第三个Agent作为裁判**。

比如审查一个合同：
- **Pro Agent**：找这个合同对我们有利的地方
- **Con Agent**：找这个合同对我们不利、有风险的地方
- **Judge Agent**：综合两方观点，给出最终评估

**这种"质疑式协作"在高风险决策里特别有用**。它强迫系统从对立视角看问题，避免"思维定式"。

**适用场景**：
- 高风险决策（合同条款、合规审查）
- 需要批判思维的场景
- 多个合理方案需要权衡时

我们项目用过这个模式做合同审查，效果惊人——比单Agent审查多发现了**40%的潜在风险点**。因为Pro和Con互相挑刺，没什么能蒙混过关。

### 模式4：Hierarchical（层级化）

【看右下角】

**多层组织结构**——CEO层规划战略，经理层拆分任务，员工层执行具体动作。像真实公司的组织架构。

**适用场景**：
- 超复杂任务（年度规划、大型项目）
- 模拟组织决策流程
- 需要多层抽象（战略→战术→执行）

举例：做一个完整的"客户营销活动"——
- **CEO Agent**：定战略——这个客户群我们要做"激活老客户"
- **Manager Agent**：拆战术——分成"邮件营销+APP推送+电话回访"
- **Worker Agent**：执行——具体写邮件、发推送、安排电话

这个模式很强大，**但也是最复杂的**。一般只在确实需要多层抽象时才用，否则容易过度设计。

### 实际项目里怎么选

我给一个简单的判断：

- 任务类型多样、需要分发 → **Supervisor**
- 流程固定、按部就班 → **Pipeline**
- 高风险决策、需要质疑 → **Debate**
- 超大型项目、多层抽象 → **Hierarchical**

**实战中最常见的是几种模式的混合**。比如总体是Supervisor，关键节点用Debate做审查，部分子流程用Pipeline——这是一个成熟多Agent系统的真实形态。

## 第三部分：5-Agent CRM团队（10分钟）

【滚动到第三节CRM案例】

光说模式太抽象。我把我们项目里一个真实简化案例展示给大家——**用5个专业Agent协作完成"销售方案生成"任务**。

【看任务描述】

"客户A是一家200家门店的零售连锁，现有CRM老旧。请生成一份完整销售方案——包含需求分析、解决方案、实施计划、报价、合规审查。"

这个任务跨5个专业领域，**完美的多Agent场景**。

我们设计了5个Agent：Researcher（需求分析师）、Architect（方案架构师）、Planner（实施规划师）、Pricer（报价分析师）、Compliance（合规审查员）。

整体协作模式：**Pipeline主体 + Debate补充**——前4个Agent按顺序工作，第5个Compliance Agent作为质量门审查所有环节。

### Agent 1：需求分析师（Researcher）

【点击第一个Agent】

**使命**：把客户的口头需求和原始资料，转化为结构化的需求清单。

**为什么是它打头阵**：因为它的输出是后面4个Agent的输入。**起点错了，全队跟着错**。

**核心职责**：
- 解析客户提供的原始资料（访谈、文档、邮件）
- 提取显性需求 + 推断隐性需求
- 需求分类：业务流程类 / 数据类 / 集成类 / 报表类
- 评估每条需求的优先级和复杂度
- 识别需求中的矛盾和模糊点，标记"需澄清"

**关键工具**：
- read_documents（读取客户上传的资料）
- search_similar_cases（查相似项目案例参考）
- query_industry_template（查行业最佳实践模板）

**Prompt心智**：让它扮演"经验丰富的需求分析师"——会刨根问底、识别隐含需求、对模糊表述提出澄清问题。

### Agent 2：方案架构师（Architect）

【点击第二个Agent】

**使命**：基于需求清单，设计技术上可行、商业上合理的解决方案。

**这是整个团队最依赖"经验"的角色**。

**核心职责**：
- 把需求映射到产品能力（如纷享销客的对应模块）
- 识别需要定制开发的部分
- 设计系统架构图
- 提出标准方案 + 升级方案（让客户做选择）
- 列出潜在技术风险

**关键设计**：让它输出**两个方案**——标准版和升级版。这给客户决策空间，也展示我们的专业度。**单一方案是新人作品，多方案是顾问作品**。

### Agent 3：实施规划师（Planner）

【点击第三个Agent】

**使命**：把方案转化为可执行的项目计划——WBS、人天、里程碑、资源配置。

**关键工具**：
- query_wbs_template（查WBS模板）
- get_team_capacity（查团队产能数据）
- query_historical_velocity（查历史项目速率）

**最值得说的工具是 query_historical_velocity**——它能查出"我们历史上做类似项目的实际速率"。新人估时容易过度乐观，有这个工具能让Agent的估算贴近真实。

### Agent 4：报价分析师（Pricer）

【点击第四个Agent】

**使命**：基于实施计划，计算合理报价。

**核心心智**：商务负责人——会权衡成本、竞品价格、客户支付能力。

**关键设计**：输出**3档报价**——保守、标准、激进。每档对应不同的项目范围和价格。这是销售实战中最有用的形式——**给客户选择，引导成交**。

### Agent 5：合规审查员（Compliance）

【点击第五个Agent】

**使命**：审查整个方案的合规性、合理性、潜在风险。

**这是最重要的Agent——它有"否决权"**。

**关键设计**：它不是Pipeline中的"下一站"，而是**作为质量门审查所有前面环节的输出**。这就引入了Debate模式——它和前面4个Agent是"对抗"关系，专门挑刺。

**核心职责**：
- 合同条款审查（付款、违约、知识产权、数据）
- 方案合规审查（数据保护法、行业监管）
- 报价合理性审查（异常高/低折扣预警）
- 实施风险审查（人天估算是否合理）

**输出**：通过/有条件通过/不通过的判定。

**实测效果**：这个Compliance Agent上线后，**避免了3次本来可能签下但有重大风险的项目**——其中一次涉及客户行业的数据合规问题，前面4个Agent都没发现，是Compliance Agent拦下来的。

### 5个Agent协作的整体流程

整体流程是这样：

1. **Researcher** 输出需求清单
2. **Architect** 基于需求清单输出方案
3. **Planner** 基于方案输出实施计划
4. **Pricer** 基于实施计划输出报价
5. **Compliance** 审查所有前面的输出

任何一个环节如果Compliance打回，**回到对应Agent修改**。这就形成了一个**主流是Pipeline，关键节点有Debate**的混合架构。

### 这种设计的核心价值

**专业化** + **流水线效率** + **质量审查**。

每个Agent的Prompt专精一个领域，可以做到很高的水平。他们之间的协作有明确规则，不会互相干扰。Compliance Agent作为质量门，保证整体输出可靠。

这就是企业级多Agent系统的真实形态——**不是炫技，是把组织协作的成熟模式映射到AI系统里**。

## 第四部分：3个主流框架（4分钟）

【滚动到第四节框架对比】

讲了这么多设计，最后说说**怎么实现**。

主流多Agent框架有3个：

### LangGraph（基于状态机）

**核心抽象**：状态图。Agent之间的流转用图的边定义，状态在图里流动。

**特点**：控制力强、可视化好、支持条件分支和循环。

**适合**：复杂业务流程、需要严格控制路径的生产系统。

**入门门槛**：中等——需要理解状态机概念。

### CrewAI（基于角色）

**核心抽象**："角色 + 任务"。每个Agent定义角色背景和擅长领域，框架自动协作完成任务。

**特点**：概念直观、原型快、最像"组团队"。

**适合**：原型验证、内容生产、调研类任务。

**入门门槛**：最低——几行配置就能跑起来。

### AutoGen（基于对话）

**核心抽象**：Agent之间通过对话协作。可以提问、回答、补充。支持人类作为其中一个"Agent"加入。

**特点**：交互灵活、支持人在回路。

**适合**：需要人介入的业务、研讨型任务。

**入门门槛**：中等——需要设计对话规则。

### 实战建议

**99%的项目不需要从框架入手**。

我看到很多团队上来就纠结"用LangGraph还是CrewAI"——这是新人最大的坑。

**正确的路径是**：
1. 先用最基础的"几个Agent + 一段调度代码"原型验证业务流程是否合理
2. 流程稳定后，发现协作复杂度高了，再引入框架

**框架解决的是"协作复杂度"问题，业务还没跑通时不需要**。过早引入框架会让你陷在"框架抽象"里，看不清业务本质。

## 第五部分：6个高频陷阱（4分钟）

【滚动到第五节陷阱】

多Agent比单Agent复杂10倍，坑也多10倍。说说我们踩过的6个真实坑。

### 陷阱1：为了多Agent而多Agent（过度设计）

一个简单任务被拆成5个Agent协作，**协作overhead高于一个Agent直接做**。

**判断标准**：能用单Agent + 不同的Prompt切换搞定的，绝不用多Agent。**Agent数量不应超过真实业务角色数**。

### 陷阱2：角色定义模糊

"需求分析师"和"方案架构师"职责重叠，**每个Agent都做了一部分，每个又都没做完整**。最后整合的内容互相矛盾。

**修正**：用 **RACI矩阵** 明确每个Agent的职责。每件事必须有且只有一个Agent是Responsible。这是项目管理的经典工具，在多Agent设计里同样适用。

### 陷阱3：上下文爆炸

Agent A的输出全部塞给Agent B，B又把所有东西塞给C…**到最后一个Agent，context已经是10万token的"大杂烩"**。性能急剧下降。

**修正**：
- Agent间传递的是 **结构化结论** 而非原始对话历史
- 关键信息走 "共享Memory"，需要的Agent去查
- 给Agent定义明确的 "输出schema"，不允许传废话

### 陷阱4：错误传播

需求分析Agent提取错了一个关键字段，**后面所有Agent都基于错的需求做事**。

**修正**：
- 关键节点设"质量门"——必须通过验证才进入下一环
- 引入Reviewer Agent检查每一步输出
- 重要决策让用户确认

### 陷阱5：无限互踢

Agent A需要B的结果，B说"先问C"，C说"先问A"——**三个Agent在循环互推**。

**修正**：
- 单次任务设最大轮数（如10轮）
- 设"打破僵局"规则——超时由Supervisor仲裁
- Agent间通信图必须是DAG（有向无环图）

### 陷阱6：调试难度爆炸

5个Agent协作输出的结果质量差。**查问题时不知道是哪个Agent的锅**。每个Agent单测都OK，但合在一起就崩。

**修正**：
- 全链路追踪日志（每个Agent的输入/输出/思考都记录）
- 每个Agent独立可replay
- 引入LangSmith / Langfuse等观测工具

**这个陷阱直接引出了下一讲的主题——评估与可观测性**。多Agent系统不可观测，就没法持续优化。

## 第六部分：5问决策框架（2分钟）

【滚动到第六节决策框架】

最后给一个**实用决策框架**——面对一个新业务场景，要不要用多Agent？按这5个问题问一遍：

**Q1**：业务流程涉及多少个真实角色？
- 1-2个 → 单Agent + Prompt切换
- 3+个 → 多Agent

**Q2**：不同步骤需要的"专业心智"差异有多大？
- 差异小 → 单Agent
- 差异大 → 多Agent

**Q3**：是否需要"质疑/对抗"机制？
- 不需要 → 单Agent或Pipeline
- 需要 → Debate模式

**Q4**：流程是固定的还是动态的？
- 固定 → Pipeline
- 动态 → Supervisor

**Q5**：业务流程是否已经稳定？
- 不稳定 → 先用单Agent验证
- 稳定 → 再考虑多Agent架构

**这5个问题中有3个以上答"是多Agent方向"，再上多Agent。否则先用单Agent**。

## 收尾（1分钟）

【滚动到总结】

把今天的核心收束成一句话：

> **单Agent是个全能员工；多Agent是个有分工的团队。团队的效率不在每个人多强，在协作设计有多好。**

回顾今天的核心：

**4种协作模式**——Supervisor、Pipeline、Debate、Hierarchical——对应不同业务场景。**真实项目里往往是这几种的混合**。

**5-Agent CRM团队**展示了企业级多Agent的真实形态——专业化分工 + Pipeline效率 + Debate审查。

**6个高频陷阱**——过度设计、角色模糊、上下文爆炸、错误传播、无限互踢、调试爆炸——都是真实踩过的。

**对我们的意义**：CRM实施项目本身就是多角色协作——售前、产品、技术、合规、交付。把这种协作模式映射到AI系统里，能**真正自动化复杂的业务流程**。

但**多Agent是高级武器**。在没有把单Agent做扎实之前，别盲目上多Agent。**先单后多，先简后繁，是工程纪律**。

下一讲我们解决一个所有Agent系统都会遇到的问题——**怎么知道它跑得好不好**？这就是"评估与可观测性"，工业级AI系统的"基础设施"。

## 备注：常见疑问

**Q: 多Agent成本是不是单Agent的好几倍？**
A: 是的。每个Agent都会调用LLM，5个Agent的Token消耗大约是单Agent的3-5倍（不是5倍，因为有些Agent处理的是子集信息）。但产出质量和可维护性的提升通常值得。**关键场景才用多Agent，日常场景单Agent即可**。

**Q: 多Agent的延迟会不会很高？**
A: Pipeline模式延迟最高（串行）。Supervisor模式如果子Agent能并行调用，延迟可控。**实战经验**：用户能接受的延迟通常是10-30秒，超出后体感很差。设计时要把延迟作为约束。

**Q: Agent之间用JSON还是自然语言通信？**
A: **关键数据用JSON**（结构化、可校验、不易错），**思考过程用自然语言**（保留语境）。两者结合是工业级做法。

**Q: 我们团队没用过LangGraph这些框架，能直接做多Agent吗？**
A: 完全可以。**前期用Python函数调用 + 简单的状态管理就够了**。等业务复杂度真的需要框架时，再迁移过去——你的Agent定义和业务逻辑都是可复用的。

**Q: 单个Agent该用多大的模型？多个Agent要不要混用？**
A: **混用是常态**。简单的Researcher可以用Haiku/Qwen-7B，关键的Compliance要用Opus/GPT-4。**按"任务复杂度"分配模型梯队**——这是上一讲第7讲讲过的"分层调度"思路。

**Q: 多Agent系统怎么做版本控制？一个Agent改了会影响其他吗？**
A: **每个Agent的Prompt独立版本管理**（像微服务一样）。改一个Agent要跑全链路回归测试，确保没破坏其他Agent的输入预期。这就涉及下一讲的评估体系。
