## 开场（2分钟）

我们进入第三部分——**工程落地**。前面8讲讲了AI是什么、大模型怎么炼成。从今天开始，我们换一个角度：**怎么用这些模型做出真正的产品**。

我先抛个问题——**你觉得"Prompt写得好"是什么样的能力？**

我猜大家的第一反应都是"会问问题"、"会描述需求"、"懂一些技巧"。

但我想说一句可能让大家不舒服的话——**这种理解，差不多就是把Prompt Engineering当成了"和ChatGPT聊天的话术"**。

## 第一部分：先打掉三个错误认知（5分钟）

【看第一节的误解对照】

我做团队调研的时候，发现大家对Prompt有三个普遍的误解。先把它们说清楚，否则后面的内容会被旧认知带偏。

### 误解1：Prompt就是问问题，越口语越好

很多人把"和ChatGPT聊天"等同于Prompt Engineering。这是个根本性的混淆。

**聊天是探索，工程是生产**。

你在ChatGPT里问"帮我写个总结"，它给你一段，你说"短一点"，它再给你一段——这叫**交互式探索**。你在和它一起摸索。

但产品里的Prompt不能这么做。**用户每次输入不一样，但Prompt是固定的**。它必须：
- 稳定输出（同样输入要有同样质量的输出）
- 能被代码解析（结构化、可程序化处理）
- 处理边界情况（输入异常、信息不足、内容跑偏）

**自然提问解决不了这些问题**。

### 误解2：Prompt写得越长越详细越好

我见过有人写一个4000字的Prompt模板，从"你是一个温柔耐心专业的助手"开始，事无巨细。

**实际上，Prompt过长会稀释关键指令**。

模型的注意力是有限的。你写3000字的细节，关键的"要返回JSON"那句指令就被淹没了。模型在回答时可能完全忽视这个核心要求。

好的Prompt是**"必要且充分"的艺术**——把核心指令放在最显眼位置，细节用例子传达，边界情况单独说明。**一个20行的精准Prompt往往胜过200行的糊涂Prompt**。

### 误解3：换更强的模型就能解决Prompt问题

我经常听到这句话——"现在Prompt不行，等GPT-5出来就好了"。

**模型升级不会自动解决糟糕Prompt的问题**。

垃圾Prompt遇到强模型，得到的是"更精致的垃圾"——更流畅的废话、更礼貌的偏离、更巧妙的幻觉。

关键认知：**Prompt是模型能力的"放大器"**。
- 好Prompt能让中等模型表现接近顶级
- 差Prompt能让顶级模型表现像垃圾
- 模型升级让上限提高，但**下限永远由Prompt质量决定**

所以Prompt工程的ROI始终是最高的——花一周时间优化Prompt，效果可能比换一个贵5倍的模型还好。

## 第二部分：看真实的Prompt差异（10分钟）

【滚动到第二节Prompt实验台】

光说概念没用。我们看4个真实CRM场景的对比。

### 场景1：客户需求提取

【点击"客户需求提取"】

这是我们项目里最高频的任务。看左右两边。

**左边糟糕的Prompt**："帮我从这段访谈里提取需求"

模型输出：

> "根据访谈，客户的需求大致包括：1. 客户希望改进销售跟进 2. 客户提到了审批流程的问题 3. 还有一些数据同步方面的痛点。建议进一步沟通确认细节。"

**问题在哪？**这是"段落式建议"。它**没法直接进入WBS**，没法导入到方案文档，没法让客户confirm每条需求。本质上——**这只是个聊天回复，不是工程产物**。

**右边工程级的Prompt**：定义了角色、要求JSON输出、规定了字段（id/category/priority/source_quote）、给了异常处理规则。

模型输出是结构化的JSON：

```json
[
  {
    "id": "REQ-001",
    "description": "销售大客户长周期跟进自动提醒",
    "category": "业务流程类",
    "priority": "P0",
    "source_quote": "销售总是忘记跟进，特别是大客户跟进周期长"
  },
  ...
]
```

**这个输出可以**：
- 直接进我们的需求管理系统
- 每条需求有原话引用，可以让客户confirm
- 字段标准化，能做汇总统计
- 优先级支持后续排期

**这才是工程产物**。

### 场景2：合同条款审核

【点击"合同条款审核"】

左边："看看这份合同有没有问题"——模型给一段建议，没什么用。

右边明确了**5个审核维度**：付款节奏、验收标准、违约金、知识产权、数据安全。每个维度按【符合/有风险/缺失】判定，必须给原文依据。

输出是结构化JSON，可以直接对接合同管理系统的合规模块。每个有风险的条款都有改进建议——**这才是法务能用的东西**。

### 场景3：销售对话总结

【点击"销售对话总结"】

左边的输出我相信大家都见过——"销售与客户进行了沟通，整体氛围良好，客户表示会考虑"。

**这种总结一文不值**。它就是销售在CRM里被催着填的那种应付式跟进记录。

右边的Prompt强制提取了**销售管理真正关心的信息**：
- 客户基本信息（行业/规模/决策角色）
- 核心痛点（按重要度排序）
- 提到的竞品（关键情报！）
- 下一步行动（具体到时间和负责人）
- 销售本次表现的信号点和改进点

**销售对话总结的价值在于"能形成下一步行动"**。如果总结完了大家不知道下一步做啥，这个总结就白做了。

### 场景4：工单分类

【点击"工单分类"】

工单分类是要进**自动化路由系统**的——分对了就自动派给对应工程师，分错了就堵塞。

左边"建议归类到功能问题"——根本没法用。代码怎么解析"建议归类到"这个文字？

右边定义了完整的分类体系（A-H共8类）、紧急程度（P0-P3）、附加字段（关键词、是否需要人工、预估解决时长）。**输出可以直接驱动派单系统**。

### 一个共同的pattern

看完4个场景，注意一个共同pattern：

**糟糕Prompt → 段落式人话 → 不可程序化 → 工程价值低**

**工程级Prompt → 结构化输出 → 可解析 → 直接进生产系统**

这个差异背后是认知的差异——**你是在"和模型聊天"，还是"用模型做工程"**？

## 第三部分：6个核心原则（6分钟）

【滚动到第三节6个原则】

不是技巧大全。这6个原则是从大量生产实践中沉淀出来的，**覆盖了90%的Prompt工程场景**。

### 原则1：明确角色

告诉模型"你是谁"——"你是一个CRM需求分析师"比"帮我分析需求"效果好得多。

为什么？因为**角色会激活模型对应的能力组合**。模型在训练时见过大量"分析师写的报告"，给它分析师身份，它就调用对应的语言风格、思维方式、输出格式。

实践技巧：角色越具体越好。"资深CRM SaaS交付经理"比"AI助手"激活的能力更精准。

### 原则2：结构化任务

把复杂任务拆成有序步骤。**"先做A，再做B，最后做C"比"完成X"效果好得多**。

原因：模型像我们一样，**复杂任务一次做容易跳步**。把它拆成有序步骤，强制顺序执行，准确率显著提升。

实践技巧：3-5步最佳。少了等于没拆，多了模型记不住。

### 原则3：给出范例（Few-shot）

给2-3个"输入→输出"的范例。**模型从范例里学到的格式和风格，比文字描述精准10倍**。

为什么？因为大模型的本质是"模式匹配"。看到一个范例，它就知道你想要什么样的产物。这比你用1000字描述格式还有效。

实践技巧：范例要覆盖**典型case + 边界case**。只给典型case，模型遇到边界就不会处理。

### 原则4：指定输出格式

明确要JSON、Markdown还是纯文本。**生产系统几乎都要结构化输出**——没有指定格式 = 没有可解析性 = 集成失败。

最常用的是JSON。指定字段名、字段类型、可选值范围。最好附一个范例。

实践技巧：JSON输出时，**明确写"直接输出JSON，不要任何前后说明文字"**。否则模型会加"好的，以下是您要的JSON：" 这种话，让JSON.parse()失败。

### 原则5：边界与异常

告诉模型"如果X，怎么办"。

最经典的例子——你让模型从访谈里提取需求。如果访谈里压根没需求怎么办？

**没说的话——模型会编造**。它的训练目标让它倾向"乐于回答"，所以宁愿编一个也不愿说"没有"。

**修正方法**：明确给"逃生口"——"如果原文没有需求，输出空数组 []，不要编造"。

实践技巧：在范例里也展示这种异常情况，让模型看到"输出空数组也是合法的"。

### 原则6：让它思考（Chain of Thought）

复杂任务加一句"先逐步思考再给答案"。让模型显式推理，**准确率能提升20%-50%**。

这就是著名的**CoT（Chain of Thought）**。原理是——模型直接给答案时，没有"思考空间"；但如果让它先写思考过程，每个token都有机会修正前面的错误。

实践技巧：用 `<thinking>` 标签让模型先思考，再输出最终答案。代码里只解析最终答案部分。

## 第四部分：一个真实可用的模板（4分钟）

【滚动到第四节模板】

下面这个Prompt是我们项目里实际用的简化版，**6个原则全部体现**。

### 拆解给大家看

**最上面**——角色定义：

```
你是一名资深的CRM实施需求分析师，专门从客户访谈记录中提取系统化的需求项。
你的输出会直接进入项目WBS和方案文档，必须准确、可追溯。
```

注意"会直接进入项目WBS"这句——它告诉模型**这不是聊天，是生产**。模型会自动调高严谨性。

**接下来**——5步任务拆解：

```
1. 先逐句扫描，标记所有"客户表达的痛点或诉求"
2. 把同类诉求合并去重，保留最具体的表达
3. 为每条需求分类：业务流程类/数据管理类/集成类/报表类/其他
4. 评估优先级：P0(必须)/P1(应该)/P2(可以)
5. 标注每条需求的原话来源（用于客户confirm）
```

每一步都是显式的、可验证的。

**然后**——一组范例：

输入是一段真实风格的访谈片段，输出是完整JSON。**这一组范例做了三件事**：
- 演示字段结构（id、description、category、priority、source_quote）
- 演示分类标准（什么样的算业务流程类）
- 演示优先级判定（什么样的P0）

**接下来**——输出格式约束：

```
输出格式：严格的JSON数组，结构同上述范例。不要任何额外文字。
```

注意"不要任何额外文字"——这一句能避免80%的JSON解析失败。

**然后**——边界处理：

```
- 如果原文中是客户的疑问而非诉求，不要提取
- 如果无法判断优先级，标记为P2并加"needs_clarification": true
- 如果访谈记录过短或与CRM无关，输出空数组而不是编造
```

每一条都对应我们项目里真实遇到过的坑。

**最后**——思考引导：

```
先在 <thinking> 标签内逐句分析，再输出最终JSON。
```

这一句让准确率提升15%-20%。代价是输出长一些、慢一些，但生产环境绝对值得。

### 这个模板的复用价值

**你可以直接拿这个结构套到自己的场景**：
- 改"角色"——比如"销售对话分析师"
- 改"5步流程"——按你的具体任务来
- 改"范例"——给你领域的真实案例
- 改"分类体系"——按你的业务来
- 改"边界情况"——按你踩过的坑来

**结构是通用的，内容是领域的**。这就是工程化Prompt的复用方式。

## 第五部分：6个生产陷阱（3分钟）

【滚动到第五节陷阱】

最后讲6个生产环境里反复出现的陷阱。**这些我们项目里都踩过**。

### 陷阱1：否定式指令的失效

写"不要使用专业术语"——模型经常照样使用。

为什么？**否定指令需要模型先想"什么是专业术语"**，反而强化了关注。

**修正**：用肯定式——"使用普通用户能理解的日常用语，举例说明"。

### 陷阱2：JSON输出包含说明文字

模型经常在JSON前加一句"好的，这是您要的JSON："，让 `JSON.parse()` 失败。

**修正**：明确写"直接输出JSON，不要任何前后说明文字"。或者代码层用正则提取 `{...}`。

### 陷阱3：长文档的"中间遗忘"

喂1万字文档让模型总结，**它经常只用了开头和结尾，忽略中间**。这就是著名的"Lost in the Middle"现象。

**修正**：分段处理 + 显式指令"必须覆盖文档的每一段"，或用RAG拆分检索（下一讲讲）。

### 陷阱4：模型的过度配合幻觉

问"客户是否提到了预算"——**即使没提到，模型也倾向编一个**。因为它被训练成"乐于回答"。

**修正**：明确给"逃生口"——"如果原文没提及，输出'NOT_MENTIONED'"，并在范例里展示这种情况。

### 陷阱5：多次微调反而变差

为了修复一个边界case加一句指令，**结果原本好的case也变差了**。Prompt改一处影响全局。

**修正**：建立**测试集**——10-20个固定的输入输出对。每次改Prompt都跑一遍。这是Prompt工程化最核心的实践——**Prompt也要做回归测试**。

### 陷阱6：模型间的差异

GPT-4上调好的Prompt，**换到Claude或DeepSeek效果可能完全变样**。

**修正**：为每个模型单独维护Prompt版本。或者用"骨架（模型无关）+ 细节（模型特定）"的两层结构。

## 收尾（30秒）

【滚动到总结】

把今天的核心收束成一句话：

> **Prompt is the new code. The model is the new compiler.**

Prompt是新的代码，模型是新的编译器。

**实践建议**：把Prompt当代码管理。建立Prompt仓库、写测试用例、做版本对比、记录变更日志。**用工程的纪律对待Prompt，回报非常丰厚**。

但Prompt也有边界——**它擅长指挥模型怎么做，但无法给模型增加新知识**。

下一讲我们讲**RAG**——让模型用上你的私有知识库。这是大模型在企业落地的第二个核心问题。

## 备注：常见疑问

**Q: Prompt要写中文还是英文？**
A: 视模型而定。Claude、GPT-4对中英文都很好。开源模型（Qwen、DeepSeek）中文Prompt表现更好。**实测最重要**——同一个Prompt中英文跑一下，看哪个效果好。

**Q: System Prompt和User Prompt怎么分？**
A: System Prompt放"角色+原则+格式约束"——这些不变的部分。User Prompt放具体输入。这样可以缓存System部分（很多API支持），节省token。

**Q: Few-shot范例越多越好吗？**
A: 不是。一般2-3个就够。**范例多了会占用上下文长度，且边际效益递减**。除非边界case特别多。

**Q: 怎么知道Prompt改得是不是更好？**
A: 必须有**测试集**。建10-20个输入和期望输出的对，每次改完跑一遍，看通过率。没有测试集的Prompt优化都是凭感觉。

**Q: 思维链CoT会让响应慢，怎么权衡？**
A: 简单任务（分类、抽取关键字段）不用CoT。复杂任务（多步推理、需求拆解、方案设计）必须用——速度损失换来的准确率提升非常值得。
