LESSON 09 / 23

PROMPT IS NOT
A QUESTION.

Prompt Engineering 的本质
你以为Prompt是问问题。但真正的Prompt是一段控制程序—— 它告诉模型扮演什么角色、按什么步骤思考、用什么格式输出、遇到什么情况怎么处理。 理解这个本质，你会发现Prompt是软件工程，不是话术。
CHAPTER · 01
Prompt is NOT...
绝大多数人对Prompt的误解
在团队里调研过——大家对Prompt的理解差异很大。先把三个最普遍的错误认知说清楚， 避免后面的内容被旧认知带偏。
误解 #1

"Prompt就是问问题，越口语越好"

很多人把和ChatGPT聊天等同于Prompt Engineering。 但聊天是"探索"，工程是"生产"。 产品里的Prompt要稳定输出、能被代码解析、要处理边界情况—— 这些都不是"自然提问"能解决的。

真相

Prompt是一段配置驱动的程序

产品级Prompt有明确的结构：角色定义、任务说明、输入约束、 推理步骤、输出格式、异常处理。每一段都有目的，每一行都被反复测试过。 它的本质是用自然语言写程序。

误解 #2

"Prompt写得越长越详细越好"

看到一些"Prompt模板"长达几千字，事无巨细。 实际上Prompt过长会稀释关键指令， 模型注意力会分散到无关细节，反而影响核心任务的完成度。

真相

Prompt是"必要且充分"的艺术

好的Prompt是把核心指令放在最显眼位置， 细节通过例子（few-shot）传达，边界情况单独说明。 长度不是目标，"任务完成度"才是。 一个20行的精准Prompt往往胜过200行的糊涂Prompt。

误解 #3

"换更强的模型就能解决Prompt问题"

经常听到："现在Prompt不行，等GPT-5出来就好了"。 但模型升级不会自动解决糟糕Prompt的问题—— 垃圾Prompt遇到强模型，得到的是"更精致的垃圾"。

真相

Prompt是模型能力的"放大器"

好的Prompt能让中等模型表现接近顶级， 差的Prompt能让顶级模型表现像垃圾。模型升级让上限提高， 但下限永远由Prompt质量决定。 所以Prompt工程的ROI始终是最高的。

CHAPTER · 03
THE 6 PRINCIPLES
工程级Prompt的6个核心原则
不是技巧大全。这6个原则是从大量生产实践中沉淀出来的，覆盖了90%的Prompt工程场景。
01
明确角色
ROLE DEFINITION
告诉模型"你是谁"。"你是一个CRM需求分析师"比"帮我分析需求"效果好得多—— 角色会激活模型对应的能力组合。
02
结构化任务
TASK DECOMPOSITION
把复杂任务拆成有序步骤。"先做A，再做B，最后做C"比"完成X"效果好—— 强制模型按顺序思考，避免跳步。
03
给出范例
FEW-SHOT EXAMPLES
给2-3个"输入→输出"的范例。模型从范例里学到的格式和风格， 比文字描述精准10倍。
04
指定输出格式
OUTPUT FORMAT
明确要JSON、Markdown还是纯文本。生产系统几乎都要结构化输出—— 没有指定格式 = 没有可解析性 = 集成失败。
05
边界与异常
EDGE CASES
告诉模型"如果X，怎么办"。"如果信息不足，输出'NEED_MORE_INFO'" 比让模型瞎猜要好得多。
06
让它思考
CHAIN OF THOUGHT
复杂任务加一句"先逐步思考再给答案"。让模型显式推理， 准确率能提升20%-50%。这就是著名的CoT。
CHAPTER · 04
A REAL TEMPLATE
真实可用的CRM需求提取Prompt模板
下面这个Prompt是我们项目里实际用的简化版。把6个原则全部体现出来， 你可以直接复用到自己的场景。
USE CASE: 客户访谈需求提取 MODEL: Claude Sonnet / DeepSeek V3 TOKEN: ~600 
# 角色定义（原则1）
你是一名资深的CRM实施需求分析师，专门从客户访谈记录中提取系统化的需求项。
你的输出会直接进入项目WBS和方案文档，必须准确、可追溯。

# 任务说明 + 步骤拆解（原则2）
请按以下步骤处理访谈记录：
1. 先逐句扫描，标记所有"客户表达的痛点或诉求"
2. 把同类诉求合并去重，保留最具体的表达
3. 为每条需求分类：业务流程类 / 数据管理类 / 集成类 / 报表类 / 其他
4. 评估优先级：P0(必须)/P1(应该)/P2(可以)
5. 标注每条需求的原话来源（用于客户confirm）

# Few-shot 范例（原则3）
范例输入：
"我们销售总是忘记跟进，特别是大客户跟进周期长容易掉。
另外现在审批要走OA，签了合同还得回CRM录一遍，太麻烦了。"

范例输出：
[
  {
    "id": "REQ-001",
    "description": "销售跟进自动提醒，针对长周期大客户",
    "category": "业务流程类",
    "priority": "P0",
    "source_quote": "销售总是忘记跟进，特别是大客户"
  },
  {
    "id": "REQ-002",
    "description": "OA审批与CRM合同数据双向同步",
    "category": "集成类",
    "priority": "P0",
    "source_quote": "签了合同还得回CRM录一遍"
  }
]

# 输出格式（原则4）
输出格式：严格的JSON数组，结构同上述范例。不要任何额外文字。

# 边界与异常（原则5）
- 如果原文中是客户的疑问而非诉求（如"这个怎么做？"），不要提取
- 如果无法判断优先级，标记为P2并加备注field "needs_clarification": true
- 如果访谈记录过短或与CRM无关，输出空数组 [] 而不是编造

# 思考引导（原则6）
先在 <thinking> 标签内逐句分析，再输出最终JSON。

# 输入
访谈记录：
{interview_transcript}

这个Prompt的关键设计点：
① 角色"资深需求分析师"激活了模型的专业模式
② 5步流程强制模型不跳步、不漏项
③ 范例锁定了字段结构和风格
④ 严格JSON 让代码可以直接 JSON.parse()
⑤ 异常处理避免模型在边界情况下编造
⑥ <thinking>标签让模型先思考再输出，准确率显著提升

CHAPTER · 05
COMMON PITFALLS
生产环境里反复出现的陷阱
这些陷阱我们项目里都踩过。提前知道它们，能少走半年弯路。
01
否定式指令的失效

"不要使用专业术语"——模型经常照样使用。 因为否定指令需要模型先想"什么是专业术语"，反而强化了关注。

修正：用肯定式——"使用普通用户能理解的日常用语，举例说明"
02
JSON输出包含说明文字

模型经常在JSON前加一句"好的，这是您要的JSON："， 让代码解析失败。这是模型的"友好倾向"在作怪。

修正：明确写"直接输出JSON，不要任何前后说明文字"， 或在代码里用正则提取 {...}
03
长文档处理时的"中间遗忘"

喂给模型1万字文档，让它总结。模型经常只用了开头和结尾，忽略中间。 这是著名的"Lost in the Middle"现象。

修正：分段处理 + 显式指令"必须覆盖文档的每一段"， 或用RAG拆分检索（下一讲讲）
04
模型"过度配合"导致的幻觉

问模型"客户是否提到了预算"，即使没提到，模型也倾向编一个。 因为它被训练成"乐于回答"。

修正：明确给"逃生口"——"如果原文没提及，输出 'NOT_MENTIONED'"， 并在范例里展示这种情况
05
多次微调Prompt后效果反而变差

为了修复一个边界case加一句指令，结果原本好的case也变差了。 Prompt是个有副作用的系统，改一处影响全局。

修正：建立测试集（10-20个固定的输入输出对）， 每次改Prompt都跑一遍。这是Prompt工程化的核心实践
06
不同模型对同一Prompt表现差异巨大

在GPT-4上调好的Prompt，换到Claude或DeepSeek上效果可能完全变样。 因为不同模型的训练偏好不同。

修正：为每个使用的模型单独维护Prompt版本， 或者用模型无关的"骨架"+模型特定的"细节"两层结构

核心收束

Prompt Engineering不是"会聊天"，是用自然语言写程序。 它有结构、有规范、有测试、有版本控制——是软件工程的一种。

"Prompt is the new code.
The model is the new compiler."

实践建议：把Prompt当代码管理。建立Prompt仓库、写测试用例、 做版本对比、记录变更日志。用工程的纪律对待Prompt，回报会非常丰厚。

但Prompt也有边界。它擅长"指挥模型怎么做"， 但无法给模型增加新知识——下一讲我们讲RAG， 解决"如何让模型用上你的私有知识"这个问题。
