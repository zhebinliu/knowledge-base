From tools
to agents.

Agent：从单轮对话到自主任务
上一讲我们让模型学会了用工具。但模型还是被动的——你说一句它做一步。 Agent是更进一步的存在——给它一个目标，它自己规划、自己执行、自己反思、自己继续， 直到任务完成。从"会用工具"到"自主完成任务"，这是质的飞跃。
CHAPTER · 01
From tools to agents
从"会用工具"到"自主决策"的飞跃
Function Calling 让模型会用工具，但每次都是用户驱动的——你问一个问题， 它最多调用几个工具回答。Agent是目标驱动的——你给它一个目标， 它自己拆解、自己规划、自己执行、自己反思，直到达成目标或失败。
LEVEL 1 · 上一讲

Tool Use
单轮交互

模型每次响应一个用户请求。它可能调用1-3次工具，但本质上是"被动执行"。 流程结束时，结果交还给用户。

  • 用户驱动——每次行动都源于用户输入
  • 有限步骤——通常2-5步就结束
  • 无长期记忆——每次对话独立
  • 无自我反思——做错了就错了
  • 无目标拆解——只解决当前问题

LEVEL 2 · 今天

Agent
自主任务

给它一个高层目标，它自己拆解成子任务，依次执行， 中途检查进展、调整策略，直到完成或确认无法完成。

  • 目标驱动——围绕一个目标持续行动
  • 多步循环——可能跑十几步、几十步
  • 有工作记忆——记住中间结果
  • 会反思——失败后调整策略
  • 能拆解——把复杂任务分解成子任务

举个对比例子：你说"客户A本月业绩怎么样"—— Tool Use查一下数据告诉你结果，结束。 Agent查了数据，发现业绩异常下降，自己决定再深挖原因—— 查最近的客户互动记录、看是否有竞品介入的迹象、对比上月差异、最后给出一份完整的"业绩异动分析"。 差别是：Agent会超出你的提问范围去主动思考。

CHAPTER · 02
The ReAct loop
Agent的核心机制：思考-行动-观察循环
Agent不是黑盒。它的核心机制叫 ReAct——Reasoning + Acting。 每一步都是"想一想 → 做一做 → 看结果"的循环。 下面看一个真实Agent处理任务的全过程。
▼ 用户给Agent的目标
分析客户A本月销售下滑的原因，给出后续建议
T
Thought · 思考
分析当前情况
决定下一步
A
Action · 行动
调用工具
或生成回答
O
Observation · 观察
看工具返回
评估结果

ReAct的精髓：每一步Agent都"显式地"思考。 它不是闷头干活，而是先说"为什么要做这一步"，再做。这样我们能看到它的思路、能发现它哪里想错了、能干预或纠正它。 这种"思考可见"是Agent可信的基础。

CHAPTER · 03
The 4 components
构成Agent的4个组件
所有Agent都由4个组件构成。理解了这4个组件，你就能拆解任何Agent系统、设计自己的Agent。
01
大脑（LLM）
THE BRAIN
Agent的核心是一个LLM。它负责"思考"——理解目标、拆解任务、决定下一步、生成回答。 通常用强模型（GPT-4、Claude Opus），因为规划能力是涌现出来的，弱模型规划经常出错。
选型经验：复杂任务必须用Frontier梯队。中等任务可以用Strong（Claude Sonnet、DeepSeek V3）。简单的工具调度用Efficient也行。
02
工具集（Tools）
THE HANDS
Agent通过工具与世界交互——查询、计算、调用API、修改数据。这是上一讲的核心内容。 工具集合定义了Agent的能力边界——能力越大，工具越多。
设计原则：工具数量适中（10个左右），分类清晰（查询/动作/工作流），权限严格（按用户身份隔离）。
03
记忆（Memory）
THE MEMORY
Agent要能记住"自己做过什么"——上一步思考的内容、工具返回的结果、当前进展。 分两层：工作记忆（当前对话的上下文）和长期记忆（跨会话的知识，通常用RAG实现）。
典型问题：上下文太长会被截断、超过窗口长度会丢信息。需要"记忆压缩"或分层记忆策略。
04
规划（Planner）
THE STRATEGY
规划是Agent的灵魂——把高层目标拆解成可执行的子任务。 有两种范式：显式规划（先列计划再执行）和隐式规划（边走边想，ReAct）。 实际项目里两种都用。
选择策略：任务清晰可预知 → 显式规划；任务模糊需探索 → 隐式规划（ReAct）；复杂项目可以混合——先做粗略计划，每个子任务用ReAct执行。

CRM Agent in practice
CRM场景下的Agent设计：5种角色
在我们的CRM实施场景里，Agent可以替代或辅助多种角色。下面是5种已经在落地的Agent设计。 每种Agent都有特定的目标、工具集和典型任务。
销售助手SALES AGENT
目标：帮销售提升业绩、降低无效工作

处理客户跟进提醒、自动生成跟进话术、分析客户行为模式、识别有价值的线索。 典型任务："这周该跟进哪些客户？给出优先级和具体话术。"

search_customer get_interaction_history analyze_engagement generate_followup_script create_task
实施顾问CONSULTANT AGENT
目标：辅助交付经理做需求分析和方案设计

从客户访谈中提取需求、关联历史相似项目、生成方案初稿、识别交付风险。 典型任务："基于这次访谈记录，给我一份初版需求清单和方案大纲。"

extract_requirements search_similar_projects generate_solution_draft identify_risks estimate_effort
客服AgentSUPPORT AGENT
目标：处理一线客户咨询，过滤简单问题

理解客户问题、检索知识库、回答简单问题、识别复杂问题转人工、记录工单。 典型任务："客户问'怎么导出报表'——回答+如果无法解决就开工单。"

kb_search get_user_context generate_answer create_ticket escalate_to_human
数据分析师ANALYST AGENT
目标：自助回答业务问题、生成报表

理解自然语言查询、生成SQL、运行查询、解读结果、给出业务洞察。 典型任务："上周华南区销售环比下降了多少？是哪些销售导致的？"

nl_to_sql execute_query generate_chart compare_periods explain_anomaly
运营经理OPS AGENT
目标：自动化日常运营任务、监控异常

监控系统状态、识别异常订单/客户、执行例行清理、生成日报、提醒关键事件。 典型任务："每天上午9点自动检查昨日订单，找出异常并生成处理建议。"

scan_orders detect_anomaly send_alert generate_daily_report auto_cleanup

设计经验：每个Agent都对应一个真实角色，目标明确、工具有限、边界清晰。 不要做"全能Agent"——目标越宽泛，效果越差。 "一个Agent做一件事，做好它"，多个Agent的协作我们下一讲讲。

CHAPTER · 06
The limits
Agent今天还做不好的5件事
Agent很强，但不是万能。知道它的边界，才能正确使用、避免踩坑。 这5个限制是我们项目里反复遇到的——也是Agent技术未来要突破的方向。
01
长程任务的"目标漂移"

执行超过20-30步的任务时，Agent经常慢慢偏离原始目标——开始查A，中途发现B，又去研究C，最后忘了用户问的是A。 这是大模型上下文管理的根本问题——越往后，原始目标在context里越远，影响越弱。

02
反思能力有限

Agent能基于"工具返回结果"调整下一步——但很难真正反思自己的策略错误。 它会一遍遍尝试相同的失败路径，因为它意识不到"这条路根本走不通"。 需要外部规则强制中断、人工干预，纯靠模型自己反思往往不够。

03
成本和延迟

Agent做一个任务可能调用LLM 10-30次，每次都消耗token。一次复杂任务可能花几块到几十块。 而且每次LLM调用都要等几秒——总耗时可能几分钟。 不适合实时交互场景，更适合"后台跑、稍后看"。

04
不可预测性

Agent每次执行同一个任务，路径可能不同——选不同的工具、走不同的分支。 这让测试和质量保证非常困难。 传统软件可以单元测试，Agent只能用"端到端评估"+"大量样本"——这是一门新学问，下下讲讲评估时会展开。

05
很难解释失败

Agent失败了——是模型不够强？工具不够好？目标太模糊？还是某一步思路错了？ 排错是个噩梦。需要完整的trace日志、可视化工具、人工审查。 "可观测性"是生产级Agent的硬指标——下下讲会专门讲这件事。

RECAP

From chat to autonomous task

今天我们走完了从"会聊天"到"能自主完成任务"的第一段路。Agent是大模型从"工具"进化到"同事"的关键一步。

"Agent是给目标，不是给步骤。
你说'帮我做',它自己想'怎么做'。"

4个核心组件构成所有Agent——大脑（LLM）、工具（Tools）、记忆（Memory）、规划（Planner）。 ReAct循环是Agent最核心的工作机制——思考-行动-观察，循环直到目标达成。

能力光谱5个等级——从纯Chat到Multi-Agent，逐级放权。 生产环境里大部分应用选L2（需要确认的Agent）最稳——既享受了Agent的智能，又保留了人类控制。 L3完全自主用在低风险高频场景；L4多Agent是下一讲的主角。

但Agent还有5个明显的局限——长程任务漂移、反思有限、成本高、不可预测、难解释。 正确的姿势是：用Agent处理它擅长的事，人类把握它弱的地方。 不要相信"完全替代人类"的鬼话，至少今天不行。

下一讲——多Agent系统。当一个Agent处理不了复杂任务时，我们让多个Agent协作。 这是当前Agent领域最前沿、也最有争议的话题。
