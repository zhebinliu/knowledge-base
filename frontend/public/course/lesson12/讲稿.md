## 开场（2分钟）

第三部分第四讲。前面我们讲了：

- 第9讲 **Prompt**——让模型会说
- 第10讲 **RAG**——让模型知道
- 第11讲 **Function Calling**——让模型能做

今天讲一个让"做"上一个台阶的东西——**Agent**。

我先问个问题——**Function Calling 和 Agent 的本质差异是什么？**

很多人觉得"Agent就是会调工具的模型"。这个理解不准确。

差异在于——**谁在驱动**。

**Function Calling**：用户驱动。你说一句，它做一步。"帮我查客户A本月业绩" → 调用工具 → 返回结果。结束。

**Agent**：目标驱动。你给一个目标，**它自己规划、自己执行、自己反思、自己继续**，直到目标达成。"分析客户A本月销售下滑的原因，给出后续建议" → 它自己决定要查哪些数据、看哪些维度、关联哪些信息、最后给出完整分析。

## 第一部分：从工具到Agent的飞跃（5分钟）

【看第一节的对照】

我们在demo里做了个很直观的对比。

**Level 1 · Tool Use**——上一讲的内容：
- 用户驱动——每次行动都源于用户输入
- 有限步骤——通常2-5步就结束
- 无长期记忆——每次对话独立
- 无自我反思——做错了就错了
- 无目标拆解——只解决当前问题

**Level 2 · Agent**——今天的主角：
- 目标驱动——围绕一个目标持续行动
- 多步循环——可能跑十几步、几十步
- 有工作记忆——记住中间结果
- 会反思——失败后调整策略
- 能拆解——把复杂任务分解成子任务

### 用一个例子感受差异

【看下面的橙色提示框】

我们项目里真实发生过——

你说"客户A本月业绩怎么样？"

**Tool Use的反应**：调用 `get_order_history`，返回数据，告诉你"客户A本月3单，97.5万"。**结束**。

**Agent的反应**：查了数据，发现下滑45%——异常。**它自己决定再深挖**——查最近的客户互动记录、看是否有竞品介入的迹象、对比上月差异、识别决策人变更——最后给出一份完整的"销售下滑根因分析"报告。

**差别在哪？**

Agent会**超出你的提问范围去主动思考**。它把"业绩怎么样"理解成"你想知道业绩状态以及背后的原因"——并主动满足这个深层需求。

这是Agent的本质——**理解目标、不止于回答问题**。

## 第二部分：ReAct循环——Agent的核心机制（10分钟）

【滚动到第二节深色背景的ReAct演示】

Agent不是黑盒。它有非常明确的工作机制——叫 **ReAct**。

**ReAct = Reasoning + Acting**——思考 + 行动。

### 三步循环

每一次迭代都是三步：

**Thought（思考）**——分析当前情况，决定下一步该做什么。  
**Action（行动）**——调用工具或生成回答。  
**Observation（观察）**——看工具返回的结果，评估是否达成目标。

这三步循环往复，直到Agent判断"目标完成"或"无法继续"。

### 看演示

【看那个动画——"分析客户A本月销售下滑的原因"】

我让动画自动跑，大家盯着看。

**Iteration 1**：
- **Thought**：用户要分析销售下滑原因。我需要先查到客户A本月的实际数据，对比历史，看下滑幅度。
- **Action**：调用 `get_order_history`
- **Observation**：本月3单97.5万，上月5单180万，下滑45%。

**Iteration 2**：
- **Thought**：下滑45%是显著异常。我需要排查可能原因——客户互动减少？竞品介入？决策人变更？先查互动记录。
- **Action**：调用 `get_interaction_history`
- **Observation**：互动减少67%，发现"主决策人王总于4月2日离职"。

**Iteration 3**：
- **Thought**：关键线索！决策人离职是销售下滑的可能根源。我需要确认新决策人是谁、关系建立情况。
- **Action**：调用 `search_customer_contacts`
- **Observation**：当前主联系人变更为李总（4月10日）。但我们公司还没有人和李总直接对接过。

**Iteration 4**：
- **Thought**：原因清晰了：决策人变更导致关系断层。建议主动建立和新决策人的关系。我应该给销售生成一份具体的行动计划。
- **Action**：生成最终分析报告

**最终输出**：完整的根因分析 + 4条具体建议 + 已自动创建的跟进任务。

### ReAct的精髓

注意一个关键设计——**每一步Agent都"显式地"思考**。

它不是闷头干活，**先说"为什么要做这一步"，再做**。这样我们能看到它的思路、能发现它哪里想错了、能干预或纠正它。

**这种"思考可见"是Agent可信的基础**。如果Agent是黑盒——你只能看到它最终的结论，不知道它怎么得出的——那它的可靠性极难保证。

ReAct的"先说再做"，是把Agent变成**可监督、可调试、可干预**的系统。

### 第二个例子

【点击"线索分配优化"】

这个例子更展示Agent的"主动思考"——

用户只说"本月销售线索分配不均，调研问题并提出优化方案"。Agent自己的推进路径：

1. 先量化——具体每个销售拿到多少线索
2. 看分配规则——是制度问题还是执行问题？
3. 看转化率——分配多的销售是不是转化也高？
4. 发现关键反直觉——分配多的销售转化反而低
5. 给出具体的规则调整方案

**没人告诉Agent要做这5步**。它根据每一步的发现，**自己决定下一步**。

这就是ReAct的力量——**让Agent能够"边做边想"，而不是按预设流程走**。

## 第三部分：Agent的4个组件（4分钟）

【滚动到第三节4个组件】

所有Agent都由4个组件构成。理解了这4个，你就能拆解任何Agent系统、设计自己的Agent。

### 组件1：大脑（LLM）

Agent的核心是一个LLM。它负责**思考**——理解目标、拆解任务、决定下一步、生成回答。

**选型经验**：
- 复杂任务（多步推理、规划）必须用Frontier梯队（GPT-4、Claude Opus）
- 中等任务（标准流程）可以用Strong（Claude Sonnet、DeepSeek V3）
- 简单工具调度用Efficient也行（Haiku、Qwen 7B）

**为什么选Frontier更稳？**——因为规划能力是涌现出来的（第7讲讲过）。弱模型规划经常出错——目标拆解不完整、工具选错、循环失控。

### 组件2：工具集（Tools）

Agent通过工具与世界交互——查询、计算、调用API、修改数据。这是上一讲的核心。

**工具集合定义了Agent的能力边界**——能做什么不能做什么，全看工具有没有。

设计原则：
- 工具数量适中（10个左右最佳）
- 分类清晰（查询/动作/工作流）
- 权限严格（按用户身份隔离）

### 组件3：记忆（Memory）

Agent要能记住"自己做过什么"——上一步思考的内容、工具返回的结果、当前进展。

分两层：
- **工作记忆**：当前对话的上下文（在context window里）
- **长期记忆**：跨会话的知识（通常用RAG实现）

**典型问题**：上下文太长会被截断、超过窗口长度会丢信息。需要"记忆压缩"或分层记忆策略。

### 组件4：规划（Planner）

规划是Agent的**灵魂**——把高层目标拆解成可执行的子任务。

有两种范式：

**显式规划**：先列计划再执行。  
"任务A → 任务B → 任务C → 完成"

**隐式规划**（ReAct）：边走边想。  
"做一步 → 看结果 → 决定下一步"

**实际项目里两种都用**：
- 任务清晰可预知 → 显式规划
- 任务模糊需探索 → 隐式规划（ReAct）
- 复杂项目混合——先做粗略计划，每个子任务用ReAct执行

### 4个组件的协作

打个比方——**Agent就像一个员工**：
- **大脑**是这个员工的智力水平
- **工具**是给他的工作设备
- **记忆**是他的笔记本和工作经验
- **规划**是他做事情的方法论

少了哪个都不行。一个有智力但没工具的员工干不了活；有工具但智力不够的员工容易做错；没记忆的员工每次都从头开始；没规划的员工只会瞎忙。

## 第四部分：Agent能力光谱（5分钟）

【滚动到第四节浅黄色背景的能力光谱】

Agent不是非黑即白的。**存在5个能力等级**。

我做了个滑动条让大家直观感受——从L0到L4，逐级放权。

【点击L0】

**L0 · 无Agent · 纯Chat**

用户问一句，模型答一句。**没有工具、没有记忆、没有规划**。这是最早的ChatGPT形态。

适用：通用问答、闲聊、写作辅助、解释概念。

【点击L1】

**L1 · Copilot · 辅助但不执行**

模型给建议、生成草稿、解释问题——**但所有动作都由人类执行**。AI只在旁边"建议"。

典型例子：GitHub Copilot——AI建议代码，开发者决定是否采纳。

适用：写作辅助、代码补全、邮件草稿、方案修改。

【点击L2】

**L2 · Agent · 自主执行 + 人工确认**

Agent自己规划、自己执行工具调用——**但敏感动作需要人类确认**。

**这是大部分企业Agent的最佳位置**。

我重复一遍——L2是大部分企业Agent的最佳位置。**既享受了Agent的智能，又保留了人类控制**。

适用：客户跟进、报告生成、数据查询、低风险操作。

典型场景：销售助手生成跟进话术并草拟邮件，但发送前要销售点击确认。客户问题分析自动出报告，但建议执行前要主管批准。

【点击L3】

**L3 · Autonomous · 完全自主 + 事后审计**

Agent自主完成任务，**不需要人工确认**，但所有操作记日志，事后可审计。

适用：低风险高频任务——日报生成、监控告警、数据清理、自动分类。

**谨慎用于涉及钱、权限、对外沟通的场景**。一旦出错没有人工把关，影响会很大。

【点击L4】

**L4 · Multi-Agent · 多Agent协作**

多个专业Agent协作完成复杂任务——一个Agent当"项目经理"，分派任务给"专家Agent"，最后汇总结果。

**这是下一讲的主角**——多Agent系统。当前Agent领域最前沿、也最有争议的话题。

### 选择策略

**怎么选L几？**两个维度——

**风险维度**：操作出错的影响有多大？
- 影响小（查询、生成内容）→ 可以用L3
- 影响中（修改数据、发邮件）→ 用L2
- 影响大（涉及钱、对外、权限）→ 必须L1或L2

**复杂度维度**：任务多复杂？
- 单步操作 → L1或简单的L2
- 多步推理 → L2或L3
- 跨领域协作 → L4

我们项目里大部分Agent都在**L2**——既能干活，又有控制点。**做高级别的Agent要慎重**——每往上一级，出问题的代价都成倍增加。

## 第五部分：CRM场景的5种Agent（5分钟）

【滚动到第五节CRM Agent设计】

理论讲完了，看落地。在我们的CRM实施场景里，Agent可以替代或辅助多种角色。下面是5种**已经在落地的Agent设计**。

### Agent 1：销售助手（Sales Agent）

**目标**：帮销售提升业绩、降低无效工作。

**能力**：处理客户跟进提醒、自动生成跟进话术、分析客户行为模式、识别有价值的线索。

**工具集**：search_customer / get_interaction_history / analyze_engagement / generate_followup_script / create_task

**典型任务**："这周该跟进哪些客户？给出优先级和具体话术。"——Agent自己分析所有客户的状态、给出排序和针对性话术。

### Agent 2：实施顾问（Consultant Agent）

**目标**：辅助交付经理做需求分析和方案设计。

**能力**：从客户访谈中提取需求、关联历史相似项目、生成方案初稿、识别交付风险。

**工具集**：extract_requirements / search_similar_projects / generate_solution_draft / identify_risks / estimate_effort

**典型任务**："基于这次访谈记录，给我一份初版需求清单和方案大纲。"——Agent自动提取需求、找相似项目、生成方案、估工作量。

**这个就是我们在做的事情**。

### Agent 3：客服Agent（Support Agent）

**目标**：处理一线客户咨询，过滤简单问题。

**能力**：理解客户问题、检索知识库、回答简单问题、识别复杂问题转人工、记录工单。

**工具集**：kb_search / get_user_context / generate_answer / create_ticket / escalate_to_human

**典型任务**：客户问"怎么导出报表"——Agent检索知识库回答 + 如果客户继续追问且无法解决，自动开工单转人工。

### Agent 4：数据分析师（Analyst Agent）

**目标**：自助回答业务问题、生成报表。

**能力**：理解自然语言查询、生成SQL、运行查询、解读结果、给出业务洞察。

**工具集**：nl_to_sql / execute_query / generate_chart / compare_periods / explain_anomaly

**典型任务**："上周华南区销售环比下降了多少？是哪些销售导致的？"——Agent自动写SQL、查数据、对比分析、给出归因。

### Agent 5：运营经理（Ops Agent）

**目标**：自动化日常运营任务、监控异常。

**能力**：监控系统状态、识别异常订单/客户、执行例行清理、生成日报、提醒关键事件。

**工具集**：scan_orders / detect_anomaly / send_alert / generate_daily_report / auto_cleanup

**典型任务**："每天上午9点自动检查昨日订单，找出异常并生成处理建议。"——Agent按定时任务跑，每天自动产出运营简报。

### 设计原则：一个Agent做一件事

把5个Agent放一起，注意一个共同点——**每个Agent都对应一个真实角色**。

目标明确、工具有限、边界清晰。

**不要做"全能Agent"**。

我见过很多项目想做"超级AI助手"——什么都能干。结果是什么都干不好——目标越宽泛，效果越差。

正确的姿势：**"一个Agent做一件事，做好它"**。

需要多种能力时，就用**多个Agent协作**——这是下一讲的主题。

## 第六部分：Agent今天还做不好的事（4分钟）

【滚动到第六节深色背景的限制】

Agent很强，但不是万能。**知道它的边界**，才能正确使用、避免踩坑。

这5个限制是我们项目里反复遇到的——也是Agent技术未来要突破的方向。

### 限制1：长程任务的"目标漂移"

执行超过20-30步的任务时，Agent经常**慢慢偏离原始目标**——开始查A，中途发现B，又去研究C，最后忘了用户问的是A。

为什么？因为大模型上下文管理的根本问题——**越往后，原始目标在context里越远，影响越弱**。

**实践应对**：在关键节点加"目标提醒"——每5步重申一次目标。或者用分层规划——主Agent盯目标，子Agent干活。

### 限制2：反思能力有限

Agent能基于"工具返回结果"调整下一步——但**很难真正反思自己的策略错误**。

它会一遍遍尝试相同的失败路径，因为它意识不到"这条路根本走不通"。

**实践应对**：需要**外部规则强制中断**——同一工具同一参数失败超过2次就强制换路。**纯靠模型自己反思往往不够**。

### 限制3：成本和延迟

Agent做一个任务可能调用LLM 10-30次。**一次复杂任务可能花几块到几十块**。

而且每次LLM调用都要等几秒——总耗时可能几分钟。

**实践应对**：**不适合实时交互场景**。Agent更适合"后台跑、稍后看"的场景——晚上跑、第二天看结果。

### 限制4：不可预测性

Agent每次执行同一个任务，路径可能不同——选不同的工具、走不同的分支。

这让**测试和质量保证非常困难**。

传统软件可以单元测试，Agent只能用"端到端评估 + 大量样本"——这是一门新学问，**第14讲会专门讲评估**。

### 限制5：很难解释失败

Agent失败了——是模型不够强？工具不够好？目标太模糊？还是某一步思路错了？

**排错是个噩梦**。需要完整的trace日志、可视化工具、人工审查。

"可观测性"是生产级Agent的硬指标——**第14讲会专门讲这件事**。

### 这5个限制的共同信息

把5个限制串起来，有个共同信息——

**Agent今天的本质是"工具组合 + LLM大脑"**。它的能力上限受限于LLM的能力上限。

具体来说：
- LLM的上下文长度限制 → 长程任务漂移
- LLM的推理能力局限 → 反思能力有限
- LLM的token成本 → Agent成本高
- LLM的不确定性 → Agent不可预测

**Agent技术的进步，本质上跟着LLM的进步走**。GPT-5、Claude 4更强了，Agent的这些限制都会缓解。

但这不是说今天的Agent没用——**只要选对场景，今天的Agent已经能产生巨大价值**。我们项目就在用，效果显著。

关键是——**用Agent处理它擅长的事，人类把握它弱的地方**。

## 收尾（30秒）

【滚动到总结】

把今天的核心串一下——

**Agent是给目标，不是给步骤**。你说"帮我做"，它自己想"怎么做"。

**4个核心组件**——大脑（LLM）、工具（Tools）、记忆（Memory）、规划（Planner）。

**ReAct循环**——思考-行动-观察的循环，是Agent最核心的工作机制。

**能力光谱5级**——L0纯Chat到L4多Agent。**生产环境大部分应用选L2最稳**。

**5个明显的局限**——知道边界，正确使用。**不要相信"完全替代人类"的鬼话**——至少今天不行。

## 备注：常见疑问

**Q: Agent和RPA有什么区别？**
A: RPA（机器人流程自动化）是按预设流程执行的——你录了步骤，它一步步重放。Agent是**自主决策**的——它根据情况选择不同路径。RPA适合稳定的、可枚举的流程；Agent适合需要判断的、变化的场景。**好的方案往往是Agent + RPA混用**——Agent负责决策，调用RPA执行确定性任务。

**Q: AutoGPT、BabyAGI这些是什么？**
A: 2023年早期的Agent框架。它们证明了"完全自主Agent"的可能性，但实际效果差强人意——成本高、容易跑飞、结果不稳定。今天的主流框架是LangGraph、LlamaIndex Agents、CrewAI、AutoGen——更工程化、更可控。

**Q: 我们应该用什么框架做Agent？**
A: 不要一上来就用框架。**先用原生API（OpenAI/Anthropic的Function Calling）实现一个简单的ReAct循环**——你会理解Agent的本质。理解了之后再选框架——LangGraph适合复杂工作流、CrewAI适合多Agent场景、原生API适合简单需求。

**Q: Agent能完全替代人类吗？**
A: 短期不能（5年内）。Agent适合替代"重复性、规则清晰"的工作——填表、汇总、初步分析、模板化沟通。**人类的价值在判断、创意、复杂决策、关系维护**——这些Agent短期内做不好。**正确的姿势是Agent+人类协作，不是替代**。

**Q: Agent对我们项目的最大机会是什么？**
A: 我个人判断是**"实施顾问Agent"**——帮交付经理做需求分析和方案生成。这是我们核心业务，每个项目都需要，并且**有大量历史项目数据可以用**。如果Agent能把方案设计的初稿质量做到80%，每个项目能节省几个人天。年化下来是巨大的价值。
