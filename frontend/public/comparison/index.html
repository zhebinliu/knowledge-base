实施工作流 · 公平客观对比

用专属工作台,
还是个人 Agent + 公司级 Skill 库?

kb-system 这套工作台 vs「员工的 Claude / Cursor + 公司沉淀的 skill 库 (实施方法论 / 售前方案 / API 开发 / 研报...)」各自的强项和短板, 以及怎么组合才不亏。

定调:这不是二选一

两条路解决的不是同一个问题。把它们当对立面来比是错的。

A — 工作台

实施工作流的标准化与团队化资产沉淀

把"实施顾问的工作流程"写进系统:文档驱动洞察、顾问勾选式调研、会议 AI、纪要模板化、引用追溯、跨会议干系人合并、协作者权限…… 顾问点鼠标干活,团队 know-how 自动沉淀。

B — 个人 Agent + 公司 Skill 库

把公司 know-how 拆成原子 skill,每个员工的 Claude / Cursor 加载即用

Skill = markdown 描述 + 工具约定。比如「实施访谈 skill」「售前方案 skill」「需求分析 skill」「API 开发 skill」。员工开 Claude 调 skill,流程被引导但仍有灵活度。建设成本 = 写 markdown,部署成本 ≈ 0。

真正的问题

不是「选 A 还是选 B」,而是「哪些环节给 A,哪些环节给 B」。

本页给出 11 个维度对比表 + 3 个具体场景,以及作者自认为最尖锐的 实操建议。

什么是「公司级 Skill 库」

这是 B 路线的核心。把公司 know-how 从「员工脑子里」搬到「Claude 可调用的 skill 文件」。

实施类 skill

客户访谈 / 需求分析 / 调研问卷 / 项目复盘

把"顾问怎么访谈、怎么提需求、怎么分类优先级"写成 skill。员工接客户 → Claude 加载该 skill → 引导式访谈 → 输出标准化文档。

售前类 skill

客户画像 / 竞品分析 / 方案撰写 / 报价计算

把销售总监脑子里的"什么客户用什么方案"沉淀进 skill。销售前线接到机会 → Claude 用 skill 自动出方案初稿,销售只改 20%。

技术类 skill

API 开发 / 代码审查 / 系统架构 / 部署运维

把团队的代码规范、架构习惯、部署清单写进 skill。工程师 Cursor + skill 写代码 → 自动遵循公司风格,减少 review 来回。

研报类 skill

行业研究 / 客户调研报告 / 培训材料

把咨询师的研报结构、表达风格、引用规范沉淀。员工调 skill → Claude 按公司报告模板出稿。

关键差异

Skill 是「中间道路」 — 比通用 agent 标准化,比工作台轻得多

通用 agent(纯 Claude Code)= 自由发挥 → 质量随 prompt 浮动
Skill 化的个人 agent(B)= 结构化引导 + 灵活执行 → 建设成本 = markdown 文件 + 评估,部署 ≈ 0
工作台(A)= UI 严格固化 → schema / DB / 协作 / 合规一应俱全,但建设成本高

十一个关键维度

绿色 = 显著占优,黄色 = 略胜,红色 = 明显短板。

维度 A · 工作台 B · 通用 Agent
上手成本 低UI 引导,顾问点鼠标 中会用 Claude + 知道何时调哪个 skill
流程一致性 强Critic + Challenger 双层评审 + 固定 schema 中skill 给结构,但执行仍由 agent 灵活完成
资产沉淀 强KB + DB + project_briefs + 跨会议合并(数据级) 中skill 本身就是资产,但每次执行结果散落
灵活度 / 边角案例 弱系统不支持的功能要等开发 强新需求 = 写一条 skill,几小时上线全员
单次任务速度 标准任务 2-5 分钟确定性高 3-15 分钟skill 加持下方差比纯 agent 小
可审计 / 合规 强[D1][K1] 引用追溯 + call_logs + 同步日志 弱中间推理可能丢,无统一日志层
多人协作 / 数据共享 强owner/collaborator/read + 项目级资产 + 跨会议同步 弱skill 全员共享,但执行结果分散在个人
首次上线时间 几个月开发(Claude 辅助下大幅缩短) 写好 skill 当天上线
长期 TCO 服务器 + 维护 + 安全固定成本高 按 token + skill 维护用越多越贵
客户感知 / 销售 强专属系统 = 行业 know-how 壁垒 中skill 库本身也是壁垒,但客户看不到
跟随 AI 前沿 中改 model_router + Claude 辅助,小时级跟进 快平台升级直接享受,skill 通用
建设成本 ⚡ 高(代码 + 测试 + 部署 + 运维) 极低写 markdown skill 即可

三个具体场景

不抽象,直接看每条路怎么做完同一件事。

1客户调研访谈 → 输出纪要 + 需求清单
A

上传音频 → 切片 ASR(100s)→ AI pipeline 出 12 字段纪要 + 12 条需求 + Stakeholder → docx 一键导出。全程顾问不写一句 prompt,所有人产出一致。

B

员工 Claude 调「实施访谈 skill」→ skill 引导 ASR + 出纪要 + 提需求。结构化但仍要 review,跨会议合并干系人 / 沉淀项目资产 / docx 导出要额外 skill 编排,且每次执行结果留在个人电脑。

结论:A 仍占优 — skill 解决了「单次怎么做」,但「跨会议合并 / 项目级共享 / 团队复盘」仍是 A 的强项。
2客户提了个新奇要求"做个客户决策链路图"
A

工作台没这个功能 → 等开发 → 即使 Claude 辅助也得 1-2 天加新页面 + 测试 + 部署。

B

写一条「决策链路 skill」(描述输入 + mermaid 输出 + 引用 KB 数据)→ 几小时上线,全员可用。客户讨论后 skill 迭代成本极低。

结论:B 完胜 — 这是 skill 化的甜蜜点,A 的固化在新颖需求上仍是痛点。
3复盘:这个客户半年里我们出过哪些产物?有哪些没兑现?
A

三栏工作区点开项目 → 所有 insight / survey / meeting / brief / 引用追溯都在,审计完整。

B

员工各自 Claude 会话历史 + 飞书 / 邮件搜索 + 没接集中数据库。skill 是工具不是仓库,执行结果散在个人。

结论:A 完胜 — 资产沉淀和合规审计是 B 路线的根本短板。
4新员工入职 → 让他能干活
A

培训 UI 操作 + 流程节点。固定路径,3-5 天上手。

B

装 Claude / Cursor + 给 skill 库地址。新员工跟着 skill 一步步走,出错 skill 即时反馈。半天到一天上手,且能学到「老员工脑子里的隐性知识」。

结论:B 完胜 — skill 是天然的 onboarding 材料,这点 A 没法比。

三个 A 不愿承认的劣势

作者自我审视,不替 A 站台。⚠️ 但下面有重要补充 — 这三条在「Claude 辅助开发」模式下被改写了。

1. 僵化 (仍然成立)

系统设计成什么样,用户就得按那样用。「顾问勾选式调研」很 opinionated,如果客户场景不适合(比如非要发问卷给客户填),系统给不了好答案。即使开发再快,用户在 UI 里仍然只能干被设计过的事。

2. 改造成本 (被改写,见下节)

加一个简单功能(比如「新增字段」)要改 backend schema / API / 前端类型 / UI 五处。同样改动 B 那边一句话搞定。

↑ 这是「人手开发」节奏。Claude 辅助下 5 处一起改,半小时端到端 commit。

3. AI 进展时滞 (被改写,见下节)

新模型(GPT-5 / Claude 4.x)能力升级,A 要走「评估 → 改 model_router → 测 → 部署」流程,B 是 Claude Code 直接帮你用上。短期内 A 总比前沿慢 1-3 个月。

↑ 改 model_router 是 Claude 几分钟的事,跟前沿基本同步。

但 Claude 改写了规则

A 工作台的「改造成本 + AI 时滞」前提是「人手开发」。Claude 辅助开发把这个前提颠覆掉了。

证据 · 今天这一个会话

7 小时,12 个 commit,0 生产事故

从 P0 视觉改造 + 议题/决议/待办增删改,到 P1 沉淀项目资产 + 跨会议改名同步 + 项目级 Drawer + 进度条 + Cmd+S + 列表搜索 + collaborator 权限,再到这个对比页本身 —— 全部落生产,CI 全绿。按传统节奏这是 2-3 周的 sprint。

改写后:改造成本

Claude 5 处一起改,半小时端到端

backend schema → API → 前端类型 → UI → 测试 → commit → push → CI → 部署。今天加项目级干系人(新表 + 4 endpoints + Drawer + 跨会议同步)从 0 到生产 1.5 小时。人手做同样工作约 3-5 天。

改写后:AI 时滞

跟前沿基本同步

新模型出来,改 model_router 加一行配置,Claude 帮你做 + 测,小时级上线。今天 minimax-m2.5 / xiaomi mimo-v2-omni 在不同 task 各用各的,model_router 整套基础设施已成熟。

所以真正的结论变了

A 工作台的"开发门槛"被 Claude 拉低,B 单一工具的"个人助手"价值仍在

组合公式更新:
· A(工作台)+ Claude(开发外挂)= 标准化工作流 / 资产沉淀 / 合规 + 快速迭代
· B(用户级 agent)= 顾问个人探索 + 一次性新颖任务
· 两者一起,通过 MCP 接口互通(A 给数据,B 给推理)

实操建议:A 当主战场,B 当外挂

不要二选一。A 是工作流和资产,B 是探索能力,各司其职。

场景 用哪个 原因
客户会议纪要 / 需求 / 干系人 / 调研 / 洞察 A 高频 + 标准化 + 需协作 + 客户能看到
项目库 / 历史复盘 / 资产搜索 A 沉淀价值 + 合规审计
一次性任务:做个图、写个奇怪报告、调研三家竞品 B 灵活 + 快,A 加不过来
个人辅助:写邮件、改文案、想方案 B 私人助手,A 不解决这层
客户的临时古怪需求 B 先试 → 反复出现就推动 A 加功能 用 B 作为 A 的"需求探针"
混合姿势:三件套都用上

A(工作台)+ B(skill 库)+ Claude 通过 MCP 互通

· A 提供:结构化数据 + 写操作 API(`get_project_status` / `get_brief` / `ask_kb` 等 8 个 MCP 工具)+ 客户面可见的系统
· B 提供:实施 / 售前 / API / 研报等领域的 skill 库,引导式干活
· Claude 提供:推理引擎 + 把 A 和 B 串起来的执行能力,同时还是 A 的开发外挂
例:销售调「售前方案 skill」→ Claude 用 skill 引导 + 通过 MCP 从 A 查客户历史项目 / 已签合同 / 客户决策人 → 出方案初稿。

最尖锐的判断(skill 化版)

1-3 人小团队 + 没有团队协作 / 合规要求:直接 B 路线。Claude + 几条核心 skill 起步,工作流跑通再说。投 A 完全不划算。

5-20 人团队 / 试水阶段 / 业务还在调整:主投 B,选择性投 A。先把 know-how 沉淀成 skill 库(投入 2-4 周),员工每天调 skill 干活。只有「跨项目资产 + 客户感知 + 合规审计」这几个 B 解决不了的场景才用 A 补。

20-100 人 / 业务稳定 / 客户交付高频:A 和 B 都要,A 是骨架,B 是肌肉。A 解决数据沉淀 + 协作 + 客户视图,B 解决灵活探索 + 新需求快速试。Claude 辅助开发让 A 的建设成本压到原来 1/5-1/10,临界点已经左移。

纷享销客这种规模(100+):A 不是选项是护城河 + skill 库是肌肉。两条都重投。专属系统 + 行业 know-how + 客户感知 + 合规审计是 B 永远做不到的;skill 库则是让所有员工战斗力倍增的杠杆。

所有规模通用:把 Claude 当成「A 的开发外挂 + B 的执行引擎 + 员工的私人助手」三重身份。A、B、Claude 三个角色通过 MCP 接口互通,A 提供数据和写操作 API,skill 提供方法论引导,Claude 完成实际推理。这是终局架构。
