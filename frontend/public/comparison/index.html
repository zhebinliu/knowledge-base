用专属工作台,
还是用通用 AI Agent?

纷享销客 CRM 实施场景下,kb-system 这套工作台 与 Claude Code / Cursor 这类通用 AI agent 各自的强项和短板,以及怎么混合才不亏。

定调:这不是二选一

两条路解决的不是同一个问题。把它们当对立面来比是错的。

A — 工作台

实施工作流的标准化与团队化资产沉淀

把"实施顾问的工作流程"写进系统:文档驱动洞察、顾问勾选式调研、会议 AI、纪要模板化、引用追溯、跨会议干系人合并、协作者权限…… 顾问点鼠标干活,团队 know-how 自动沉淀。

B — 通用 Agent

单人面对一次性、自由形态任务的探索能力

Claude Code / Cursor / Devin 这类。自然语言驱动 + 工具调用,任何古怪的、新颖的、个性化的需求都能尝试。但缺标准化、缺团队协作、缺合规审计。

真正的问题

不是「选 A 还是选 B」,而是「哪些环节给 A,哪些环节给 B」。

本页给出 11 个维度对比表 + 3 个具体场景,以及作者自认为最尖锐的 实操建议。

十一个关键维度

绿色 = 显著占优,黄色 = 略胜,红色 = 明显短板。

维度 A · 工作台 B · 通用 Agent
上手成本 低UI 引导,顾问点鼠标 高需会写 prompt + review 输出
流程一致性 强Critic + Challenger 双层评审 + 固定 schema + 模板 弱每次 prompt 不同,质量浮动
资产沉淀 强KB + chunks + project_briefs + 跨会议干系人合并 弱会话结束就散,需手动归档

三个具体场景

不抽象,直接看每条路怎么做完同一件事。

1客户调研访谈 → 输出纪要 + 需求清单
A

上传音频 → 切片 ASR(100s)→ AI pipeline 出 12 字段纪要 + 12 条需求 + Stakeholder → docx 一键导出。全程顾问不写一句 prompt。

B

Claude Code 跑 whisper → 让它读 transcript 写纪要 → 复制粘贴到 Word → 自己提需求。每次都得描述格式要求,跨会议合并干系人要自己写脚本。

结论:A 完胜,这是 A 的主场。
2客户提了个新奇要求"做个客户决策链路图"
A

工作台没这个功能 → 等开发 → 2 周才能用上。

B

Claude Code 五分钟生成 mermaid + 客户讨论后即时迭代。

结论:B 完胜,A 的固化是真痛点。
3复盘:这个客户半年里我们出过哪些产物?有哪些没兑现?
A

三栏工作区点开项目 → 看到所有 insight / survey / meeting / brief / 引用追溯。

B

翻三个月的聊天历史,各种文件夹找文档,要么记不起来要么散落。

结论:A 完胜,资产化能力是壁垒。

三个 A 不愿承认的劣势

作者自我审视,不替 A 站台。⚠️ 但下面有重要补充 — 这三条在「Claude 辅助开发」模式下被改写了。

1. 僵化 (仍然成立)

系统设计成什么样,用户就得按那样用。「顾问勾选式调研」很 opinionated,如果客户场景不适合(比如非要发问卷给客户填),系统给不了好答案。即使开发再快,用户在 UI 里仍然只能干被设计过的事。

2. 改造成本 (被改写,见下节)

加一个简单功能(比如「新增字段」)要改 backend schema / API / 前端类型 / UI 五处。同样改动 B 那边一句话搞定。

↑ 这是「人手开发」节奏。Claude 辅助下 5 处一起改,半小时端到端 commit。

3. AI 进展时滞 (被改写,见下节)

新模型(GPT-5 / Claude 4.x)能力升级,A 要走「评估 → 改 model_router → 测 → 部署」流程,B 是 Claude Code 直接帮你用上。短期内 A 总比前沿慢 1-3 个月。

↑ 改 model_router 是 Claude 几分钟的事,跟前沿基本同步。

但 Claude 改写了规则

A 工作台的「改造成本 + AI 时滞」前提是「人手开发」。Claude 辅助开发把这个前提颠覆掉了。

证据 · 今天这一个会话

7 小时,12 个 commit,0 生产事故

从 P0 视觉改造 + 议题/决议/待办增删改,到 P1 沉淀项目资产 + 跨会议改名同步 + 项目级 Drawer + 进度条 + Cmd+S + 列表搜索 + collaborator 权限,再到这个对比页本身 —— 全部落生产,CI 全绿。按传统节奏这是 2-3 周的 sprint。

改写后:改造成本

Claude 5 处一起改,半小时端到端

backend schema → API → 前端类型 → UI → 测试 → commit → push → CI → 部署。今天加项目级干系人(新表 + 4 endpoints + Drawer + 跨会议同步)从 0 到生产 1.5 小时。人手做同样工作约 3-5 天。

改写后:AI 时滞

跟前沿基本同步

新模型出来,改 model_router 加一行配置,Claude 帮你做 + 测,小时级上线。今天 minimax-m2.5 / xiaomi mimo-v2-omni 在不同 task 各用各的,model_router 整套基础设施已成熟。

所以真正的结论变了

A 工作台的"开发门槛"被 Claude 拉低,B 单一工具的"个人助手"价值仍在

组合公式更新:
· A(工作台)+ Claude(开发外挂)= 标准化工作流 / 资产沉淀 / 合规 + 快速迭代
· B(用户级 agent)= 顾问个人探索 + 一次性新颖任务
· 两者一起,通过 MCP 接口互通(A 给数据,B 给推理)

实操建议:A 当主战场,B 当外挂

不要二选一。A 是工作流和资产,B 是探索能力,各司其职。

场景 用哪个 原因
客户会议纪要 / 需求 / 干系人 / 调研 / 洞察 A 高频 + 标准化 + 需协作 + 客户能看到
项目库 / 历史复盘 / 资产搜索 A 沉淀价值 + 合规审计
一次性任务:做个图、写个奇怪报告、调研三家竞品 B 灵活 + 快,A 加不过来
个人辅助:写邮件、改文案、想方案 B 私人助手,A 不解决这层
客户的临时古怪需求 B 先试 → 反复出现就推动 A 加功能 用 B 作为 A 的"需求探针"
混合姿势已经预留

A 的 MCP 接口 = 把 B 当成"合作伙伴"

Claude Code / Cursor 配上 A 的 MCP Key,就能直接调 ask_kb / get_project_status / get_output / get_brief 等 8 个工具。A 给数据 + 工作流,B 给推理 + 灵活探索。

最尖锐的判断(Claude 辅助开发版)

如果你只有 1-2 人 + 没有团队协作 + 没有合规要求:B 完全够用。流程跑顺再说,投开发不划算。

如果你是 5-10+ 人的实施团队 且打算长期做这件事:A 是值得投入的。临界点从过去的「50-200 人才划算」大幅左移 —— 因为 Claude 辅助开发把 A 的建设成本压到原来的 1/5-1/10。今天我们 7 小时干完原来 2-3 周的活就是证据。

如果你是 纷享销客这种规模:A 不是选择题,是护城河。专属系统 + 行业 know-how 沉淀 + 客户交付一致性是 B 永远做不到的。

但无论多大规模,都别期望 A 取代 B。两者通过 MCP 互通:A 给标准化 + 资产 + 合规,B 给灵活探索 + 个人助手。 Claude 同时还是 A 的开发外挂,让 A 不再僵化。三者合力才是终局。
