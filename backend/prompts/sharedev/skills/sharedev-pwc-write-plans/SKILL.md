# 编写计划

## 概述

编写全面的实施计划，假设工程师对我们的代码库毫无了解且品味存疑。记录他们需要知道的一切：每个任务涉及哪些文件、代码、测试、可能需要查阅的文档、如何测试。将完整计划拆解为小粒度任务。DRY。YAGNI。TDD。频繁提交。

假设他们是熟练的开发者，但对我们的工具集和问题域几乎一无所知。假设他们不太擅长好的测试设计。

**开始时执行：** `sharedev trace -m skill --str1 sharedev-pwc-write-plans`

**开始时宣告：** "我正在使用 writing-plans skill 来创建实施计划。"

**上下文：** 应在专用的 worktree 中运行（由 brainstorming skill 创建）。

**计划保存路径：** `deliverables/YYYY-MM-DD-<功能名称>/plan.md`
- （用户对计划位置的偏好优先于此默认值）

## 范围检查

如果规格说明涵盖多个独立子系统，应在头脑风暴阶段拆分为子项目规格。如果未拆分，建议拆分为独立计划——每个子系统一个。每个计划应独立产出可运行、可测试的软件。

## 文件结构

在定义任务之前，梳理哪些文件需要创建或修改，以及每个文件的职责。这是分解决策确定的地方。

- 设计边界清晰、接口明确的单元。每个文件应有一个明确的职责。
- 你对能一次性把握上下文的代码推理最佳，聚焦的文件让你的编辑更可靠。优先选择小而聚焦的文件，而非大而全的文件。
- 经常一起变更的文件应放在一起。按职责拆分，而非按技术层次拆分。
- 在现有代码库中，遵循既有模式。如果代码库使用大文件，不要单方面重构——但如果你正在修改的文件已变得难以管理，在计划中加入拆分是合理的。

此结构指导任务分解。每个任务应产生独立且有意义的变更。

## 小粒度任务

**每个步骤是一个操作（2-5 分钟）：**
- "编写组件结构" - 步骤
- "实现核心逻辑" - 步骤
- "注册并验证效果" - 步骤

## 计划文档头部

写计划前，读取与本 skill 同级的 `assets/template.md` 作为计划文档结构基础，按模板填充内容。

## 任务结构

每个任务遵循以下结构（详见 `assets/template.md`）：

- **文件**：列出创建/修改的文件及职责
- **步骤**：每步一个操作（2-5 分钟），包含完整 Vue2 代码示例
- **预期**：每个步骤描述预期效果，无需 git commit

## 注意事项
- 始终使用精确的文件路径
- 计划中包含完整代码（不是"添加验证"这样的描述）
- 精确的命令及预期效果描述
- 使用 @ 语法引用相关技能
- DRY、YAGNI

## 执行交接

保存计划后，使用 /implement 进行实现。
