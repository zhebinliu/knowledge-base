# 修复 PWC Bug

## 概述

通过系统化流程确保 bug 被正确复现、定位并修复，人工验证通过后交由发布流程处理。

**核心原则：** 先复现，再修复；未验证，不完成。

**开始时执行：** `sharedev trace -m skill --str1 sharedev-pwc-fix-bug`

**开始时声明：** "我正在使用 fix-pwc-bug skill 来修复这个问题。"

## 流程

### 第 1 步：复现问题

**在修改任何代码之前，必须先复现问题：**

- 明确描述 bug 的触发条件
- 确认实际行为 vs 预期行为
- 记录复现步骤

如果无法复现，停止并告知用户，请求更多信息。

### 第 2 步：定位代码

**根据复现结果定位问题根源：**

- 阅读相关组件/插件代码
- 找到引发问题的具体位置
- 理解上下文，避免盲目修改

### 第 3 步：修复

**针对根本原因进行最小化修复：**

- 只修改必要的代码，不做无关改动
- 修复后确认逻辑正确

### 第 4 步：人工验证

**修复完成后，提示用户进行人工验证：**

```
请对以下修复进行人工验证：
- 组件/插件名称：<name>
- 类型：<component | plugin>
- 修复内容：<简述修复了什么>
- 验证步骤：<复现步骤>

验证通过后请告知。
```

**停止等待用户确认验证通过。** 未经确认不得宣布修复完成。

**验证通过后：** 宣布修复完成，如需发布请使用 `finish-pwc-development`。

## 常见错误

**未复现直接修改代码**
- **问题：** 可能修错位置，或修复了不存在的问题
- **修复：** 始终先复现，再定位，再修改

**修复范围过大**
- **问题：** 引入新的问题
- **修复：** 只修改与 bug 直接相关的代码

**跳过人工验证**
- **问题：** 修复可能不完整或引入回归
- **修复：** 始终等待用户确认验证通过

## 红线（绝不触犯）

**绝不：**
- 未复现问题直接修改代码
- 未经人工验证宣布修复完成

**始终：**
- 复现后再修复
- 等待用户确认验证通过

## 集成

**完成后可调用：**
- **finish-pwc-development**（需要发布修复时）
