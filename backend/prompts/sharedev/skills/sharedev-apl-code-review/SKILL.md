# APL 代码审查技能

## 严格审批模式（默认开启）

本技能默认按“审查优先于修复，文档优先于结论”执行。除非用户明确要求只做口头快速检查，否则必须遵循以下硬规则：

1. 必须先说明审查范围、输入文件和当前阶段。
2. 必须先执行编译/静态分析检查，再进入人工审查。
3. 审查输出必须先给问题，再给结论；不能只给笼统总结。
4. 发现阻断问题时，不得直接进入发布建议，必须明确标记阻断。
5. 若需要落盘审查结果，必须优先生成或更新 `REVIEW.md`。
6. 若当前需求目录存在 `PRD.md / ARCH.md / TASKS.md`，则 `REVIEW.md` 必须写入同一目录。
7. 未经用户明确授权，不得在审查技能里执行 `sharedev apl push`。

### 审查输出契约

每次审查结束，至少要交付以下内容：

| 输出 | 要求 |
|------|------|
| Findings | 按严重度排序，优先列出阻断项 |
| Assumptions / Open Questions | 列出无法从代码静态确认的风险 |
| Review Decision | `通过 / 有条件通过 / 不通过` |
| REVIEW.md | 能写文件时必须落盘；不能写文件时必须明确说明原因 |
| REVIEW.md 更新 | 发布后必须更新 REVIEW.md，记录发布信息 |

## 平台行为矩阵

| 平台 | 审查过程沟通 | REVIEW 落盘 | 发布动作 |
|------|---------------|-------------|----------|
| Trae IDE | 按阶段回报审查进度 | 优先写入需求目录 | 默认不发布 |
| Claude Code | 按阶段回报审查进度 | 优先写入需求目录 | 默认不发布 |
| Codex | 按阶段回报审查进度 | 优先写入需求目录 | 默认不发布 |

无论在哪个平台：
- “继续”不等于“允许 push”。
- “帮我 review”不等于“允许自动修复并发布”。
- 审查结论必须包含文件位置和未解决风险。

## 跨环境执行约定

本技能需要同时适配 Trae IDE、Claude Code、Codex。执行时不得假设某个 IDE 或 Agent 专属工具一定存在，应按以下能力抽象执行：

| 能力 | 优先方式 | 降级方式 |
|------|----------|----------|
| 任务跟踪 | 使用当前环境的任务/Todo 工具 | 在回复中维护简洁 checklist，并随进度更新 |
| 用户确认 | 使用当前环境的交互提问工具 | 直接向用户提出明确问题，等待回复后继续 |
| 命令执行 | 使用可用终端工具执行 `sharedev` 命令 | 工具不可用时给出精确命令和阻断原因 |
| 审查结论输出 | 生成结构化 REVIEW 结果 | 无法写文件时直接输出审查结论与问题分级 |

后文出现 `AskUserQuestion`、任务列表或类似表述时，均表示上述抽象能力，不表示硬性绑定某个环境的专属工具。

## ⚠️ 重要提示：流程执行规范

**在执行本技能时，必须遵循以下原则：**

1. **严格按流程执行**: 必须按照文档定义的流程顺序执行，不得跳过任何步骤

7. **发布隔离**: 审查流程默认不包含发布；只有用户明确要求并确认后，才可执行任何发布动作

## 技能概述

本技能负责对APL开发代码进行系统化审查，通过预检查、规范合规性检查、代码质量检查、平台限制检查等流程，识别代码中的问题并按严重程度分级，支持自动修复和生成审查报告。

## 核心流程

### Phase 0: 加载 APL 文档参考

**⚠️ 重要**: 在开始代码审查前，必须参考实际的 APL 文档！

**参考文档路径**:
- **APL API 文档**: `<aplApiDocs>`
  - ObjectDataAPI.md - 对象数据操作 API（验证 Fx.object 用法）
  - HttpAPI.md - HTTP 请求 API（验证 Fx.http 用法）
  - MessageAPI.md - 消息通知 API（验证 Fx.message 用法）
  - LogAPI.md - 日志记录 API（验证 log 用法）
- **APL 数据类型**: `<aplDataTypeDocs>`
  - List.md, Map.md, String.md, DateTime.md 等
- **APL 平台限制**: `<enterpriseEA>/.sharedev/docs/apl/pages/func-apl/start/limit.md`
  - 调用次数限制、执行时间限制、内存限制等
- **APL 对象定义**: `<objectsRoot>`
  - 包含所有对象的字段定义，用于验证字段名称

**执行步骤**:
1. 根据代码中使用的 API，读取对应的 API 文档验证用法
2. 根据代码中使用的对象，读取对应的对象定义验证字段名称
3. 读取平台限制文档，验证代码是否符合限制

### Phase 1: 检查前置文档和预检查

在开始代码审查前，检查必要的前置文档是否存在，并执行基础的代码预检查。

#### 1.1 检查 TASKS.md

```
检查流程:
1. 扫描 {项目根目录}/deliverables/ 目录下的子目录
2. 查找最新的需求目录（按时间戳排序）
3. 检查是否存在 TASKS.md 文件
4. 如果存在 → 读取任务列表，确认哪些任务已完成 → 然后进入代码审查
5. 如果不存在 → 直接进入代码审查
```

**目录结构**:
```
deliverables/
└── [需求名称-时间戳]/
    ├── PRD.md
    ├── ARCH.md
    ├── TASKS.md
    └── REVIEW.md
```

#### 1.2 检查代码文件

```
检查流程:
1. 根据 TASKS.md 或用户指定，确定要审查的代码文件
2. 检查代码文件是否存在
3. 读取代码文件内容
```

#### 1.3 文档缺失处理

当 TASKS.md 不存在时：
1. 询问用户要审查哪些代码文件
2. 或者扫描项目目录查找最近修改的 .groovy 文件

#### 1.4 预检查（语法和编译检查）

执行基础的代码预检查，确保代码可正常编译。

##### 1.4.1 文件存在性检查

| 检查项 | 检查方式 | 失败处理 |
|--------|----------|----------|
| 代码文件存在 | 检查文件路径 | 🔴 标记为严重问题 |
| 文件可读取 | 尝试读取文件内容 | 🔴 标记为严重问题 |

##### 1.4.2 Groovy 语法检查

| 检查项 | 检查方式 | 失败处理 |
|--------|----------|----------|
| 括号匹配 | 统计括号数量 | 🔴 标记为严重问题 |
| 引号匹配 | 统计引号数量 | 🔴 标记为严重问题 |
| 关键字拼写 | 检查Groovy关键字 | 🔴 标记为严重问题 |
| 闭包语法 | 检查闭包格式 | 🔴 标记为严重问题 |

##### 1.4.3 编译错误检查

| 检查项 | 检查方式 | 失败处理 |
|--------|----------|----------|
| 未闭合的代码块 | 检查大括号配对 | 🔴 标记为严重问题 |
| 未定义的变量 | 检查变量声明 | 🟡 标记为警告 |
| 类型不匹配 | 检查类型转换 | 🟡 标记为警告 |

##### 1.4.4 sharedev 编译和静态分析检查

**⚠️ 强制性流程**: 必须执行 sharedev 的编译和静态分析检查！

**执行步骤**:

```bash
# 1. 执行编译检查
sharedev apl compile <apiName>

# 2. 执行静态分析
sharedev apl analyze <apiName>
```

**检查项**:

| 检查项 | 命令 | 失败处理 |
|--------|------|----------|
| 编译检查 | `sharedev apl compile <apiName>` | 🔴 标记为严重问题，阻断审查 |
| 静态分析 | `sharedev apl analyze <apiName>` | 🔴 标记为严重问题，阻断审查 |

**执行时机**:
- 在代码文件存在性检查通过后
- 在静态代码审查之前
- 作为质量门控的强制性检查

**失败处理**:
1. 如果编译失败：记录编译错误，提示用户修复，阻断后续审查
2. 如果静态分析失败：记录违规项，提示用户修复，阻断后续审查
3. 只有两者都通过，才能继续进行代码审查

**示例输出**:
```bash
# 成功示例
[sharedev] APL function UIEvt_FillStoreName__c passed compile check.
[sharedev] APL function UIEvt_FillStoreName__c passed analysis.

# 失败示例
[sharedev] Pre-submit compile check failed: Cannot find matching method...
```

##### 1.4.5 预检查结果

预检查通过条件：
- ✅ 代码文件存在且可读
- ✅ 无明显的语法错误
- ✅ sharedev apl compile 通过
- ✅ sharedev apl analyze 通过

预检查不通过时：
- 🔴 记录严重问题
- 跳过后续检查
- 直接生成审查报告

### Phase 2: 规范合规性检查

检查代码是否符合APL开发规范。

#### 2.1 命名规范检查

| 检查项 | 规范要求 | 检查方式 | 问题级别 |
|--------|----------|----------|----------|
| API命名规范 | API名称不能包含下划线（除了结尾的 `__c`） | 正则匹配 | 🔴 严重 |
| 变量命名 | 驼峰命名法 (camelCase) | 正则匹配 | 🟡 警告 |
| 常量命名 | 大写字母+下划线 (UPPER_SNAKE) | 正则匹配 | 🟡 警告 |
| 函数命名 | 清晰表达功能意图 | 语义检查 | 🟢 建议 |
| 保留字使用 | 避免使用 owner, this, delegate | 关键字检查 | 🔴 严重 |

**API命名规范示例**：
```groovy
// ✅ 正确的API命名
EvtMatchContactOnAccountCreate__c    // 事件监听器
BtnUpdateAccount__c                  // 按钮函数
UIEvtFormValidate__c                 // UI事件
AccountQuery__c                      // 普通函数

// ❌ 错误的API命名
Evt_MatchContactOnAccountCreate__c   // 包含下划线
Btn_UpdateAccount__c                 // 包含下划线
UIEvt_FormValidate__c                // 包含下划线
Account_Query__c                     // 包含下划线
```

**变量命名规范示例**：
```groovy
// ✅ 正确
def accountName = "测试客户"
def MAX_RETRY_COUNT = 3
def queryAccountList() { ... }

// ❌ 错误
def account_name = "测试客户"  // 应使用驼峰命名
def maxRetryCount = 3          // 常量应使用大写
def do() { ... }               // 函数名不清晰
```

#### 2.2 错误处理检查

| 检查项 | 规范要求 | 检查方式 | 问题级别 |
|--------|----------|----------|----------|
| API调用错误处理 | 所有API调用都有错误处理 | 检查返回值处理 | 🔴 严重 |
| 错误日志完整性 | 错误日志包含详细信息 | 检查日志内容 | 🟡 警告 |
| 用户友好提示 | 提供用户友好的错误提示 | 检查错误消息 | 🟡 警告 |
| 业务/系统错误区分 | 区分业务错误和系统错误 | 检查错误类型 | 🟢 建议 |

**错误处理示例**：
```groovy
// ✅ 正确
def (Boolean error, Object data, String errorMessage) = Fx.object.create(...)
if (error) {
    log.error("创建客户失败, 对象ID: ${objectId}, 错误: ${errorMessage}")
    Fx.message.throwException("操作失败,请联系管理员")
}

// ❌ 错误
def result = Fx.object.create(...)  // 缺少错误处理
```

#### 2.3 日志记录检查

| 检查项 | 规范要求 | 检查方式 | 问题级别 |
|--------|----------|----------|----------|
| 关键步骤日志 | 关键步骤有日志记录 | 检查log调用 | 🟡 警告 |
| 日志级别正确 | 使用正确的日志级别 | 检查log.info/error | 🟢 建议 |
| 日志内容完整 | 日志包含必要的上下文信息 | 检查日志内容 | 🟢 建议 |

**日志记录示例**：
```groovy
// ✅ 正确
log.info("开始处理数据,总数: ${dataList.size()}")
log.error("处理失败,对象ID: ${objectId}, 错误: ${errorMessage}")

// ❌ 错误
// 缺少日志记录
```

#### 2.4 平台限制合规性检查

| 限制项 | 限制值 | 检查方式 | 问题级别 |
|--------|--------|----------|----------|
| Fx.object调用 | ≤300次/函数 | 统计调用次数 | 🔴 严重 |
| Fx.http调用 | ≤50次/函数 | 统计调用次数 | 🔴 严重 |
| 按钮执行时间 | ≤20秒 | 估算复杂度 | 🔴 严重 |
| 流程执行时间 | ≤300秒 | 估算复杂度 | 🔴 严重 |
| 计划任务时间 | ≤600秒 | 估算复杂度 | 🔴 严重 |
| 内存限制 | ≤256MB | 评估数据量 | 🟡 警告 |
| 批量操作上限 | ≤1000条/次 | 检查批量大小 | 🟡 警告 |

### Phase 3: 代码质量检查

检查代码的质量和可维护性。

#### 3.1 性能优化检查

| 检查项 | 规范要求 | 检查方式 | 问题级别 |
|--------|----------|----------|----------|
| 循环内 API 调用 | 禁止在循环内进行 Fx.object/Fx.http 调用 | 检查循环体 | 🔴 严重 |
| 简单数据遍历 | 允许使用 each 进行数据遍历 | 检查循环体 | ✅ 允许 |
| 批量操作替代 | 循环内单条操作应改为批量操作 | 检查 API 调用 | 🟡 警告 |
| 批量操作 | 使用批量操作代替循环单条操作 | 检查循环内API调用 | 🔴 严重 |
| 重复查询 | 避免重复查询相同数据 | 检查查询逻辑 | 🟡 警告 |
| 字段精简 | 控制查询返回字段数量 | 检查columns参数 | 🟢 建议 |
| 缓存使用 | 合理使用Fx.cache | 检查缓存使用 | 🟢 建议 |

**性能优化示例**：
```groovy
// ❌ 错误: 循环单条操作
dataList.each { item ->
    Fx.object.create("AccountObj", item, CreateAttribute.builder().build())
}

// ✅ 正确: 批量操作
Fx.object.batchCreate("AccountObj", dataList, CreateAttribute.builder().build())
```

#### 3.2 安全检查

| 检查项 | 规范要求 | 检查方式 | 问题级别 |
|--------|----------|----------|----------|
| 输入验证 | 验证输入参数 | 检查参数校验 | 🔴 严重 |
| 敏感信息日志 | 不记录敏感信息到日志 | 检查日志内容 | 🔴 严重 |
| 硬编码密钥 | 不在代码中硬编码密钥 | 检查敏感字符串 | 🔴 严重 |
| 权限检查 | 检查数据访问权限 | 检查权限逻辑 | 🟡 警告 |

**安全检查示例**：
```groovy
// ✅ 正确
def name = context.data?.name
if (!name) {
    Fx.message.throwException("客户名称不能为空")
}
log.info("处理客户: ${name}")

// ❌ 错误
def apiKey = "sk-xxxxx"  // 硬编码密钥
log.info("用户密码: ${password}")  // 记录敏感信息
```

#### 3.3 代码可维护性检查

| 检查项 | 规范要求 | 检查方式 | 问题级别 |
|--------|----------|----------|----------|

## APL 代码规范注意事项

- ⚠️ **循环使用限制**：允许使用 `each` 进行简单数据遍历，但禁止在循环内进行 API 调用
  - ✅ 允许：`result.dataList.each { item -> log.info(...) }` - 简单遍历
  - ❌ 禁止：`dataList.each { item -> Fx.object.create(...) }` - 循环内 API 调用
  - **建议**：使用批量操作替代循环内单条操作
- ❌ **禁止使用 Range 表达式**（0..<n 等）- 会触发 SecurityException
- ❌ **禁止导入外部 Java 包和外部 Groovy 包**
- ❌ **禁止在闭包里使用变量**：owner；this；delegate
- ⚠️ **Map 取值需要类型转换**：`map[key]` 返回 Object 类型，需要显式转换
  - 错误：`map[key] << item` - Object 不能使用 << 操作符
  - 正确：`List list = map[key] as List; list << item`
- ⚠️ **QueryTemplate.AND() 参数格式**：每个 Map 只能包含一个键值对
  - 错误：`QueryTemplate.AND(["field1": value1, "field2": value2])` - Map 包含多个键值对会报错
  - 正确：`QueryTemplate.AND(["field1": value1], ["field2": value2])` - 每个参数是一个只包含一个键值对的 Map
- ⚠️ **log.error() 方法限制**：只接受一个 String 参数，不支持两个参数
  - 错误：`log.error("错误信息", exception)` - 不支持传入异常对象
  - 正确：`log.error("错误信息: ${e.message}")` - 只传入错误信息字符串
- ⚠️ **HashSet 操作限制**：避免使用 `<<` 操作符，使用 `add()` 方法替代
  - 错误：`emailSet << userInfo.email as String` - 类型转换问题
  - 正确：`String email = userInfo.email as String; emailSet.add(email)` - 先转换再添加
- 🔴 **【强制性】所有代码改动必须添加详细注释和逻辑讲解**
  - 文件头部必须有业务逻辑说明块（功能概述、执行流程、性能优化、数据映射）
  - 代码块必须有分隔线和步骤说明（使用 `// ============================================================` 格式）
  - 关键代码行必须有行内注释（变量用途、查询条件说明）
  - 复杂逻辑必须解释"为什么这样做"
  - **缺少注释的代码视为不合格，必须补充完整**
